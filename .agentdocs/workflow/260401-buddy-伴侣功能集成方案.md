# .agentdocs/workflow/260401-buddy-伴侣功能集成方案.md

## 任务概览

目标是在 **OpenAWork** 中引入参考自 `temp/claude-code-sourcemap` 的 buddy / companion 能力，并在不破坏现有聊天主工作流的前提下，形成一份**可评审、可拆解、可直接进入实施**的分阶段集成方案。

本方案覆盖以下问题：

1. Buddy 在 OpenAWork 中的**产品定位**是什么
2. 哪些能力值得迁移，哪些必须删减或后置
3. 应落在哪些模块：`apps/web`、`apps/desktop`、`apps/mobile`、`packages/shared-ui`、`packages/shared`、`services/agent-gateway`
4. 账号级偏好、会话级状态、消息流、系统提示注入分别如何分层
5. Web / Desktop / Mobile 如何分阶段 rollout
6. 风险、实验开关、验收标准与实施顺序如何定义

## 当前分析

### 参考 buddy 能力结论

参考实现并不是“一个会说话的小宠物组件”，而是一组能力的组合：

- **Companion 身份模型**：稳定 bones（由用户标识推导）+ 持久化 soul（name/personality）
- **显式入口**：`/buddy` 命令与输入区触发
- **上下文注入**：通过隐藏的 companion intro 提示告诉主模型“buddy 不是你”
- **轻量展示**：ASCII sprite、bubble、pet 动效、teaser 通知
- **瞬时反应态**：`reaction` / `petAt` 等不进入主聊天历史的 UI 辅助状态

其中最有迁移价值的不是终端 ASCII 实现，而是：

1. 稳定的 companion 身份
2. 清晰的 companion / 主助手边界
3. 低打扰、弱入侵的陪伴式反馈
4. 显式可控的入口与偏好开关

### OpenAWork 现有接入面

#### Web / Desktop 主链路

- `apps/web/src/pages/ChatPage.tsx`
  - 当前聊天主页面，负责会话加载、消息流、右侧运行面板、slash 命令执行、session runtime polling
  - 是 buddy 首期最合适的主承载位
- `apps/web/src/components/chat/ChatComposer.tsx`
  - 当前输入区、附件栏、语音入口、slash 菜单容器
  - 是 buddy dock / trigger / bubble 的最佳挂点
- `apps/web/src/components/chat/ChatPageSections.tsx`
  - 负责消息区渲染、shared-ui 主题变量桥接
  - 不适合作为 buddy 主承载位，但适合处理 buddy 相关卡片或系统反馈
- `apps/web/src/pages/chat-stream-state.ts`
  - 已经把 `RunEvent` 归并为 plan/task/tool/compaction 等右侧面板状态
  - 适合让 buddy 观察现有事件流，生成本地 reaction，而不污染主消息模型
- `apps/desktop/src/App.tsx`
  - 直接复用 `../../web/src/pages/ChatPage.js`
  - 说明 Web 方案如果落在 `ChatPage + shared-ui`，桌面端几乎可零额外业务代码跟进
- `apps/desktop/src/components/layout/Layout.tsx`
  - 桌面端是 glass workbench 布局，buddy 需要符合这里的视觉节奏，不应像终端彩蛋那样跳脱

#### Mobile 独立链路

- `apps/mobile/src/screens/ChatScreen.tsx`
  - 移动端聊天是独立实现，不复用 Web `ChatPage`
  - 当前已具备 `DialogueModeSelector`、`AgentActivityPanel`、附件/语音栏等轻量容器，但没有适合放复杂浮动 companion 的稳定空间
- `apps/mobile/src/navigation/AppNavigator.tsx`
  - 说明移动端首期更适合做“设置可见 + 简化卡片/芯片”，不适合直接跟随 Web/Desktop 一起上复杂 companion 浮层

#### 共享 UI 与协议面

- `packages/shared-ui/src/*`
  - 适合承载 Web/Desktop 可复用的 companion 视觉组件
  - **不适合**承载跨 Web/React Native 通用的 buddy 策略逻辑，因为 shared-ui 不是 RN 共享层
- `packages/shared/src/index.ts`
  - 已承载 `Message` / `RunEvent` / `CommandDescriptor` 等跨端协议
  - 更适合作为 companion 类型、reaction cue、command action 等跨端纯类型/纯逻辑落点

#### 后端与设置面

- `services/agent-gateway/src/routes/settings.ts`
  - 已有 `user_settings` 表读写模式，适合承载账号级 companion 偏好
- `services/agent-gateway/src/session-workspace-metadata.ts`
  - 当前 session metadata 是严格 allowlist，不适合在 Phase 1 随意塞入大量 buddy 字段
- `services/agent-gateway/src/routes/command-descriptors.ts`
  - 统一定义 slash / palette 命令描述
- `services/agent-gateway/src/routes/commands.ts`
  - 当前 server command 走 `/sessions/:sessionId/commands/execute`
- `services/agent-gateway/src/routes/stream-system-prompts.ts`
  - 当前支持 request-scoped system prompts，适合将 companion intro 作为**额外系统提示**注入，而不是篡改消息真相源

### 关键判断

1. **Buddy 更适合放在前端体验层，而不是 agent-core 核心能力层。**
2. **Companion 的“策略/类型”应该做成跨端纯逻辑，而不是绑死在 Web shared-ui 或 gateway。**
3. **首期应优先覆盖 Web/Desktop，共用同一实现；Mobile 延后为简化版。**
4. **Phase 1 不建议引入新的 session metadata 字段，优先用 user_settings + 前端瞬时状态。**

## 方案设计

### 核心原则

1. **体验先于复刻**：优先迁移 buddy 的“陪伴感、低打扰反馈、主助手边界”，而不是复刻 ASCII sprite 与 teaser 活动代码。
2. **体验层优先**：Buddy 首先是聊天工作台的 companion layer，而不是 Agent runtime 的一等核心能力。
3. **前端为主，网关最小侵入**：前端负责 UI、reaction、动画与面板；网关只负责账号级偏好、可选 context 注入与命令/设置注册。
4. **不改变主消息真相模型**：Buddy 默认不写入 `session_messages` 主历史，也不新增 message role；reaction 仅为 UI 附加态。
5. **用户明确可控**：必须支持启用、静音、隐藏、无障碍安静模式、减少动画与 feature flag kill-switch。
6. **Web/Desktop 先行，Mobile 渐进降级**：复用最多的地方优先，避免三端同时重造。

### 硬边界（Oracle 评审后冻结）

1. **不改 `agent-core` 状态机与工具契约**：Buddy 不进入 core agent semantic loop，只做体验层与 gateway 注入装饰。
2. **不改消息真相模型**：Buddy 不新增 message role，不把 reaction 写入 `session_messages` 主历史。
3. **不迁终端特有实现**：不迁 teaser 活动逻辑、不迁 ASCII/Ink 渲染、不迁 CLI footer 交互模型。

### 非目标（Phase 1 明确不做）

- 不迁移终端 ASCII sprite / Ink 渲染代码
- 不迁移彩虹 teaser 与日期窗口逻辑
- 不把 buddy 做成独立第二助手消息流
- 不在首期引入基于 LLM 的 observer 二次推理链
- 不为了 buddy 重写 Chat 架构、右侧面板或消息协议

### 能力裁剪

| 能力块 | 建议 | Phase 1 | 原因 |
| --- | --- | --- | --- |
| Companion 身份模型（bones + soul） | 保留概念，按 OpenAWork 重写 | ✅ | 是 buddy 持续性与个性化核心 |
| 用户级持久化（enabled/muted/profile） | 迁移 | ✅ | Web/Desktop/Mobile 都需要稳定偏好 |
| `/buddy` 显式入口 | 迁移，但改为 OpenAWork 命令模型 | ✅ | 必须有低摩擦发现路径 |
| Companion context 注入 | 迁移 | ✅ | 没有这层，buddy 会退化成纯 UI 挂件 |
| 基础 dock + bubble UI | 迁移，但不复刻终端布局 | ✅ | 提供实际存在感 |
| Reaction 引擎 | 先做事件驱动版 | ✅ | 能快速建立“活着”的感觉，无需新模型链路 |
| Pet / 轻动效 | 可选增强 | ⏳ | 可延后，不阻塞主闭环 |
| 自动 observer（基于消息内容/LLM） | 后置 | ⏳ | 高不确定、高打扰风险 |
| Teaser / 彩虹活动代码 | 不迁移 | ❌ | 与 OpenAWork 主产品价值无关 |
| ASCII 多帧 sprite | 不迁移 | ❌ | 终端实现，不适合当前多端形态 |

### 推荐架构

#### 1. 分层归属

| 层 | 职责 | 推荐位置 |
| --- | --- | --- |
| Companion 协议层 | 类型、设置结构、reaction cue、注入模式 | `packages/shared/src/companion.ts`（新增） |
| Companion 视觉组件 | Dock、Bubble、Settings Card、Profile Card | `packages/shared-ui/src/companion/*`（新增） |
| Web/Desktop 控制器 | 订阅 `RunEvent`、处理 slash/client command、维护瞬时状态 | `apps/web/src/pages/chat-page/use-companion-controller.ts`（新增） |
| 用户偏好存储 | enabled/muted/style/profile/injectionMode | `services/agent-gateway user_settings`（账号级真相源） + `apps/mobile` 本地缓存 |
| Companion 上下文注入 | 根据偏好与当前输入决定是否附加 system prompt | `services/agent-gateway/src/routes/stream-system-prompts.ts` 旁扩展 |

#### 2. 状态模型

##### 账号级持久化（SSOT）

推荐使用 gateway `user_settings` 新增**单一 key**：

- `companion_preferences_v1`

建议结构：

```ts
type CompanionInjectionMode = 'off' | 'mention_only' | 'always';

interface CompanionPreferences {
  enabled: boolean;
  muted: boolean;
  reducedMotion: boolean;
  verbosity: 'minimal' | 'normal';
  injectionMode: CompanionInjectionMode;
  themeVariant: 'default' | 'playful';
}

interface CompanionProfile {
  name: string;
  species: 'otter' | 'cat' | 'duck' | 'fox';
  rarity: 'common' | 'rare' | 'epic';
  personalityTags: string[];
  hatchedAt: string;
  version: 1;
}

interface CompanionSettingsRecord {
  preferences: CompanionPreferences;
  profile: CompanionProfile | null;
  updatedAt: string;
}
```

说明：

- `CompanionProfile` 建议由 gateway 基于 `user.sub` 做确定性生成并与 soul 合并后返回给客户端，避免客户端各端各算各的
- 不建议在客户端暴露生成 salt，避免 profile 漂移
- **不建议长期存自由文本 persona / 自由文本 prompt 片段**；首期只保留受约束的 `personalityTags`、`styleId` 等结构化字段，避免隐私与 prompt 漂移问题
- `apps/mobile` 本地 SQLite 只作为缓存与离线兜底，不作为账号级并列真相源；冲突时以 gateway 返回的 `updatedAt` 更新为准

##### 前端瞬时态（不持久化到消息真相层）

推荐放在 Web 本地聊天控制器 / UI state：

- `panelOpen`
- `dockVisible`
- `currentReaction`
- `currentCueId`
- `cooldownUntil`
- `suppressedForSession`

这些状态**不进入 `session_messages`，也不进入 `RunEvent` 主协议**。

##### 会话级覆盖（Phase 2 可选）

仅当产品明确需要“某个会话关闭 buddy”时，才扩展 session metadata allowlist，新增最小字段：

- `companionDisabled?: boolean`
- `companionInjectionMode?: 'off' | 'mention_only' | 'always'`

Phase 1 不建议扩展，因为当前 `session-workspace-metadata.ts` 为严格 allowlist，越晚扩展越能避免语义膨胀。

### Companion 上下文注入策略

推荐把 buddy 引入模型的方式定义为：

> **request-scoped system prompt augmentation**，而不是 message history mutation。

接入点：

- `services/agent-gateway/src/routes/stream-system-prompts.ts`
- `services/agent-gateway/src/routes/stream.ts`
- `services/agent-gateway/src/routes/stream-runtime.ts`

实现原则：

1. 不写入 `session_messages`
2. 不增加新的 `MessageRole`
3. 不把 companion 反应作为 assistant 正式输出
4. 只在满足条件时额外附加一条 system prompt
5. `stream.ts` 与 `stream-runtime.ts` 必须使用同一 companion prompt builder，避免恢复执行与后台恢复时出现人格漂移

推荐默认策略：

- `injectionMode = mention_only`（默认）
  - 用户输入中显式出现 `/buddy`、buddy 名字、或点击 companion 面板中的“与 buddy 互动”时注入
- `injectionMode = always`（高级选项）
  - 每轮请求都附加 companion 说明
- `off`
  - 完全不注入，仅保留 UI companion

推荐注入语义：

- 告诉主模型：当前对话旁有一个 companion
- 它不是主助手
- 当用户显式对 companion 说话时，主助手尽量退让、简短响应
- companion prompt 保持短小、固定模板化，建议长度上限 **300–500 字符**，并在多轮工具回路中允许按轮次衰减或跳过

### Reaction 引擎

首期不建议做 LLM observer，而建议做**基于现有 `RunEvent` 的规则驱动 reaction engine**。

原因：

- OpenAWork 已有丰富 `RunEvent`：`tool_result`、`task_update`、`permission_asked`、`question_asked`、`compaction` 等
- Web `ChatPage` 已有 `chat-stream-state.ts` 负责归并这些事件
- 可以低成本做出“会看、会点头、会吐槽一点点”的效果，而无需额外模型调用

推荐 Phase 1 cue 规则：

- `permission_asked` → “等你确认一下这步”
- `question_asked` → “这里需要你的决定”
- `tool_result.isError === true` → “这步看起来卡住了”
- `task_update.status === 'done'` → “这一段完成了”
- `compaction` → “我帮你把上下文整理好了”
- `done + 无错误 + 无 reaction 冷却` → “继续下一步？”

约束：

- 每轮最多 1 条 reaction
- 连续 10–20 秒冷却
- 滚动大量历史 / 屏幕阅读器 / reduced motion 模式下默认收敛为静默或文本标签
- UI 呈现优先复用现有 `assistant_event` / 状态卡片风格与前端辅助渲染通道，避免为 buddy 新造一套消息落库与回放机制

### UI 方案

#### Web / Desktop Phase 1

推荐做成 **Composer 右下角 dock + 短气泡 + Companion Panel**：

- 主挂点：`apps/web/src/components/chat/ChatComposer.tsx`
- 状态控制：`apps/web/src/pages/ChatPage.tsx`
- 组件复用：`packages/shared-ui/src/companion/CompanionDock.tsx`

具体表现：

1. 默认显示一个小型 companion dock（头像/名字/静默状态）
2. 有 reaction 时，bubble 在 dock 上方短暂浮现
3. 点击 dock 打开 companion panel：
   - 开关启用
   - 静音
   - reduced motion
   - context injection mode
   - 查看 profile / personality
4. 小屏宽度下退化为 composer 顶部或 top bar 的 chip，不与输入区争空间

理由：

- 不污染消息流
- 不抢右侧任务/工具面板主位置
- Desktop 直接复用 Web 页面，收益最大
- 与当前 glass workbench 风格兼容，容易做成低打扰的工作台伴侣，而不是突兀挂件

#### Mobile Phase 2

移动端推荐降级为：

- 顶部小型 companion chip / 折叠卡片
- 设置页开关
- 极简 reaction 文案
- 暂不提供浮动 bubble 覆盖输入区

理由：

- `ChatScreen.tsx` 当前空间紧凑，已有 `DialogueModeSelector`、活动列表、附件条、输入框
- 移动端更容易受遮挡、键盘弹起、可访问性与触控目标影响

### 命令与入口设计

#### 推荐路线：把 `/buddy` 设计成 **client command**

现状：

- `command-descriptors.ts` 支持 `execution: 'client' | 'server'`
- 但 `composer-slash-items.ts` 目前只把 `execution === 'server'` 的命令放进 composer slash 列表

建议：

1. 扩展 `CommandAction`：
   - `{ kind: 'open_companion_panel' }`
   - `{ kind: 'toggle_companion_mute' }`（可后置）
2. 在 `services/agent-gateway/src/routes/command-descriptors.ts` 中注册：
   - `/buddy`（composer + palette）
3. 修改 `apps/web/src/pages/chat-page/composer-slash-items.ts`，允许将部分 client command 暴露到 composer slash 中
4. `/buddy` 默认执行“打开 companion panel / 聚焦 dock”，而不是走 server roundtrip
5. Settings toggle、`/buddy` slash、companion card 三个入口必须共用同一设置写入链路（统一调用 `/settings/companion`），禁止各自维护影子状态机

理由：

- Buddy 首期主要是体验层入口，不是后端工作流命令
- 避免为了打开一个面板走 `/sessions/:id/commands/execute`
- 同时保持“palette 与 composer 共享同一服务端命令注册表”的现有设计原则

### 设置与 API 设计

推荐新增：

- `GET /settings/companion`
- `PUT /settings/companion`

返回结构：

```ts
{
  profile: CompanionProfile | null,
  preferences: CompanionPreferences,
  feature: {
    enabled: boolean,
    mode: 'off' | 'beta' | 'ga'
  }
}
```

设计意图：

- Web/Desktop 直接读 gateway 作为账号级 SSOT
- Mobile Phase 2 可同步使用同一接口，并在本地 SQLite 做缓存
- 若 feature flag 关闭，也能安全回包并在 UI 中显示“功能未开启”而不是前端报错

#### Feature flag 与默认值

推荐增加服务端 feature gate：

- `enabled: boolean`
- `mode: 'off' | 'beta' | 'ga'`

默认策略：

- 未初始化 / feature 未开启时，`GET /settings/companion` 返回安全默认值，不报错
- UI 与 gateway 均支持 kill-switch，一旦关闭即可完全退化为无 companion 状态

#### Telemetry / rollout 指标

建议 rollout 期间至少记录：

- `companion_prompt_injection_count`
- `companion_prompt_token_estimate`
- `companion_reaction_render_count`
- `companion_settings_toggle_count`
- `companion_error_count`

用于判断：

- 注入命中率是否过低（说明默认策略太隐蔽）
- token 增量是否过高（说明 prompt 太长或触发过频）
- 错误率是否只发生在恢复链路（说明 `stream-runtime.ts` 漏同步）

### 外部实践收敛（方案必须遵循）

结合终端 UI / xterm.js / Ink / feature flag 实践，Buddy 方案需要满足：

1. **Buddy 是渲染层，不是 stdout 噪音**
2. **默认不抢占键盘，不影响 IME / 输入法**
3. **支持 aria-live / reduced motion / 安静模式**
4. **Reaction 更新要限频与冷却**
5. **持久化区分账号级偏好与会话内临时状态**
6. **跨端统一策略，渲染差异化落地**
7. **必须有 feature flag + kill-switch + fallback 默认值**
8. **Buddy 输出折叠在体验层，不能污染主对话视图**

### 平台矩阵

| 平台 | Phase 1 | Phase 2 | Phase 3 |
| --- | --- | --- | --- |
| Web | Companion dock + panel + `/buddy` + event-driven reactions + optional prompt injection | 细化动效 / 会话级覆盖 | 更智能 observer |
| Desktop | 直接复用 Web `ChatPage` 方案 | 桌面端原生窗口级细节 | 通知/系统托盘联动（可选） |
| Mobile | 仅设置占位 / 不落地 UI | 简化 chip/card + 设置同步 + 轻量 reaction | 更细粒度交互 |

## 风险与控制

| 风险 | 说明 | 控制策略 |
| --- | --- | --- |
| 打扰主工作流 | Buddy 过于活跃会损害生产场景 | 默认最小 verbosity、cooldown、静音、可完全关闭 |
| 与主模型边界不清 | Companion 被主模型“代演” | 使用 request-scoped system prompt，默认 mention-only |
| 状态源漂移 | Web、Desktop、Mobile 各自保存一套偏好导致不一致 | Gateway `user_settings` 作为账号级唯一真相源，移动端只做缓存 |
| 恢复链人格漂移 | `stream.ts` 与 `stream-runtime.ts` 注入逻辑不一致 | 共用同一 prompt builder，并补恢复链回归测试 |
| 三端实现割裂 | Web/Desktop 与 Mobile 体验差异过大 | 策略/类型统一放 `packages/shared`，渲染按平台降级 |
| 过早侵入 session metadata | 造成元数据语义膨胀 | Phase 1 不增 metadata；先走 user_settings + 本地瞬时态 |
| 结构化画像变成自由 prompt | companion profile 被存成自由文本，形成隐私/语义失控 | 首期仅存结构化字段，不持久化自由文本 persona |
| 动效与无障碍冲突 | 屏幕阅读器、低性能设备体验差 | reduced motion、aria-live、安静模式、低频更新 |
| 为 buddy 重写聊天体系 | scope 膨胀拖慢主链路 | 明确 buddy 是体验层 companion，不改消息真相模型 |
| Mobile 首期空间不足 | 输入区与键盘冲突 | Mobile 延后到 Phase 4，仅做简化版 |

## 验收标准

- Buddy 在 Web/Desktop 可显式开启/关闭/静音，并记住账号级偏好
- `/buddy` 能通过 slash 命令打开 companion panel，不污染主聊天消息
- Buddy reaction 基于现有 `RunEvent` 触发，且不会写入 `session_messages`
- Gateway 能按用户偏好返回 companion settings，并在 `stream.ts` / `stream-runtime.ts` 两条链路中一致附加 request-scoped companion prompt
- Desktop 无需重复业务实现，仅通过复用 Web `ChatPage` 获得同能力
- Mobile 在 Phase 2 之前不阻塞主链路；Phase 2 之后至少具备设置同步与简化 UI
- 关闭 feature flag 后，UI 与 gateway 都能优雅退化为无 buddy 状态
- Settings toggle、slash、companion card 三个入口最终落到同一设置写入接口，不出现“显示已切换、实际未生效”的漂移

## 推荐实施顺序

1. **先做 settings/profile 协议层**（低风险、高复用）
2. **再做 Web/Desktop dock + panel**（直观看见结果）
3. **随后接 event-driven reaction**（用现有 RunEvent 提升“活性”）
4. **再做 companion prompt injection**（加强边界语义）
5. **最后才考虑 Mobile 与 observer 增强**

## Notes

- Buddy 在 OpenAWork 中的最佳定位是：**工作台级 companion layer**，不是第二聊天代理，也不是纯彩蛋。
- 首期最值得做的不是“会动的宠物”，而是“有稳定身份、能低打扰反馈、并且和主助手边界清晰的 companion”。
- Web 与 Desktop 共享 `ChatPage` 是本方案最大的成本优势，应优先吃满这条复用链。
- 当前 `composer-slash-items.ts` 只暴露 server command，若采用 `/buddy` client command，需要一并扩展该路径。
- 当前 `session-workspace-metadata.ts` 为严格 allowlist，任何会话级 buddy 覆盖都应在 Phase 2 以后再做。
- 首期暂不进入代码实现；当前交付物以设计方案为主。
- 方案必须明确“Buddy 是体验层 companion，不是主 Agent 能力”的边界，避免 scope 膨胀。
- 需要结合 `.serena/memories/ui_style_orientation.md` 中的桌面工作台风格，避免移植后出现视觉割裂。

## Oracle 评审回收（2026-04-01）

- 评审结论：总体方向正确，真正高风险点不在“是否做 companion”，而在**多入口、多状态源、多恢复链路**带来的行为漂移。
- 已吸收的修正：
  1. Gateway 是账号级单一真相源；移动端本地只做缓存。
  2. Companion prompt 注入必须同时覆盖 `stream.ts` 与 `stream-runtime.ts`。
  3. 三入口（Settings / slash / card）统一走同一设置写入链路。
  4. Companion 配置不长期持久化自由文本 persona。
  5. 建议复用现有 `assistant_event` 风格的轻量通道，而不是新造消息类型。
