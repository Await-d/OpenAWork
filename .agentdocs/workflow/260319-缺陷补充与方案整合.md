## Part 1：MVP 前必须补充的功能

### P1-A：AI 感知文件过滤（.agentignore）

**问题**：当前无独立于 .gitignore 的 AI 文件过滤层，Agent 可能读取 `.env`、密钥文件、大型二进制文件，存在安全底线风险。

**归属包**：`packages/agent-core/src/filesystem/ignore.ts`

```typescript
export interface AgentIgnoreManager {
  // 加载忽略规则（.gitignore + .agentignore + 用户全局规则）
  loadRules(projectRoot: string): Promise<IgnoreRuleSet>;
  // 检查文件是否应对 Agent 不可见
  shouldIgnore(filePath: string): boolean;
  // 列出被忽略的文件（调试用）
  listIgnored(dir: string): Promise<string[]>;
  // 运行时添加规则（不持久化）
  addRuntimeRule(pattern: string): void;
}

export interface IgnoreRuleSet {
  gitignorePatterns: string[];    // 从 .gitignore 继承
  agentignorePatterns: string[];  // .agentignore 独有
  userGlobalPatterns: string[];   // ~/.config/openwork/.agentignore
  builtinPatterns: string[];      // 内置安全默认值（见下）
}

// 内置安全默认值（无需用户配置）
const BUILTIN_IGNORE_PATTERNS = [
  '.env', '.env.*',              // 环境变量
  '**/*.pem', '**/*.key',        // 证书/私钥
  '**/id_rsa', '**/id_ed25519',  // SSH 密钥
  '**/.aws/credentials',         // AWS 凭据
  '**/*.p12', '**/*.pfx',        // 证书包
  'node_modules/**',             // 依赖包
  '**/__pycache__/**',           // Python 缓存
  '**/*.lock',                   // 锁文件（大文件）
];
```

**`.agentignore` 文件格式**（与 .gitignore 语法相同）：
```
# AI 专用忽略规则
secrets/
private-data/
*.key
config/production.json
```

**集成点**：
- `FileBrowserAPI.read()` 调用前检查 `shouldIgnore()`
- LSP `touchFile()` 跳过被忽略文件
- 工具沙箱 `file_read` / `file_write` 调用前检查

**UI**：设置页「文件过滤」tab，显示当前有效规则（gitignore + agentignore 合并视图），支持添加自定义规则，预览哪些文件被过滤。

### P1-B：Onboarding 流程

**问题**：当前方案完全缺少首次安装引导，新用户面对空白界面无从下手。

**归属**：`apps/mobile/src/onboarding/` + `apps/desktop/src/onboarding/`

#### 流程设计

```
首次启动检测（无 config 文件）
    ↓
步骤 1：选择连接模式
    ├─ Host（本地）  → 步骤 2A
    ├─ Client（扫码连接桌面）→ 步骤 2B
    └─ Cloud Worker → 步骤 2C

步骤 2A（Host 模式）：
    选择 Provider + 输入 API Key → 实时验证
    ↓
    选择 workspace 目录（原生文件选择器）
    ↓
    配置允许目录范围（双层授权，见 P1-D）
    ↓
    健康检查（Provider 连通性 + 工具沙箱）
    ↓
    Hello World 测试任务（「帮我列出当前目录的文件」）
    ↓
    成功页 + 示例命令卡片

步骤 2B（Client 模式）：
    相机扫描桌面端 QR 码（见 P1-C）
    ↓
    自动建立连接 → 显示 Host 会话列表

步骤 2C（Cloud Worker 模式）：
    登录云控制台 → 启动 Worker（含计费确认）
    ↓
    等待 Worker 就绪 → 获取连接凭据
    ↓
    一键深链接 / 手动 URL + Token 输入
    ↓
    连接验证 → 进入主界面
```

#### UI 组件

```typescript
// 步骤进度管理
export interface OnboardingState {
  step: 'mode' | 'provider' | 'workspace' | 'health' | 'complete';
  mode: 'host' | 'client' | 'cloud';
  providerConfig?: { id: string; apiKey: string; verified: boolean };
  workspaceRoot?: string;
  healthCheckResult?: HealthCheckResult;
}

export interface HealthCheckResult {
  providerReachable: boolean;
  toolSandboxOk: boolean;
  workspaceAccessible: boolean;
  errors: string[];
}
```

**组件清单**：
- `OnboardingWizard`：步骤进度条 + 步骤页切换
- `ModeSelector`：Host / Client / Cloud 三卡片选择
- `ProviderSetup`：API Key 输入 + 实时验证（绿色 ✓ / 红色错误提示）
- `WorkspaceSelector`：原生文件选择器封装
- `HealthChecker`：启动检查进度条（逐项打勾）
- `HelloWorldTask`：可交互的示例任务演示

**跳过条件**：config 文件已存在 → 直接进入主界面；`--skip-onboarding` 标志（CI/自动化场景）。

**验收标准**：全新安装 5 分钟内完成首次任务；Onboarding 完成率 ≥ 80%（首周内测数据）。

### P1-C：移动端 QR 码配对桌面（Host/Client 连接）

**问题**：当前只有「通过 Gateway 云端路由」的方案，移动端如何直连本地桌面 Host 完全没有设计。

**归属**：`packages/pairing/`（新包，Host 和 Client 共用）

```typescript
// packages/pairing/src/index.ts

export interface PairingSession {
  token: string;      // 一次性 token，SHA-256 随机，30 秒有效
  qrData: string;     // JSON：{ hostUrl, token, version }
  hostUrl: string;    // 局域网地址（如 http://192.168.1.5:7788）
  expiresAt: number;
}

export interface PairingManager {
  // Host 侧：生成配对码
  generatePairingCode(): Promise<PairingSession>;
  // Host 侧：等待 Client 连接确认
  waitForClient(token: string): Promise<ClientInfo>;
  // Client 侧：使用 token 连接
  connectWithToken(hostUrl: string, token: string): Promise<void>;
  // 双侧：验证连接状态
  verifyConnection(): Promise<boolean>;
  // 双侧：断开连接
  disconnect(): Promise<void>;
}

export interface ClientInfo {
  deviceName: string;  // 设备名称（如「iPhone 15」）
  platform: 'ios' | 'android';
  connectedAt: number;
}
```

**安全设计**：
- Token 单次使用，连接建立后立即失效
- QR 码 30 秒超时自动刷新
- 连接使用 TLS（自签证书 + 证书指纹验证）
- 局域网内通信，不经过云端
- 防火墙穿透 fallback：通过 Agent Gateway 隧道（仅 Client 扫码但无法直连时）

**UI**：
- Host 侧：`QRCodeDisplay`（二维码 + 倒计时 + 已连接设备列表）
- Client 侧：`QRCodeScanner`（相机扫码 → 自动连接 → 进入 Host 会话列表）
- 连接成功后：移动端显示桌面端会话列表、执行进度、权限批准入口

### P1-D：Artifacts 管理 + 文件变更视图

**问题**：Agent 执行任务后生成的文件、文档、报告没有统一管理，用户不知道 Agent 做了什么、改了哪些文件。

**归属**：`packages/artifacts/`（新包）

```typescript
// packages/artifacts/src/index.ts

export interface RunArtifact {
  id: string;
  sessionId: string;
  type: 'file_created' | 'file_modified' | 'document' | 'log' | 'summary';
  name: string;
  path?: string;           // 本地文件路径
  mimeType?: string;
  sizeBytes?: number;
  createdAt: number;
  preview?: string;        // 前 20 行或前 2KB 预览
}

export interface FileChange {
  path: string;
  status: 'added' | 'modified' | 'deleted' | 'renamed';
  oldPath?: string;
  linesAdded?: number;
  linesDeleted?: number;
  diffSnippet?: string;    // 关键变更片段（前 50 行）
}

export interface ArtifactManager {
  // 列出本次 Run 的所有产出物
  list(sessionId: string): Promise<RunArtifact[]>;
  // 用系统默认程序打开
  open(artifactId: string): Promise<void>;
  // 下载到本地
  download(artifactId: string, dest: string): Promise<void>;
  // 生成分享链接
  share(artifactId: string): Promise<string>;
  // 导出整个 Run 的产出物包（zip）
  exportAll(sessionId: string): Promise<string>;
}

export interface FileBrowserAPI {
  // Run 引发的文件变更（类 git status）
  getChanges(sessionId: string): Promise<FileChange[]>;
  // 全文搜索
  searchText(query: string, options?: { maxResults?: number }): Promise<FileSearchResult[]>;
  // 文件名搜索
  searchFiles(pattern: string): Promise<string[]>;
  // 符号搜索（依赖 LSP）
  searchSymbols(query: string): Promise<SymbolResult[]>;
  // 读取文件内容
  read(path: string): Promise<{ content: string; encoding: string }>;
}
```

**自动捕获时机**：
- tool `write_file` 执行后 → 自动创建 `file_created/file_modified` Artifact
- `session.summarize()` → 摘要作为 `summary` Artifact
- Run 结束 → 汇总展示 ArtifactList

**UI 组件**：
- `ArtifactList`：Run 结束后产出物面板（图标 + 名称 + 大小 + 操作按钮）
- `ArtifactPreview`：内联预览（文本/图片/Markdown 语法高亮）
- `FileStatusPanel`：变更文件列表（类 git diff 视图，可点击展开 diff）
- `FileSearch`：全文 / 文件名 / 符号三种搜索模式

**平台差异**：桌面端直接读文件系统（Tauri 权限控制）；移动端通过 Host API 代理访问。

### P1-E：细粒度权限语义（4 级）

**问题**：当前 agent-core 工具沙箱只有 allow/deny 两级，缺少 session 级和 permanent 级，用户体验差（每次都要确认），且无法撤销已授予权限。

**归属**：扩展 `packages/agent-core/src/permission/`

```typescript
export type PermissionDecision =
  | 'once'       // 允许本次，下次仍提示
  | 'session'    // 本会话内始终允许（会话结束后失效）
  | 'permanent'  // 永久允许（写入配置文件，可撤销）
  | 'reject';    // 拒绝，Agent 优雅降级（不中断 Run，记录到审计日志）

export interface PermissionRequest {
  requestId: string;
  sessionId: string;
  toolName: string;
  scope: string;          // 如 'file:write:/Users/me/docs/'
  reason: string;         // Agent 解释为什么需要此权限
  riskLevel: 'low' | 'medium' | 'high';
  previewAction?: string; // 将要执行的操作预览（如「将创建文件 foo.ts」）
}

export interface PermissionManager {
  // 检查工具是否有权限（返回 null 表示需要用户确认）
  check(toolName: string, scope: string, sessionId: string): Promise<PermissionDecision | null>;
  // 记录用户决策
  reply(requestId: string, decision: PermissionDecision): Promise<void>;
  // 列出已授予的永久权限
  listPermanent(): Promise<GrantedPermission[]>;
  // 撤销永久权限
  revoke(permissionId: string): Promise<void>;
  // 禁用工具（从 LLM 视野中完全移除，防 prompt injection）
  disableTool(toolName: string): void;
  // 检查工具是否被禁用
  isDisabled(toolName: string): boolean;
}

export interface GrantedPermission {
  id: string;
  toolName: string;
  scope: string;
  grantedAt: number;
  sessionId?: string;  // session 级有值，permanent 级为 undefined
}
```

**UI 设计规则**：
- 显示 `reason`（为什么需要此权限）和 `previewAction`
- `riskLevel: 'high'` → 不显示「永久允许」选项，需额外输入确认文字
- `permanent` 选项显示「⚠️ 将永久授权」二次确认
- 设置页「权限管理」tab 可查看和撤销所有永久权限
- 所有决策写入 Audit Log

**disabled_tools 语义**：与 `allowedTools` 白名单不同，`disabledTools` 使工具从工具注册表中完全不暴露给 LLM，LLM 不知道该工具存在，防止 prompt injection 诱导调用。

## Part 2：多 Agent 接口统一

### 问题

当前存在两套不一致的多 Agent 接口定义：
- `260318-高级能力扩展方案.md`：`AgentGraph` / `AgentNode` / `AgentEdge` / `MultiAgentOrchestrator.createGraph()`

### 统一标准接口（以 DAG 命名为准，因决策层协议 D1-D4 已基于此）

```typescript
  output?: unknown;

export interface DAGEdge {
  id: string;
  source: string;
  target: string;
```

## Part 3：跨方案架构缺陷修复

### A3-1：Context Window 管理（Auto Compact）

**问题**：omo 研究文档提到 Auto Compact（95% 时自动摘要压缩），但没有在任何实施方案中落地。

**归属**：`packages/agent-core/src/context/compact.ts`

```typescript
export interface ContextCompactor {
  // 检查当前 token 使用率
  getUsageRatio(): number;  // 0.0 - 1.0
  // 触发压缩（返回压缩后的摘要消息列表）
  compact(messages: Message[], strategy: CompactionStrategy): Promise<Message[]>;
  // 是否应该触发压缩
  shouldCompact(usageRatio: number): boolean;
}

export type CompactionStrategy =
  | 'summarize'     // 用 AI 生成摘要替换历史消息（质量最高）
  | 'truncate'      // 直接截断最旧的消息（最快）
  | 'sliding'       // 滑动窗口（保留最近 N 条 + 系统提示）

// 自动触发规则
const AUTO_COMPACT_THRESHOLD = 0.95;  // 95% 时触发
const COMPACT_TARGET_RATIO = 0.6;     // 压缩到 60%
```

**触发时机**：每次消息发送前检查 `getUsageRatio()`，达到阈值则自动压缩并继续当前会话（无需用户操作）。

### A3-2：流式响应断点续传

**问题**：`T-09`（失败重试策略）只做了重试，没有做流式响应的断点续传，网络中断后用户需要重新发起请求。

**归属**：`packages/agent-core/src/stream/recovery.ts`

```typescript
export interface StreamRecoveryManager {
  // 保存当前流式输出的检查点（每收到 N 个 token 保存一次）
  checkpoint(sessionId: string, content: string): void;
  // 恢复上次中断的流（返回已收到的内容 + 继续请求的 offset）
  recover(sessionId: string): Promise<StreamCheckpoint | null>;
  // 清除检查点（流式完成后调用）
  clear(sessionId: string): void;
}

export interface StreamCheckpoint {
  sessionId: string;
  partialContent: string;  // 已收到的内容
  tokenCount: number;
  savedAt: number;
  canResume: boolean;      // 模型是否支持续写
}
```

**策略**：
- 每 50 个 token 保存一次检查点（内存 + 本地 SQLite）
- 网络中断后 UI 显示「连接中断，是否从上次位置继续？」
- 支持续写的模型（如 Claude）：发送已有内容作为 assistant 前缀，让模型续写
- 不支持续写的模型：提示用户重发，但保留已有内容供参考

### A3-3：统一错误体验

**问题**：各方案独立设计错误处理，没有统一的用户侧错误展示规范。

**归属**：`packages/agent-core/src/error/` + UI 组件库

```typescript
export type AgentErrorCategory =
  | 'network'         // 网络中断、超时
  | 'auth'            // 认证失败、API Key 无效
  | 'rate_limit'      // 模型 API 限流（429）
  | 'model'           // 模型错误（500、overloaded）
  | 'tool'            // 工具执行失败
  | 'permission'      // 权限拒绝
  | 'context_overflow'// Context 超限
  | 'unknown';

export interface AgentError {
  category: AgentErrorCategory;
  message: string;           // 用户友好的错误描述
  technicalDetail?: string;  // 开发者模式下展示
  retryable: boolean;        // 是否可以重试
  retryAfterMs?: number;     // 限流时的建议等待时间
  action?: ErrorAction;      // 建议操作
}

export type ErrorAction =
  | { type: 'retry' }                        // 重试按钮
  | { type: 'switch_model' }                 // 切换模型
  | { type: 'check_settings'; tab: string }  // 跳转设置页
  | { type: 'contact_support' };
```

**错误展示规范**：
- `network`：「网络连接中断，请检查网络后重试」+ 重试按钮
- `rate_limit`：「请求过于频繁，${retryAfterMs/1000}秒后自动重试」+ 倒计时
- `context_overflow`：「对话过长，已自动压缩历史消息」（触发 Auto Compact 后提示）
- `auth`：「API Key 无效，请在设置中重新配置」+ 跳转设置
- `tool` (permission rejected)：「已跳过此操作（权限拒绝），继续其他任务」

### A3-4：审计日志（Audit Log）

**问题**：工具调用、权限决策、Artifact 生成无结构化日志，合规场景和调试均无法满足。

**归属**：`packages/agent-core/src/audit/`

```typescript
export type AuditEntry =
  | { type: 'message'; role: 'user' | 'assistant'; contentHash: string; timestamp: number }
  | { type: 'tool_call'; toolName: string; args: unknown; result: unknown; durationMs: number; timestamp: number }
  | { type: 'permission'; toolName: string; scope: string; decision: PermissionDecision; reason: string; timestamp: number }
  | { type: 'artifact'; name: string; path: string; sizeBytes: number; timestamp: number }
  | { type: 'model_switch'; fromModel: string; toModel: string; reason?: string; timestamp: number }
  | { type: 'compaction'; strategy: CompactionStrategy; beforeTokens: number; afterTokens: number; timestamp: number };

export interface AuditLogManager {
  append(sessionId: string, entry: AuditEntry): void;
  list(sessionId: string, filter?: { type?: string; from?: number; to?: number }): Promise<AuditEntry[]>;
  export(sessionId: string, format: 'json' | 'markdown'): Promise<string>;
  // 导出前自动脱敏（正则匹配 API key / token / password 字段）
  sanitize(content: string): string;
```
