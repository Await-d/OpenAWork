# .agentdocs/workflow/260416-team-创建实施方案.md

## Task Overview
- 目标：基于已完成的设计分析，产出 Team 会话创建流的实施方案，覆盖前端创建向导、shared/web-client DTO、gateway canonical 校验、template metadata 扩展与测试策略。
- 范围：`apps/web` Team 创建向导与状态机、`packages/web-client` Team/Workflows DTO、`services/agent-gateway` Team/Workflows 路由与 metadata/schema、相关测试。
- 非目标：本次不直接实现代码，不切换 Team 主路由，不引入 Team 主实体表。

## Current Analysis
- 当前 Team 页只支持 `workspace -> createThread`，没有“先选团队再创建 session”的正式流程。
- 现有可复用能力已经足够支撑第一版实施：
  - 工作区：`team_workspaces`
  - 运行时 roles 绑定：`use-team-runtime-role-bindings.ts`
  - agent catalog：`/agents` + `ManagedAgentRecord`
  - team-playbook 模板：`workflow_templates`
- 当前主要阻塞不是 UI，而是缺少统一 DTO/schema，以及 gateway 的 canonical 裁决点。

## Implementation Strategy

### 原则
1. 不先造 Team 主实体；先把 Team 作为 session 创建前可解析配置。
2. 前端提交“选择意图”，gateway 生成 canonical snapshot。
3. 解析后的 Team 结果挂在 `sessions.metadata_json.teamDefinition`。
4. 继续复用 `workflow_templates(category='team-playbook')`，但补充 `metadata_json` 承载 team-template 语义。

## Acceptance Criteria
- 用户在没有 workspace 时，必须先创建 workspace 才能创建 Team session。
- 创建 session 时，四个 required roles 都必须绑定 agent 后才能提交。
- optional members 可为空，但若存在必须全部来自 managed agent catalog。
- 选择 saved template 后，required roles / optional members / defaultProvider 必须正确预填。
- gateway 创建出的 session metadata 必须包含 canonical `teamDefinition` snapshot。
- 模板修改不会回写已创建 session 的 teamDefinition。

## Risks
- 继续复用 `/threads` 会让语义混乱，应优先新建 `/sessions` 路由。
- 若 template metadata 继续靠 `nodes/edges` 反解，后续维护成本会快速上升。
- 若前端先算最终 team 结果，后端校验会长期分裂。

## PR-1 Detailed Breakdown

只交付 **前端 blank team session 向导骨架**，不接真实后端新接口、不接模板预填。用户可以从 Team 页打开“创建会话”向导，完整填写：
- 会话标题
- 4 个 required roles 的 agent 绑定
- optional members 选择

但最终提交可以先走本地假提交流程或禁用提交按钮，直到 PR-2 后端接口准备好为止。

### PR-1 Scope
- 包含：前端 UI、前端状态机、表单校验、入口接线、测试骨架
