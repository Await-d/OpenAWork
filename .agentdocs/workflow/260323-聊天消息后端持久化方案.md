# 260323 聊天消息后端持久化方案

## Task Overview

用户要求对当前 Web Chat 的后端保存现状进行详细了解，并制定一套正确、稳妥、可实施的方案。

本方案聚焦以下问题：

1. Web Chat 当前是否真正把聊天消息持久化到后端；
2. 为什么现有 `sessions.messages_json` 没有形成完整闭环；
3. 正确的持久化落点应该放在哪里；
4. 后续应如何分阶段实施，避免只修一半。

## Current Analysis

### 现状链路

1. `apps/web/src/pages/ChatPage.tsx` 中，发送消息时会：
   - 先把 user message 追加到前端内存状态；
   - 再调用 `client.stream(sessionId, message, ...)` 发往网关；
   - 流式完成后把 assistant message 再追加到前端内存状态。
2. `apps/web/src/hooks/useGatewayClient.ts` 通过：
   - `WS /sessions/:id/stream`
   - 失败回退 `SSE /sessions/:id/stream/sse`
   把 `message` 和 `model` 发到后端。
3. `services/agent-gateway/src/db.ts` 的 `sessions` 表确实存在 `messages_json` 字段，但：
   - `services/agent-gateway/src/routes/stream.ts` 不会在流式过程中写回消息；
   - `services/agent-gateway/src/routes/sessions.ts` 的 `PATCH /sessions/:id` 只更新 `title` / `metadata_json`；
   - `messages_json` 仅在 `POST /sessions` 初始化为空数组，或在 `POST /sessions/import` 时写入。
4. `GET /sessions/:id` 目前返回的是 `messages_json` 字段，但 Web 端读取的是 `session.messages`，存在契约错位。

### 已确认的关键缺口

1. **写入缺口**：聊天流式完成后，没有任何后端写库动作。
2. **读取缺口**：后端返回 `messages_json`，前端却期望 `messages` 数组。
3. **上下文缺口**：流式路由发给模型的 `messages` 只包含本轮用户输入，不包含历史消息。
4. **抽象复用缺口**：`packages/agent-core/src/sqlite-session-store.ts` 提供了 SessionStore 能力，但并未被 gateway 复用，且其表结构/驱动/多租户语义与 gateway 当前设计不一致。

### 额外风险

1. `sessions.messages_json` 继续采用“整段 JSON 覆盖写回”的方式，存在并发丢消息风险。
2. gateway 使用 `node:sqlite` 的同步 API，如果在每个 `text_delta` 上写库，会拖慢 WS/SSE 流式路径。
3. `channels/github/cron` 中存在直接 `INSERT sessions` 却未写 `user_id` 的代码路径，需要在正式实施时顺手收口。

## Solution Design

### 推荐方向

**以 agent-gateway 作为聊天消息的唯一事实源（Source of Truth），在 gateway 内补齐消息持久化；复用 agent-core 的是“抽象思路/接口形态”，而不是直接复用其 SQLite 实现。**

### 为什么推荐放在 gateway

1. Web Chat、Desktop、Channel 等真实入口都汇总在 gateway，最靠近真实流式边界；
2. 鉴权、`user_id`、会话归属已经在 gateway 层成立；
3. 若把消息保存职责放给前端补调 API，会引入更多一致性问题；
4. 若直接搬用 agent-core 的 `SQLiteSessionStore`，会遇到以下不匹配：
   - SQLite 驱动不同（`better-sqlite3` vs `node:sqlite`）；
   - schema 不同（gateway 额外有 `user_id` / `title`）；
   - agent-core 当前实现不适合直接承接多用户 gateway 场景。

### 数据模型建议

#### 推荐方案：新增 append-only 消息表

新增 `session_messages` 作为消息真相源，建议字段至少包括：

- `id`
- `session_id`
- `user_id`
- `seq`
- `role`
- `content_json`
- `status`（`final` / `streaming` / `error`）
- `client_request_id`
- `created_at_ms`
- `updated_at`

约束与索引建议：

- `(session_id, seq)` 用于稳定排序；
- `UNIQUE(session_id, client_request_id, role)` 用于幂等去重；
- `(session_id, created_at_ms)` 作为辅助索引；
- `sessions.messages_json` 保留一段时间用于兼容，但不再作为长期真相源。

#### 不推荐方案：继续只改写 `sessions.messages_json`

虽然这是最小改动路径，但存在以下问题：

1. 每次写入都要读全量 JSON、拼接、整段覆盖；
2. 并发下更容易出现 lost update；
3. 后续如果要做分页、断线恢复、幂等写入、工具调用事件落库，扩展空间差。

### 保存时机建议

#### 第一阶段：先做“边界落库”

1. 收到客户端 user message 后，立即落一条 user message；
2. 内存中聚合 assistant 文本；
3. 收到 `done` 后，一次性落一条 assistant final message；
4. 不在每个 `text_delta` 上同步写库。

同时需要满足三个 P0 约束：

1. **幂等去重**：WS 与 SSE fallback 必须复用同一个 `client_request_id`；
2. **顺序稳定**：消息读取时不能只依赖秒级时间戳排序，必须有 `seq` 或等价稳定顺序键；
3. **兼容回读**：在新表上线初期，`GET /sessions/:id` 必须能够兼容解析旧的 `messages_json`，避免历史会话看起来“丢失”。

#### 第二阶段：再考虑低频 checkpoint

如果需要在异常断线时保留部分生成结果，可增加低频草稿写入：

- 每 250~500ms 或累计到一定字符数时，更新 assistant 草稿；
- 必须节流，避免同步 SQLite 把流式链路拖慢。

### 读取与兼容策略

1. `GET /sessions/:id` 先保持对前端的返回 shape 稳定，并优先返回 `messages` 数组；
2. 在新表完全接管前，后端需要兼容从旧 `messages_json` 解析历史记录；
3. 后端内部改为从 `session_messages` 组装 `messages` 数组返回；
4. 如有必要，保留 `messages_json` 作为兼容字段，逐步迁移前端与其它调用方；
5. `packages/web-client/src/sessions.ts` 的 `Session` 类型应与 gateway 实际响应对齐，避免继续出现 `messages` / `messages_json` 契约漂移。

### 模型上下文策略

本轮建议分两段做：

#### P0：先保证“保存 + 回放”

目标是：刷新页面、重新进入会话，能够从后端恢复完整聊天记录。

#### P1：再决定“历史是否回灌给模型”

如果后续要让模型真正具有多轮上下文，再在 `stream.ts` 中把最近 N 条消息或摘要注入上游 `messages`。这一项必须绑定 token 截断策略，否则会造成成本和上下文窗口失控。

## Staged Execution Strategy

为满足“>3 文件改动必须分段”的约束，实施时按以下阶段推进：

1. **Stage A（读取兼容 + 测试先行）**
   - `services/agent-gateway/src/routes/sessions.ts`
   - `packages/web-client/src/sessions.ts`
   - `apps/web/src/pages/ChatPage.tsx` 或对应测试文件
2. **Stage B（存储与迁移）**
   - `services/agent-gateway/src/db.ts`
   - `services/agent-gateway/src/<new message repository>.ts`
   - 必要的测试文件
3. **Stage C（流式写入闭环）**
   - `services/agent-gateway/src/routes/stream.ts`
   - `services/agent-gateway/src/routes/sessions.ts`
   - 对应测试文件

每个 Stage 完成后都需要单独执行验证，再进入下一阶段。

## Acceptance Criteria

1. Web Chat 发送一轮消息后，刷新页面仍能看到 user / assistant 历史；
2. `GET /sessions/:id` 返回的消息字段与 Web / mobile / desktop 客户端约定一致；
3. gateway 流式过程中不会因高频同步写库明显拖慢输出；
4. 失败路径不会把会话写坏，也不会产生明显重复消息；
5. 如未开启历史回灌，至少保证“保存与回放”正确；如开启历史回灌，必须有 token 控制策略。

## Risks and Decision Notes

1. **短期正确性优先于长期炫技**：先做边界落库，不急于做 chunk 级断点续传。
2. **抽象复用要适度**：优先复用 SessionStore 的接口思想，而不是强行复用 agent-core 的 SQLite 实现。
3. **兼容优先**：GET 接口返回 shape 先向前兼容，再逐步清理 `messages_json` 历史包袱。
4. **实施前提**：本次仅完成详细方案与执行计划，尚未开始代码改动；代码实施需在用户明确确认后进入执行阶段。
