# .agentdocs/workflow/260319-对话团队agent决策层协议.md

## 任务概览

基于 HelloAGENTS 路由机制的研究，为「对话团队 Agent」补充决策层协议设计。
现有 `packages/multi-agent/` 已覆盖执行层（DAG 调度、角色分工、事件流），
本文档专注于**执行前的决策层**：何时拉起多 Agent、如何追问补全信息、用户控制粒度、失败升级路径。

---

## 核心思路来源

参考 [HelloAGENTS](https://github.com/hellowind777/helloagents) 的 AGENTS.md 路由协议（G4），
其核心设计洞察：**AI 系统的最大问题不是「执行能力不足」，而是「在错误的时机做了错误的事」**。
解决方案：在进入执行前，强制完成路由评估 + 信息充分性验证 + 用户授权确认。

## 决策层四大模块

### 模块 D1：对话路由层

**位置**：`packages/agent-core/src/routing.ts`  
**职责**：每条用户输入在执行前必须经过路由评估，决定处理路径。

#### 路由等级

| 等级 | 含义 | 触发条件 | 执行方式 |
|------|------|----------|----------|
| R0 | 直接回答 | 不需要执行操作（问答/解释/查询） | 单轮回复 |
| R1 | 单 Agent 快速执行 | 目标明确、单点操作、无需多方案 | 直接执行 |
| R2 | 多 Agent 简化流 | 需分析后定位、局部决策 | 确认后执行 |
| R3 | 多 Agent 完整流 | 架构级、跨模块、多方案对比、高风险 | 评估→确认→执行 |

#### 五维评估维度

```typescript
export interface RoutingDimensions {
  // 维度1：是否需要执行操作
  // false → 直接 R0
  needsAction: boolean;

  // 维度2：目标可定位性
  // 'direct'    → 目标文件/位置/内容可直接确定 → 倾向 R1
  // 'analysis'  → 需分析后才能定位 → R2
  // 'open'      → 开放式目标/跨模块/架构未定 → R3
  targetLocatability: 'direct' | 'analysis' | 'open';

  // 维度3：决策范围
  // 'none'         → 路径唯一，无需决策 → R1
  // 'local'        → 有局部设计决策 → R2
  // 'architectural' → 架构级/多方案/技术栈未定 → R3
  decisionScope: 'none' | 'local' | 'architectural';

  // 维度4：影响范围
  // 'single'      → 单点可逆 → R1
  // 'multi'       → 多点部分可逆 → R2
  // 'cross-system' → 不可逆/跨系统 → R3
  impactScope: 'single' | 'multi' | 'cross-system';

  // 维度5：风险等级（EHRB）
  // 'high' → 强制 R3，且执行前必须用户确认
  riskLevel: 'safe' | 'moderate' | 'high';
}
```

#### 判定规则（取最高等级）

```
needsAction = false           → R0（直接回答）
riskLevel = 'high'            → 强制 R3
decisionScope = 'architectural'
  OR impactScope = 'cross-system'
  OR targetLocatability = 'open' → R3
decisionScope = 'local'
  OR impactScope = 'multi'
  OR targetLocatability = 'analysis' → R2
其余                           → R1
```

#### 完整接口

```typescript
export interface RoutingDecision {
  level: 'R0' | 'R1' | 'R2' | 'R3';
  dimensions: RoutingDimensions;
  // R2/R3 且信息不充分时，返回需追问的问题（按依赖链排序）
  clarifications?: ClarificationQuestion[];
  reason: string;
}

export interface ClarificationQuestion {
  // 信息依赖链：goal → constraint → deliverable → acceptance
  // 每次只追问链上当前最缺的那个维度
  dimension: 'goal' | 'constraint' | 'deliverable' | 'acceptance';
  question: string;
  options?: Array<{ label: string; description?: string }>; // 2-4 个选项
}

export interface ConversationRouter {
  /**
   * 子 Agent 身份标记前缀。
   * Orchestrator 派发子任务时自动注入到 prompt 头部。
   * 子 Agent 检测到此前缀后，跳过路由评估，直接执行任务，
   * 返回纯结果（不包装状态格式，不向用户追问）。
   *
   * 默认值：'[SUB-AGENT] Execute directly. Skip routing. Return results only.'
   * 可按团队约定自定义，但 orchestrator 与子 Agent 侧必须保持一致。
   */
  subAgentPromptPrefix?: string;
}

// Orchestrator 内部派发时自动注入前缀
function buildSubAgentPrompt(role: AgentRole, task: string): string {
  const prefix = role.subAgentPromptPrefix
    ?? '[SUB-AGENT] Execute directly. Skip routing. Return results only.';
  return `${prefix}\n\nRole: ${role.displayName}\nTask: ${task}`;
}
```

#### 入口 Agent vs 子 Agent 行为对照

| 行为 | 入口 Agent | 子 Agent |
|------|-----------|----------|
| 路由评估 | 执行完整五维评估 | 跳过 |
| 信息追问 | 沿依赖链追问用户 | 跳过 |
| 执行确认 | R2/R3 必须等用户确认 | 跳过 |
| 输出格式 | 包装状态头/下一步引导 | 纯结果，不包装 |
| 失败处理 | 升级到主 Agent 根因分析 | 上报错误，由主 Agent 决策 |

### 模块 D4：失败升级策略

**位置**：`packages/multi-agent/src/dag.ts`  
**职责**：定义子 Agent 失败后的重试、根因分析、用户升级完整路径。

#### 扩展 DAGNode

```typescript
export interface RetryPolicy {
  maxRetries: number;            // 最大重试次数，默认 1
  backoffMs: number;             // 指数退避基数（ms），默认 1000
  escalateOnExhaustion: boolean; // 重试耗尽后上报主 Agent，默认 true
}

export interface DAGNode {
  // ... 原有字段（id/type/agentRole/label/status 等）

  // 新增：失败处理策略
  retryPolicy?: RetryPolicy;
  failureEscalationLog?: FailureEscalationRecord[];
}

export interface FailureEscalationRecord {
  attempt: number;
  error: string;
  timestamp: number;
  rootCauseAnalysis?: RootCauseAnalysis;
  resolvedAt?: number;
}

export interface RootCauseAnalysis {
  // 五维根因分类
  category:
    | 'logic_error'        // 逻辑错误（任务描述有歧义/子 Agent 理解偏差）
    | 'missing_dependency' // 依赖缺失（工具/数据/权限不可用）
    | 'env_issue'          // 环境问题（网络/服务不可达）
    | 'input_format'       // 输入格式错误（上游节点输出与预期不符）
    | 'model_capability';  // 模型能力边界（任务超出当前模型能力）

  whyRetryFailed: string;      // 为何之前的重试未生效
  affectedNodes: string[];     // 同类问题的其他节点 ID（系统扫描）
  fixSuggestion: string;       // 可操作的修复建议
  requiresHuman: boolean;      // 是否需要用户介入
  autoFixApplied?: string;     // 若已自动修复，记录修复内容
}
```

#### 失败升级执行流程

```
子 Agent 节点失败
  ↓
按 retryPolicy 重试（指数退避）
  ↓
重试耗尽 且 escalateOnExhaustion = true
  ↓
主 Agent 执行根因分析（RootCauseAnalysis）
  ├─ 根因分类（五选一）
  ├─ 为何重试未生效
  ├─ 扫描其他节点是否有同类问题
  └─ 生成修复建议
  ↓
可自动修复（requiresHuman = false）
  → 修改任务描述后重新派发该节点
  → 记录 autoFixApplied
  ↓
需要用户介入（requiresHuman = true）
  → 触发 human_approval_required 事件
  → 展示根因分析结果 + 修复建议
  → 等待用户决策（重试/跳过/取消）
```

| 模块 | 归属包 | 依赖关系 |
|------|--------|----------|
| D1 路由层 | `packages/agent-core` | 无新依赖，扩展现有 SessionContext |

| D4 失败升级 | `packages/multi-agent` | 扩展现有 DAGNode，依赖 D2 事件 |

**接口边界原则**：
- D1（路由层）在 `agent-core` 内实现，`multi-agent` 包消费路由决策，不反向依赖
- D3（身份协议）通过 prompt 约定实现，不引入新的运行时依赖
- D4（失败升级）的根因分析由主 Agent（Orchestrator）执行，不下放到子 Agent

  evaluate(input: string, context: SessionContext): Promise<RoutingDecision>;

#### 信息依赖链（追问顺序不可乱序）

```
目标范围(goal)
  └─ 约束条件(constraint)      ← 「用什么做」依赖「做什么」
       └─ 成果规格(deliverable)  ← 「做成什么样」依赖前两者
            └─ 验收标准(acceptance) ← 「怎么验证」依赖全部前序
```

每轮只追问链上**首个不充分的维度**，用户回复后重新评估，最多 3 轮。
超过轮次上限后，以已有信息进入执行，DESIGN 阶段补充剩余细节。

### 模块 D2：执行模式控制

**位置**：`packages/multi-agent/src/orchestrator.ts`  
**职责**：让用户在 R2/R3 确认时选择执行自主程度，控制 AI 团队的自主权边界。

#### 两种模式

```typescript
export type WorkflowMode =
  | 'interactive'  // 关键节点暂停等待用户确认（默认，推荐复杂任务）
  | 'delegated';   // 全自动推进，仅高风险时中断（适合信任度高的任务）
```

#### 行为对照

| 场景 | interactive | delegated |
|------|-------------|----------|
| 子 Agent 分配方案确认 | 暂停，展示方案，等待用户选择 | 自动选推荐方案 |
| 阶段间推进 | 每阶段结束暂停确认 | 自动推进 |
| 多方案对比 | 展示所有方案，用户选择 | 自动选评分最高方案 |
| 高风险操作（EHRB） | 暂停，展示风险，等待确认 | 强制暂停（不可绕过）|
| 阻断性失败 | 暂停，展示失败详情 | 强制暂停 |

#### 扩展后的 DAGEvent

```typescript
export type DAGEvent =
  // 原有事件
  | { type: 'node_started'; nodeId: string; timestamp: number }
  | { type: 'node_completed'; nodeId: string; output: unknown; timestamp: number }
  | { type: 'node_failed'; nodeId: string; error: string; timestamp: number }
  | { type: 'edge_activated'; edgeId: string; timestamp: number }
  | { type: 'dag_completed'; result: unknown; timestamp: number }
  // 新增：interactive 模式关键节点确认
  | {
      type: 'human_approval_required';
      nodeId: string;
      plan: string;            // 当前节点执行计划摘要
      options: string[];       // 用户选项（继续/调整/取消）
      autoResolveMs?: number;  // delegated 模式下超时自动选第一项（可选）
    }
  // 新增：delegated 模式高风险强制降级
  | {
      type: 'risk_escalation';
      nodeId: string;
      riskDetail: string;      // 风险描述
      suggestedAction: string; // 建议操作
    };
```

#### 模式切换规则

```
用户确认时选择模式 → 设置 WorkflowMode
delegated 执行中：
  遇到 risk_escalation → 强制暂停 → 用户决策后：
    继续         → 恢复 delegated
    手动处理     → 切换为 interactive
    取消         → 终止 DAG
```

### 模块 D3：子 Agent 身份协议

**位置**：`packages/multi-agent/src/roles.ts`  
**职责**：区分「入口 Agent」和「子 Agent」，子 Agent 跳过路由直接执行，避免嵌套追问。

#### 问题背景

没有身份协议时，子 Agent 收到任务 prompt 后会走完整路由评估，向用户发起追问——
这在团队协作场景中会产生大量无效交互（用户已经在入口层确认过了）。

#### 接口扩展

```typescript
export interface AgentRole {
  id: string;
  displayName: string;
  description: string;
  systemPrompt: string;
  allowedTools: string[];
  model?: string;
  maxTokens?: number;

  /**
   *
