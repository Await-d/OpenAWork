# T-01 跨会话记忆系统实施

## Task Overview
基于既有 `T-01-跨会话记忆系统` 方案，将 OpenAWork 的跨会话记忆能力真正落地到 gateway、agent-core 与 web 设置页，交付可持久化、可注入、可管理、可验证的最小完整产品闭环。

## Current Analysis
- 现有设计文档已明确数据模型、注入格式、API 面与验证矩阵，但业务代码尚未开始。
- Oracle 校验确认：SQLite 持久化应放在 `services/agent-gateway`，纯类型/算法逻辑应放在 `packages/agent-core`；记忆 block 的构造应在 stream 编排层检索、在 `buildRoundSystemMessages()` 统一拼装，避免多轮重复注入。
- `apps/web/src/pages/SettingsPage.tsx` 已达 1478 行，新增记忆管理 UI 必须拆出独立 tab content / hook，不能继续堆在页面文件内。
- 当前目标以 **可工作的产品功能** 为准：先实现统一写入模型下的 CRUD + 注入 + 设置页管理，再补自动提取挂点与验证；所有阶段必须共用同一数据模型与去重/注入管线。

## Solution Design
- gateway：新增 `memories` / `memory_extraction_logs` 表、SQLite store、`/memories` 路由与用户设置读写。
- agent-core：新增 `memory/` 纯逻辑模块，包含类型、Zod schema、注入器、提取器、去重器与公共 helper，经 `src/index.ts` 导出。
- stream：在请求级编排阶段检索记忆 block，在 `buildRoundSystemMessages()` 注入；请求终态触发自动提取并通过 extraction log 做幂等。
- web：新增“记忆管理”tab，提供统计、过滤、列表、编辑、删除、手动提取与开关控制；三态与响应式必须完整。

## Complexity Assessment
- 原子步骤：7 个核心实施步骤（数据层 / API / 注入 / 提取 / UI / 文档 / 验证） → +2
- 并行流：后端存储与前端 UI 可在契约冻结后并行 → +2
- 模块/系统/服务：agent-core + agent-gateway + apps/web → +1
- 单步 >5min：stream 编排、自动提取、集成测试均超过 5 分钟 → +1
- 持久化审查产物：需同步 workflow / runtime / index → +1
- OpenCode 可用：→ -1
- **总分：6**
- **选定模式：Full orchestration**
- **路由理由：跨三层系统改动且需要长期保存实施状态与验证证据，必须使用独立实施工作流。**

## Success Criteria
- 记忆可通过 API 创建、更新、删除、筛选与统计，且严格按用户隔离。
- 新请求在开关开启且存在可召回记忆时，会把 `<user-memory>` block 注入 system prompts；关闭开关时完全不注入。
- 请求完成后可自动提取结构化记忆，并通过 extraction log 保证同一 `clientRequestId` 不重复提取。
- 设置页提供记忆管理 tab，具备 loading / empty / error 三态、移动端 375px 可用、键盘可操作。
- 类型检查、受影响测试、构建、必要的手动 QA 全部有通过证据。

## Test Plan
- 单测：memory schema、injector 排序/预算、deduplicator、extractor 解析。
- gateway 集成测试：memories CRUD/stats/extract、越权访问、stream on/off 注入、提取幂等。
- web 测试：记忆 tab 数据加载、开关切换、错误态与删除/提取交互。
- 构建验证：`pnpm typecheck`、受影响包测试、`pnpm --filter @openAwork/web build`。
- 手动 QA：登录后进入 Settings → 记忆管理，验证列表/搜索/提取/删除/开关；新会话触发后检查记忆注入效果与降级路径。
