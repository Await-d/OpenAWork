# 团队页统一开发主计划

## Task Overview

将已完成的 TeamPage 相关方案文档合并为一份统一开发主计划，目标是让开发者只看这一份文档就能理解：

- 为什么做
- 做成什么
- 先做什么
- 两个人如何并行做
- 哪些风险必须提前规避

本计划不新增产品方向，只对既有方案做统一管理、去重、顺序重排与实施收口。

## Current Analysis

- TeamPage 的对标分析、设计冻结和双人实施分工，现已全部吸收进本主计划。
- 开发阶段不再需要并行查看多个 TeamPage 方案文档。
- 当前新增要求已落实：**只保留本主计划作为唯一开发入口**，并按开发可执行顺序组织内容。
- 2026-04-12 新发现：当前 TeamPage 的真实数据链并没有形成独立 Team 域，而是大量复用了 `session / sharedSession / workingDirectory` 语义；如果继续按旧方案直接推进页面与交互实现，会把错误边界固化到更多模块中。
- 用户已明确要求：**暂停继续写代码，先重新检查并整理 Team 的会话创建、工作区边界与内容隔离设计；办公室视图不承担管理语义。**

## Solution Design

- 用一份主计划承载四类信息：
  1. 背景与冻结结论
  2. 最终产品/页面设计基线
  3. 统一实施顺序
  4. 双人并行开发分工与验收
- 历史方案结论已全部吸收进本主计划，开发执行以本主计划为准。
- 从 2026-04-12 起，本计划的真正主线调整为：
  1. **先冻结 Team / Personal / Shared 三域边界**
  2. **先设计 TeamWorkspace 根对象与独立写入路径**
  3. **再决定哪些现有 session/shared-session 能力以桥接方式复用**
  4. **最后才恢复 TeamPage 页面和交互实现**

## 方案调整（2026-04-12）

> 详细任务实施方案：`./260412-Team详细任务实施方案.md`
>
> 执行关系补充（2026-04-12）：`260410-团队页统一开发主计划.md` 仍是 **唯一开发入口与最终裁决文档**；`260412-Team详细任务实施方案.md` 作为当前 A-E 阶段的**执行颗粒度补充**使用，提供阶段拆分、第一批受影响文件、前端状态替换顺序、热点模块与风险/验证口径。两者若有表述粒度差异，以本主计划的冻结边界与顺序为准。

### 调整原因
- 当前 `/team` 仍然把普通 `session` 与 `sharedSession` 作为主状态来源：会话创建仍走 `createSessionsClient(...).create(...)`，工作区边界仍来自 `workingDirectory / parentSessionId / metadata_json`，任务/消息/评审大量仍是 session 投影。
- 这与用户要求的“**Team 中的内容需要跟对话中的内容隔离，不是完全复用**”直接冲突。
- 因此，旧方案中“先接真实数据、先完善 TeamPage 页面”的顺序必须暂停，改为“**先修正领域边界，再恢复实现**”。

### 新的冻结结论
- `Team` 不是 `Chat/Sessions` 的一个视图标签，而必须是独立协作域。
- `Personal`（个人对话）、`Team`（团队工作区）、`Shared/Bridge`（显式分享/导入桥）必须三域分离。
- `workingDirectory` 只能作为 session 属性，不能继续承担 Team workspace 身份。
- 办公室视图只负责展示 Team 状态，不承载成员管理、权限管理、创建入口等管理语义。

## 吸收 multica 理念的设计原则（2026-04-12）

> 用户明确希望把 `temp/multica` 中的理念引入 OpenAWork。这里吸收的是“产品/协作哲学与对象模型原则”，不是原仓库的 UI 或实现细节逐项照搬。

### 我们要吸收的核心理念

#### 1. Agent 是正式队友，不是一次性工具
- multica 最值得学的，不是“多几个 agent 名字”，而是把 agent 设计成 workspace 里的正式协作成员。
- 对 OpenAWork 的落点：
  - `TeamRoleBinding` 继续保留，并升级为 Team 内的一等对象
  - `interaction-agent / planner / researcher / executor / reviewer` 应视为 Team 内正式执行角色
  - 后续 UI、权限和运行状态都应把这些角色当“队友”而不是“工具按钮”

#### 2. 控制面与执行面分离
- multica 的强点是：平台负责控制、路由、监控，runtime 负责执行。
- 对 OpenAWork 的落点：
  - Team 页面应是控制面（TeamWorkspace / TeamThread / MissionRun 总控）
  - 真实执行仍可复用现有 agent runtime、多 agent orchestrator、task system
  - 不把办公室页误做成执行器本身，也不把 UI 当执行链真相源

#### 3. 任务驱动，而不是一次性 prompt 驱动
- multica 强调的是任务生命周期、可恢复执行、可监控队列，而不是“发一句 prompt 就结束”。
- 对 OpenAWork 的落点：
  - Team 的主流程要围绕 `TeamThread / TeamTask / TeamReview / MissionRun` 组织
  - interaction-agent 的价值不只是改写文案，而是把需求转成可执行任务结构
  - 任务拆解/派发必须最终进入 Team-owned task/run 模型，而不是停留在消息流里

#### 4. Workspace 是硬边界
- multica 的 workspace 是实体边界，不是 metadata 推导出来的标签。
- 对 OpenAWork 的落点：
  - `TeamWorkspace` 必须继续作为一级根对象
  - `workingDirectory` 只能降级为属性，不再承担身份边界
  - `sharedSession` 只能作为 bridge/import 来源，不再反向定义 Team 根状态

#### 5. 线程/对话与任务双轨并存
- multica 不是“只有 issue”也不是“只有 chat”，而是 thread + task 双轨。
- 对 OpenAWork 的落点：
  - `TeamThread` 承载协作上下文与对话
  - `TeamTask` 承载执行对象与状态生命周期
  - 不能再用普通 session 既充当 Team 对话又充当 Team 任务主锚点

#### 6. 技能要复利沉淀
- multica 的另一条理念是 skills compound，而不是每次从 prompt 重来。
- 对 OpenAWork 的落点：
  - TeamTemplate / WorkflowTemplate / CapabilityProfile 需要形成 Team 的可复用执行资产
  - 后续的 Team 设计不应只停留在“派任务”，还要保留“沉淀做法”的位置

### 我们不能直接照搬的部分

#### 1. 不能照搬它的产品形态
- multica 更偏 issue/board/workspace operating shell。
- OpenAWork 当前 Team 仍需要兼容已有 Chat / Sessions / Shared 生态，所以不能直接抄成 issue tracker 外壳。

#### 2. 不能照搬它的执行假设
- multica 明显有 local daemon / runtime-first 的执行模型。
- OpenAWork 已有自己的 gateway、task-system、session-run-events、multi-agent、child session 模式，应该复用既有执行底座，而不是再造一套 daemon 中心架构。

#### 3. 不能破坏已冻结的三域边界
- multica 的理念要吸收到：
  - `Personal`
  - `Team`
  - `Shared/Bridge`
 这套边界里。
- 不能因为学习“agent teammate”就把 shared session আবার抬回 Team 主状态中心。

### multica 理念并入 OpenAWork 后的推荐语义

建议把 OpenAWork Team 的目标语义明确为：

- `TeamWorkspace` = 团队边界与控制面根对象
- `TeamInstance` = 团队配置层（模板、角色绑定、能力配置）
- `MissionRun` = 一次真实执行运行实例
- `TeamThread` = 人和 agent 的协作线程
- `TeamTask` = 拆解后的可执行任务对象
- `TeamReview` = 执行链中的审批/评审对象
- `Shared/Bridge` = Personal 内容进入 Team 的显式桥接层

### 对当前 OpenAWork Team 设计的直接影响

#### 可以增强的部分
- 统一交互代理（`interaction-agent`）的定位可以保留，并升级为 Team 的正式前门队友
- TeamRoleBinding 的执行角色建模是对的，应继续保留并做实
- TeamTemplate / WorkflowTemplate 的资产化方向是对的，应继续推进

#### 必须修正的部分
- 不能再让 Team 页主状态建立在 `selectedSharedSessionId` 上
- 不能再让 Team 创建路径走普通 `/sessions.create`
- 不能把 interaction-agent 只做成文案改写器，后续必须接到 TeamTask / MissionRun 链路

### 吸收顺序建议

按风险最低的方式，建议这样吸收 multica 理念：

1. **先吸收对象模型理念**：TeamWorkspace / TeamInstance / MissionRun / TeamThread / TeamTask
2. **再吸收协作哲学**：agent 是队友、任务驱动、控制面/执行面分离
3. **最后再吸收前门执行流**：interaction-agent 作为 Team 前门，负责分析 → 拆解 → 派发

### 当前明确不做
- 不直接复制 multica 的 issue/board UI 壳
- 不引入一套平行于现有 gateway/task-system 的 daemon-first 执行主链
- 不在边界未冻结前抢先实现“前门 agent 派发”

### 当前架构判定
- **当前实现不可直接继续扩展。**
- 继续在现有 `session/sharedSession` 投影层上叠加 Team 功能，只会把错误边界固化进更多 DTO、路由和页面状态里。

## 新目标与非目标

### 新目标
- 输出一份明确的 Team / Personal / Shared 三域隔离方案。
- 冻结 `TeamWorkspace` 作为一级根对象，并设计其最小实体集合、路由边界、权限边界与写入边界。
- 明确哪些现有 chat/session 基础设施可以复用，哪些语义必须拆分。
- 在边界确认后，再启动下一轮 TeamPage / gateway / web-client 改造。

### 新非目标
- 本轮不继续推进 TeamPage 的页面细化、真实接线或办公室交互增强。
- 本轮不继续把 session/sharedSession 投影包装成“临时可用的 Team 真实模型”。
- 本轮不在办公室视图中继续塞入管理功能。

## A-01 现有耦合点梳理（2026-04-12 完成）

### 一句话冻结结论
- **当前 Team 不是独立协作域，而是“普通 `sessions` + `shared-with-me` + `metadata_json(workingDirectory / parentSessionId)` 派生图”的聚合视图。**
- 因此，后续 A-02 ~ D-03 的所有冻结与迁移，都必须先切 `teamWorkspaceId` 主锚点，再切 Team-owned 写入/读模型，最后才允许 shared-session 降级为 bridge。

### 结合 `260412-Team详细任务实施方案.md` 的执行依据
- 本次 A-01 结论同时吸收 `260412-Team详细任务实施方案.md` 中的三类实施颗粒度：
  1. **第一批热点文件**：`services/agent-gateway/src/routes/team.ts`、`services/agent-gateway/src/routes/sessions.ts`、`services/agent-gateway/src/session-workspace-resolution.ts`、`packages/web-client/src/team.ts`、`packages/web-client/src/sessions.ts`、`apps/web/src/pages/TeamPage.tsx`、`apps/web/src/pages/team/use-team-collaboration.ts`、`apps/web/src/pages/team/runtime/team-runtime-reference-data.tsx`、`apps/web/src/pages/team/runtime/use-team-runtime-projection.ts`、`apps/web/src/pages/team/runtime/team-runtime-model.ts`
  2. **前端高风险替换链**：先冻结壳层 UI 状态，再收敛 projection 层，再迁最脆弱消费端，最后删除 `selectedSharedSessionId` 旧状态源
  3. **迁移顺序约束**：先立 Team canonical model，再立 Team 读模型，再降级 shared session 为 bridge，最后恢复 interaction-agent 与页面深化

### 后端 / contract 现状耦合点

#### 1. `/team/runtime` 直接拼普通 `sessions` 与 shared-session 聚合
- 关键文件：`services/agent-gateway/src/routes/team.ts`
- 当前实现直接查询普通 `sessions` 表，再把结果映射成 Team runtime 的 `sessions` 数组；同时又直接复用 `listSharedSessionsForRecipient(...)` 与 `buildMergedSessionTaskProjection(...)` 生成 `sharedSessions` 与 `runtimeTaskGroups`。
- 这意味着 Team runtime 不是独立域 DTO，而是把：
  - 普通 session 行
  - shared-session 结果
  - 按 `workspacePath` 派生的任务投影
  重新拼成一个页面读模型。

#### 2. Team share 管理直接建立在普通 `sessionId` 之上
- 关键文件：`services/agent-gateway/src/routes/team.ts`
- 当前 Team share 的创建、更新、删除都围绕 `session_shares.session_id` 运转，创建前先查 `sessions WHERE id = ? AND user_id = ?`，返回体再从 `session.metadata_json` 派生 `workspacePath`。
- 这说明 Team 协作分享现在仍然是“对普通 session 开共享”，而不是 Team 原生对象的分享/导入模型。

#### 3. shared detail 返回的 `session` 仍然是普通 session 公共响应
- 关键文件：
  - `services/agent-gateway/src/routes/session-shared-read-routes.ts`
  - `services/agent-gateway/src/routes/session-route-helpers.ts`
- 当前 `GET /sessions/shared-with-me/:sessionId` 会先拿到 owner 的共享访问记录，再拼出一份普通 `SessionRow`，最后通过 `toPublicSessionResponse(...)` 返回。
- 换言之，Shared 域当前没有自己的独立 session contract，仍然是“share 元信息 + 普通 session 公共响应”的组合。

#### 4. `workingDirectory` / `parentSessionId` 都存活在 `sessions.metadata_json`
- 关键文件：
  - `services/agent-gateway/src/routes/sessions.ts`
  - `services/agent-gateway/src/session-workspace-resolution.ts`
- 当前普通 session 创建/更新会把 `workingDirectory`、`parentSessionId` 写回 `metadata_json`；而 Team/shared 视图中的 `workspacePath` 又会递归读取 `parentSessionId`，向父 session 追溯工作区。
- 这意味着 Team 当前看到的“工作区边界”，本质上来自普通 session metadata 及其父子链，而不是 `TeamWorkspace` 一等对象。

### 前端 / 页面状态现状耦合点

#### 1. `/team` 顶层入口直接建立在 `sharedSessions / selectedSharedSession*` 上
- 关键文件：
  - `apps/web/src/App.tsx`
  - `apps/web/src/pages/TeamPage.tsx`
  - `apps/web/src/pages/team/runtime/team-runtime-reference-data.tsx`
- 当前 `/team` 进入 `TeamPage` 后，主链会立即走到 `useResolvedTeamRuntimeReferenceData()`，再把 `useTeamCollaboration()` 里的 `sharedSessions`、`selectedSharedSessionId`、`selectedSharedSession` 送入 `useTeamRuntimeProjection()`。
- 结论：Team 顶层没有自己的主键对象，主视图是以“当前共享会话选择结果”为真相源。

#### 2. `selectTeam(teamId)` 实际上是在选择 `sharedSession.sessionId`
- 关键文件：`apps/web/src/pages/team/runtime/team-runtime-reference-data.tsx`
- 当前所谓的 `teamId`，实际上是通过 `sharedSessions.some((session) => session.sessionId === teamId)` 命中，再调用 `setSelectedSharedSessionId(teamId)`。
- 结论：当前 UI 层的“Team 选择”并不是 Team 实体选择，而是 shared-session 选择。

#### 3. Team 页面中的新建动作仍然走 `createSessionsClient(...).create(...)`
- 关键文件：
  - `apps/web/src/pages/team/runtime/team-runtime-reference-data.tsx`
  - `packages/web-client/src/sessions.ts`
- 当前 Team 页面创建新条目时，会调用普通 `/sessions` 创建接口，并把 `workspacePath` 写到 `metadata.workingDirectory`，随后跳转到 `/chat/:sessionId`。
- 结论：Team 创建路径依旧回落到 Personal Chat Session API，这正是 C-01 要切断的错误写入路径。

#### 4. Team runtime 契约本身已直接引入 sessions/shared contract
- 关键文件：
  - `packages/web-client/src/team.ts`
  - `packages/web-client/src/sessions.ts`
- 当前 `TeamRuntimeReadModel.sharedSessions` 直接使用 `SharedSessionSummaryRecord[]`，`runtimeTaskGroups.tasks` 直接使用 `SessionTask[]`。
- 结论：不仅页面状态，连 web-client Team contract 自身都还是 sessions/shared 域的再包装。

#### 5. `useTeamCollaboration()` 的核心副作用全挂在 `selectedSharedSessionId`
- 关键文件：`apps/web/src/pages/team/use-team-collaboration.ts`
- 当前共享详情加载、presence 心跳、runtime task 拉取、默认选中逻辑，都以 `selectedSharedSessionId` 为驱动，并大量经由 `createSessionsClient` 调 `getSharedWithMe / getTasks / touchSharedPresence`。
- 结论：当前 Team hook 更像“shared-session 协作页容器”，而不是 Team 域状态容器。

### 三域归类后的最终判定
- **Personal 域**：普通 `session`、`/chat`、`/sessions`、用户自己的 message/task/run 链
- **Shared / Bridge 域**：`sharedSessions`、`selectedSharedSessionId`、`workspacePath`、`/sessions/shared-with-me/*`、`createSessionsClient` 在 Team 内承担的桥接调用
- **Team 域（当前缺失）**：缺少真正的 `TeamWorkspace` 主键、缺少 Team-owned thread/task/review/artifact contract、缺少 Team 自己的创建与读取主链

### 结合 multica 官方资料后的边界校准
- `multica` 官方更接近“**workspace 级协作控制面 + agent teammate + thread/task 双轨 + control plane / runtime 分层**”模型，而不是“团队聊天页”。
- 因此，对 OpenAWork 的直接校准是：
  1. `TeamWorkspace` 必须是 workspace 级硬边界
  2. `TeamThread` 与 `TeamTask / MissionRun` 必须双轨并存，不能继续共用普通 session 主锚点
  3. Team 页面是控制面，不是执行器本身
  4. Personal 更偏个人入口与个人对话；Shared 更偏桥接与复用能力层

### 对 A-02 / A-03 / A-04 的直接影响
- **A-02**：`TeamWorkspace` 最小实体集合必须围绕 Team-owned 根对象冻结，不能再把 `workspacePath` 当主键
- **A-03**：`auth / SSE / WS / agent runtime / artifacts / telemetry` 继续归入“可复用基础设施”；`sessionId / sharedSession / workingDirectory / selectedSharedSessionId` 相关语义归入“必须拆分”
- **A-04**：办公室视图必须只消费 Team snapshot，不再直接消费 shared-session 管理与创建动作

## 新的核心对象模型（待后续实现冻结）

### 1. Personal Workspace
- 个人工作区 / 个人对话容器。
- 持有个人 chat sessions、个人草稿、个人临时产物。

### 2. TeamWorkspace
- Team 的一级根对象。
- 持有成员、角色、可见性、TeamThread、TeamTask、TeamArtifact、TeamReview、TeamAuditEvent。

### 3. Shared / Bridge Layer
- 只负责“显式分享 / 引用 / 导入”。
- 不是 Team 的主状态来源，也不是 Team 的真相源。

## 三域隔离总表（2026-04-12 冻结稿）

| 维度 | Personal（个人域） | Team（团队域） | Shared / Bridge（桥接域） |
| --- | --- | --- | --- |
| 根对象 | `PersonalWorkspace` / `PersonalChatSession` | `TeamWorkspace` | `SharedLink` / `ImportedReference` |
| 主用途 | 个人对话、个人草稿、个人临时产物 | 团队线程、团队任务、团队产物、团队评审 | 显式分享、引用、导入、回链 |
| 内容所有权 | 用户个人拥有 | TeamWorkspace 拥有 | 只保存桥接关系，不拥有业务内容 |
| 主状态锚点 | `personalSessionId` | `teamWorkspaceId` / `teamThreadId` | `linkId` / `sourceId` / `targetId` |
| 路由前缀 | `/chat`、`/sessions` | `/team/:teamWorkspaceId` | `/team/:teamWorkspaceId/import`、`/team/:teamWorkspaceId/shares` |
| 消息语义 | 个人对话消息 | TeamThread 内消息 | 引用、导入结果、共享记录 |
| 任务语义 | 个人会话任务 / 运行任务 | TeamTask（归属 Team） | 不直接承载任务，只能关联 |

## Team 独立域 API 草案（A 方案，2026-04-12）

- Team API 的主锚点必须是 `teamWorkspaceId`。
- Team 内的写入对象必须归属 `TeamWorkspace` 或 `TeamThread`，而不是普通 `sessionId`。
- Shared / Bridge 只负责显式分享、引用、导入，不再承载 Team 主视图的真实状态。

## Team 前端状态模型草案（B 方案，2026-04-12）

> 本节的目标是冻结 Team 前端的状态锚点与分层方式，避免继续沿用当前 `useTeamCollaboration -> selectedSharedSessionId -> workspacePath projection` 的错误主链。

### 当前状态问题（必须停止延续）
- 当前 Team 前端主状态仍然建立在：
  - `sessions`
  - `sharedSessions`
  - `selectedSharedSessionId`
  - `selectedSharedSession`
  - `workspacePath`
- 这意味着 Team UI 的“当前对象”并不是 `TeamWorkspace` 或 `TeamThread`，而仍然是某个 shared session 的衍生选择结果。
- 如果继续扩展这条状态链，Team 会一直只是 session 投影层。

## 状态冻结原则

### 必须遵守
- `/team` 的主状态来源是 `activeTeamWorkspaceId`。
- Team 主内容切换优先围绕 `activeTeamThreadId`，而不是 shared session。
- Team 页面的一切领域数据都从 `teamWorkspaceSnapshotState` 或 Team 子域列表状态进入。

## 迁移实施草案（C 方案，2026-04-12）

> 本节把“边界设计”翻译成后续真正可执行的实施顺序。原则是：**先换主锚点，再换写入路径，再换读模型，最后收口旧投影。** 不允许反过来先修页面再补领域边界。

### 实施总原则
- 任何阶段都不允许把 Team 的新功能继续绑定到 `/sessions` 或 `sharedSession` 主状态上。

- 办公室视图始终只读；管理和创建动作进入 Team 专属入口。
- 每一阶段都要可独立回归，避免大爆炸迁移。

### Phase 0：冻结合同与命名空间（必须先做）

## 背景与冻结结论

### 为什么要做
- 当前 TeamPage 仍偏“成员 / 任务 / 消息 / 共享会话 / 审计 / presence”的协作页。
- 目标是把它升级成 **多 Agent 运行工作台**，复刻的是参考库的“使用方式”，不是技术结构或视觉皮肤的逐像素复制。

### 学什么 / 不学什么
- **学习**：Supervisor/子 Agent/任务看板/结构化事件流/团队运行闭环。
- **不学习**：参考库的 Electron 主进程中心化结构、README 中未落地的 team 模块边界、全能桌面总控台式实现。

### 已冻结方向（不得反复讨论）
- Web 优先
- 工作区一级
- 总控大盘
- 主对象：`Workspace → Team Instance → Mission Run`
- 执行角色独立于权限成员角色
- 人工消息默认全部先进入 `interaction-agent`
- 设计原则：`upstream-native first / merge-first`
- 画面结构可参考参考库，但**任务动画统一替换为 Buddy/Hubby 动画层**

## 目标与非目标

### 目标
- 让开发者交付一个可运行的 Web Team Runtime 总控页
- 使用现有上游真相源完成并轨：`run events / task projection / session tree / shared session / approval/question / artifacts / agents/capabilities / workflow template`
- 让 TeamPage 成为：
  - 工作区首页入口
  - Team Instance 运行总控
  - 人工介入与共享协作主入口

### 非目标
- 首期不做桌面端/移动端完整总控同步
- 首期不并入自定义 DAG 编辑器

## 产品与页面设计冻结稿

### 核心对象模型

#### 1. Workspace
- 一级总控对象
- 对应会话工作区 / 项目
- 首页以工作区卡片或分组展示

#### 2. Team Instance
- 工作区下的一组固定协作编排配置
- 包含：模板来源、执行角色映射、默认 Agent/能力集、上下文策略
- 不等于一次运行

#### 3. Mission Run
- 团队实例上的一次具体执行
- 包含：目标、状态、开始时间、当前阻塞、当前活跃角色、最近事件、关联会话树
- 是 TeamPage 的真正主对象

#### 4. Execution Role
- 如：`interaction-agent / planner / researcher / executor / reviewer`
- 与 `owner/admin/member` 这类权限成员角色分离

#### 5. Shared Context
- 复用共享会话、评论、presence、审批、问题回复等现有链路
- 是 Mission Run 的上下文层，不单独成为主对象

#### 6. Output Surface
- 复用 artifacts、文件上下文、Git 变更、结果摘要
- 是运行结果层

### 页面信息架构

#### 顶部：总控指标卡
- 运行/阻塞/待审批/待回答
- 会话活跃
- 角色负载
- 成本用量
- 变更冲突
- 失败告警

#### 一级：工作区卡片
- 每张卡片展示该工作区的团队实例摘要与最新 Mission Run 状态
- 点击进入工作区总控视图

#### 二级：工作区内 7 个主 Tab
- `总览`：Mission Run 概况、活跃角色、关键状态
- `会话/Agent`：父子会话树、活跃 Agent、运行状态
- `任务看板`：任务推进、状态泳道、阻塞点
- `文件上下文`：共享会话、文件、工作区上下文
- `消息时间线`：事件流、角色消息、审批与提问
- `产物`：artifacts、结果输出、版本切换
- `Git/变更`：文件改动、冲突、提交相关信息

### 统一交互代理
- `interaction-agent` 是团队实例的统一人工入口
- 人工消息默认全部先进入该角色
- 首期职责仅限：**理解并改写需求**
- 首期不承担：完整拆解、汇总、审批守门

### 动画表现冻结约束
- 页面结构、分区方式、Tab 组织、总控布局可强对齐参考库
- 参考库原任务动画**不复用**
- TeamPage / 团队运行界面中的任务动画统一替换为系统内 **Buddy/Hubby 动画表现层**
- Buddy/Hubby 默认只承担：
  - 任务运行中提示
  - 阻塞 / 待审批 / 待回答提示
  - `interaction-agent` 的轻量视觉代理
- Buddy/Hubby 不承担：
  - 替代 DAG / 时间线主视图
  - 逐条复述消息流或任务明细

## 风险清单

1. **按 Tab 平分导致热点冲突**：必须按真相源边界拆，而不是按 Tab 平均拆。
2. **DTO 未冻结就接 UI**：A 先接真实字段，B 再改合同，会出现高频返工。
3. **共享会话与运行事件语义漂移**：B 同时碰 team/shared/session-run-events，必须保持 operate / 审计语义一致。
4. **shared-ui 过早卷入**：非必要不提前改通用组件，避免波及 ChatPage。
5. **默认开 schema 主线**：当前已有上游支持，除非被阻塞，否则不把 schema 迁移作为主任务。
6. **Buddy/Hubby 语义越位**：动画层只做状态与交互代理的辅助表达，不替代 DAG/时间线主视图。
