# .agentdocs/workflow/260416-team-创建流程设计分析.md

## Task Overview
- 目标：分析 Team 页当前“先创建工作区，再在工作区下创建会话并选择团队/agent/模板”的目标流程，确认现状、缺口、需要调整的文件与设计边界。
- 范围：`apps/web` Team 页与 Agents/Workflows 相关 UI、`packages/web-client` Team/Agents/Workflows DTO、`services/agent-gateway` Team/Agents/Workflows 路由与 agent catalog。
- 非目标：本次不直接实现，不做路由迁移，不引入新的 Team 持久化实体。

## Current Analysis
- 当前 Team 主流程仍然是 **workspace-first + 直接 createThread**：`TeamSessionSidebar` 里“新建会话”按钮直接调用 `data.createSession(workspacePath)`，最终走 `teamClient.createThread(teamWorkspaceId, { metadata })`。
- 当前并不存在“创建会话向导”或“先选团队再创建会话”的正式步骤；也没有把 saved team template 作为 create-session 输入。
- “团队模板”目前实际上是 `category='team-playbook'` 的 workflow template：结构是 `nodes/edges`，不是团队角色/成员定义 schema。
- “必需角色”已经有运行态 UI 雏形：`use-team-runtime-role-bindings.ts` 固定了 planner / researcher / executor / reviewer 四个 role，并根据 `/agents` 的 `canonicalRole` 自动建议绑定。
- “可选成员”目前没有被正式建模：模板弹窗只维护一个扁平的 `selectedRoles: Set<string>`，没有 required/optional 分层，也没有 extra members schema。

## Gap Analysis

### Gap A — 缺少“创建会话前选择团队”的正式前端流程
- 当前只有“新建会话”按钮，没有向导/弹窗。
- `TeamPage.tsx` 和 `TeamSessionSidebar.tsx` 需要新增一个 session creation flow 容器，而不是继续直接调用 `createThread`。

### Gap B — 缺少 Team definition 的共享 DTO/schema
- 现在 `CreateTeamThreadInput` 只有 `title` 和 `metadata`。
- 需要一个能表达以下信息的 DTO：
  - `workspaceId`
  - `templateId?`
  - `requiredRoles`
  - `optionalMembers`
  - `agentBindingMap`
  - `defaultProvider?`

### Gap C — required roles / optional members 未分层
- 现有 `roleCards` 已能表达“核心角色绑定”。
- 但 optional agent members 还没有独立 UI/数据结构。

### Gap D — 模板与 session 创建没有直连
- 现在可以创建 `team-playbook` 模板，但不能在创建 session 时选择模板并自动展开/补全角色。
- `WorkflowsPage.tsx` 已经能拼 query 跳转 Team，但 `TeamPage.tsx` 当前并不消费这些 query。

### Gap E — 后端缺少 canonical 裁决点
- 如果前端直接把“最终团队结果”算好提交，后端很容易和 `/agents` catalog、模板、required roles 校验分裂。
- 最安全的做法是：前端只提交**选择意图**，gateway 统一展开模板、补齐核心角色并校验 agent 合法性，然后创建 session 并把解析后的 Team 快照持久化到 session metadata。

## Recommended Design

### 总体原则
把 Team 视为“**session 创建前可解析的配置**”，不是一个新的、先于 session 持久化的主实体。

### 前端建议
1. 保持 workspace-first：没有 workspace 时先创建 workspace。
2. 点击“新建会话”后，进入 3 段式向导：
   - 第一步：选择来源（空白团队 / 内置模板 / 已保存模板）
   - 第二步：绑定 4 个 required roles
   - 第三步：选择 optional agent members
3. 运行态 `use-team-runtime-role-bindings.ts` 可复用，但要抽出一层“创建用 bindings model”，避免直接与运行态 UI 状态耦合。

### DTO / API 建议
新增专用 create-session payload，而不是继续复用纯 `createThread`：

```ts
interface CreateTeamSessionInput {
  title?: string;
  teamWorkspaceId: string;
  source?: {
    kind: 'blank' | 'builtin-template' | 'saved-template';
    templateId?: string;
  };
  requiredRoleBindings: Array<{
    role: 'planner' | 'researcher' | 'executor' | 'reviewer';
    agentId: string;
  }>;
  optionalAgentIds?: string[];
  defaultProvider?: string | null;
}
```

## Detailed Interface Draft

### 1. 前端创建向导状态（Web-only）

建议新增一个只存在于 Web UI 的本地状态模型，用于承载向导过程：

```ts
type TeamSourceKind = 'blank' | 'builtin-template' | 'saved-template';

type RequiredCoreRole = 'planner' | 'researcher' | 'executor' | 'reviewer';

interface TeamSessionCreationDraft {
  teamWorkspaceId: string;
  title: string;
  source: {
    kind: TeamSourceKind;
    templateId?: string;
  };
  requiredRoleBindings: Array<{
    role: RequiredCoreRole;
    agentId: string;
  }>;
  optionalAgentIds: string[];
  defaultProvider: string | null;
}
```

用途：
- `TeamSessionSidebar` 只负责打开向导，不再直接创建 thread。
- `NewTeamSessionModal` / `use-team-session-creation.ts` 负责维护这份 draft。
- 页面关闭前，这份 draft 不需要持久化到数据库。

### 2. Gateway 创建输入 DTO（canonical input）

建议 `packages/web-client/src/team.ts` 扩展为：

```ts
export type TeamSessionTemplateSourceKind = 'blank' | 'builtin-template' | 'saved-template';

export interface TeamRequiredRoleBindingInput {
  role: 'planner' | 'researcher' | 'executor' | 'reviewer';
  agentId: string;
}

export interface CreateTeamSessionInput {
  title?: string;
  source?: {
    kind: TeamSessionTemplateSourceKind;
    templateId?: string;
  };
  requiredRoleBindings: TeamRequiredRoleBindingInput[];
  optionalAgentIds?: string[];
  defaultProvider?: string | null;
}
```

注意：
- 前端提交的是“用户选择意图”，不是最终真相。
- gateway 仍然要二次解析 template、补齐 required roles，并过滤非法 agentId。

### 3. Session metadata 中保存的 Team 快照（resolved snapshot）

当前 session metadata 已经支持 `teamWorkspaceId`。建议把创建后的 canonical 结果落到 session metadata 中，新增一个 `teamDefinition` 节点：

```ts
interface PersistedTeamDefinition {
  source: {
    kind: 'blank' | 'builtin-template' | 'saved-template';
    templateId?: string;
    templateName?: string;
  };
  requiredRoleBindings: Array<{
    role: 'planner' | 'researcher' | 'executor' | 'reviewer';
    agentId: string;
    agentLabel: string;
  }>;
  optionalMembers: Array<{
    agentId: string;
    agentLabel: string;
    canonicalRole?: string;
  }>;
  defaultProvider?: string | null;
  createdAt: string;
}
```

推荐 metadata 形状：

```json
{
  "teamWorkspaceId": "workspace-1",
  "workingDirectory": "/repo/apps/web",
  "teamDefinition": {
    "source": {
      "kind": "saved-template",
      "templateId": "tpl-1",
      "templateName": "研究团队模板"
    },
    "requiredRoleBindings": [
      { "role": "planner", "agentId": "oracle", "agentLabel": "Oracle" },
      { "role": "researcher", "agentId": "librarian", "agentLabel": "Librarian" },
      { "role": "executor", "agentId": "hephaestus", "agentLabel": "Hephaestus" },
      { "role": "reviewer", "agentId": "momus", "agentLabel": "Momus" }
    ],
    "optionalMembers": [
      { "agentId": "atlas", "agentLabel": "Atlas", "canonicalRole": "reviewer/verifier" }
    ],
    "defaultProvider": "claude-code",
    "createdAt": "2026-04-16T10:00:00.000Z"
  }
}
```

这份 snapshot 的意义是：
- 会话创建后，不再依赖模板实时反解；
- 模板后续修改不会污染旧 session；
- runtime UI 可以直接读这份 canonical 结果。

## API Draft

### 推荐方案 A：新增专用路由（更清晰）

比起继续滥用 `/threads`，更建议新增：

```http
POST /team/workspaces/:teamWorkspaceId/sessions
Authorization: Bearer <token>
Content-Type: application/json
```

请求体：

```json
{
  "title": "研究团队 2026-04-16",
  "source": {
    "kind": "saved-template",
    "templateId": "tpl-1"
  },
  "requiredRoleBindings": [
    { "role": "planner", "agentId": "oracle" },
    { "role": "researcher", "agentId": "librarian" },
    { "role": "executor", "agentId": "hephaestus" },
    { "role": "reviewer", "agentId": "momus" }
  ],
  "optionalAgentIds": ["atlas"],
  "defaultProvider": "claude-code"
}
```

成功响应：

```json
{
  "id": "session-1",
  "title": "研究团队 2026-04-16",
  "state_status": "idle",
  "metadata_json": "...含 teamDefinition 的最终 metadata..."
}
```

### 为什么不推荐继续扩 `/threads`
- `/threads` 当前语义太窄：只负责“在 workspace 下开一个 thread”；
- 如果继续向里面塞 teamDefinition，会让老调用和新语义混在一起；
- 单独 `/sessions` 路由更符合产品语义，也更利于后续加校验和审计。

## Gateway Validation Draft

建议在 `services/agent-gateway/src/routes/team.ts` 里新增专用 schema：

```ts
const createTeamSessionSchema = z.object({
  title: z.string().min(1).max(200).optional(),
  source: z
    .object({
      kind: z.enum(['blank', 'builtin-template', 'saved-template']),
      templateId: z.string().min(1).optional(),
    })
    .optional(),
  requiredRoleBindings: z
    .array(
      z.object({
        role: z.enum(['planner', 'researcher', 'executor', 'reviewer']),
        agentId: z.string().min(1),
      }),
    )
    .length(4),
  optionalAgentIds: z.array(z.string().min(1)).default([]),
  defaultProvider: z.string().nullable().optional(),
});
```

Gateway 处理顺序建议：
1. 校验 workspace 存在；
2. 读取并解析 template（如果 `source.templateId` 存在）；
3. 合并 template 默认绑定与用户覆盖；
4. 强校验 4 个 required roles 都有 agent；
5. 用 `agent-catalog.ts` / `/agents` 所在数据源校验 agentId 合法；
6. 去重 optionalAgentIds，禁止与 requiredRoleBindings 重复；
7. 生成最终 `teamDefinition` snapshot；
8. 创建 session 并写入 metadata。

## Table Strategy Draft

### 推荐最小策略：不新增 Team 主表

短期建议：
- **不新增** `team_definitions` / `team_memberships` 之类的主表；
- 继续复用：
  - `team_workspaces`
  - `workflow_templates`
  - `sessions.metadata_json`

### 为什么现在不建议新表
- 你的目标首先是“把创建流走通”，不是先建设一套 Team 领域模型；
- 现在最缺的是 canonical DTO 和 gateway 裁决点，不是存储空间；
- 一旦新表先上，前端/模板/runtime 之间很容易出现第二套真相。

### 什么时候才值得新增 `team_definitions` 表
只有当后续出现下面任一需求时，再考虑：
- team 需要跨多个 session 复用且独立编辑；
- team 需要单独分享、权限控制、版本管理；
- session 需要引用“某个活动中的 team 实体”而不是 creation snapshot。

### 模板表的最小扩展建议

当前 `workflow_templates` 只有：
- `name`
- `description`
- `category`
- `nodes_json`
- `edges_json`

建议最小扩一列：

```sql
ALTER TABLE workflow_templates ADD COLUMN metadata_json TEXT NOT NULL DEFAULT '{}';
```

用于存团队模板专属 metadata，例如：

```json
{
  "teamTemplate": {
    "requiredRoles": ["planner", "researcher", "executor", "reviewer"],
    "defaultProvider": "claude-code",
    "optionalAgentIds": ["atlas"],
    "defaultBindings": {
      "planner": "oracle",
      "researcher": "librarian",
      "executor": "hephaestus",
      "reviewer": "momus"
    }
  }
}
```

这样 `team-playbook` 模板就不需要只靠 `nodes/edges` 猜角色。
