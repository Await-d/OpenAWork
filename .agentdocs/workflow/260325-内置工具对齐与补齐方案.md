# 内置工具对齐与补齐方案

## Task Overview
在已完成 `websearch/list/read/grep/write/todowrite/todoread` 默认命名对齐的基础上，继续按参考库补齐缺失的内置工具；本批次优先实现高确定性的 `glob`，并以 BDD + TDD 方式推进，避免直接引入高风险语义漂移。

## Current Analysis
- 当前模型默认可见工具名已收敛到参考风格，但缺失工具仍包括 `glob`、`webfetch`、`bash`、`task`、`edit`、`apply_patch`、`skill`、`question`、`batch` 等。
- 其中 `glob` 与当前工作区安全模型最匹配：可复用 `workspace-paths`、agentignore、防越权限制。
- `webfetch` 需要确认格式转换语义（尤其 HTML → Markdown），风险高于 `glob`，但仍属于可控二批次。
- `bash/task/edit/apply_patch/skill` 涉及权限、安全边界或复杂产品语义，后续需要单独设计与更强测试覆盖。
- 目前已补齐并验证：`glob`、`webfetch`、`edit`。
- 目前已补齐并验证：`glob`、`webfetch`、`edit`、`batch`、`skill`、`bash`、`apply_patch`、`question`、`task`。
- `edit` 已按 OpenCode 风格接入：`filePath + oldString + newString + replaceAll?`，带“先 read 后 edit”检查与审批。
- `apply_patch` 仍缺 parser/harness，不适合在没有明确 patch 语言与失败恢复设计前仓促接入。
- `question/skill/task/batch/bash` 仍需结合现有产品交互与权限模型决定优先级。

## Solution Design
- Phase 1：补齐 `glob`
  - 目标：默认对外暴露 `glob`，并在 sandbox 中实际可执行。
  - 语义：限定工作区内搜索，遵守 agentignore，支持 `path + pattern`，返回匹配路径列表。
- Phase 2：补齐 `webfetch`（仅在格式语义足够明确时落地）
  - 目标：支持 `url + format(text|markdown|html)`。
  - 约束：默认只读、支持超时、避免把“伪 markdown”冒充真正转换结果。
- Phase 3：高风险工具单独设计
  - `bash/task/edit/apply_patch/skill/question/batch` 进入下一轮，按权限模型与参考语义分别建模。

## BDD Scenarios

### GLOB-01 工作区文件匹配
- Given 工作区内存在多层目录与多种扩展名文件
- When agent 调用 `glob` 并传入 `path` 与 `pattern`
- Then 返回仅包含工作区内、符合模式且未被 agentignore 屏蔽的路径列表

### GLOB-02 默认暴露名与能力目录一致
- Given Gateway 构建默认工具定义与 capabilities
- When 系统列出 runtime tools
- Then `glob` 对模型与能力目录都可见，且不会与 reference-only 条目重复展示

### GLOB-03 越权路径被拒绝
- Given 用户传入工作区外部路径
- When 调用 `glob`
- Then 工具返回错误而不是泄露外部文件结构

### WEBFETCH-01 明确格式返回（待二批次）
- Given 远端 URL 可访问
- When 调用 `webfetch` 并指定 `format`
- Then 返回与格式一致的正文内容，且超时/失败时给出结构化错误

### EDIT-01 先读后改
- Given 同一会话没有成功 read 过目标文件
- When 调用 `edit` 且 `oldString` 非空
- Then 工具拒绝执行，并提示必须先读取文件

### EDIT-02 单次精确替换
- Given 同一会话已成功 read 过目标文件
- When 调用 `edit(filePath, oldString, newString)`
- Then 仅替换第一处精确命中，写回文件并保留审批链路

### EDIT-03 多处命中保护
- Given `oldString` 在文件中出现多次
- When 未设置 `replaceAll`
- Then 工具拒绝执行，要求提供更多上下文或显式打开 `replaceAll`

### EDIT-04 空 oldString 直接创建/覆写
- Given `oldString === ''`
- When 调用 `edit`
- Then 工具允许直接创建/覆写文件，并仍然走权限审批

## TDD Plan
1. 先写 `tool-definitions` / `tool-sandbox` / `capabilities` 的 `glob` 红灯测试。
2. 最小实现 `glob` 工具并接入 runtime definitions、sandbox、capabilities。
3. 跑定向测试、LSP、build 变绿。
4. 再评估 `webfetch` 是否进入本轮；若格式语义不够稳，停在设计与测试草案，不仓促实现。
5. `edit` 按 OpenCode 风格补齐：先写红灯测试，再接入 runtime definitions、sandbox 权限与“先 read 后 edit”检查。
6. 对剩余高风险工具继续并行探索，若语义/权限不清，则在证据收敛后只问最小必要问题。

## Notes
- 当前轮以“高确定性可交付”为原则，不为追求数量牺牲工具语义一致性。
- 已完成：`glob` 与 `webfetch` 已接入默认模型可见工具、sandbox 执行链路与 capabilities 展示，并通过定向测试、LSP 与 build 验证。
- Oracle caveat 1：`glob` 当前是受限版 glob 语义（`* / ** / ?`），返回绝对路径，后续若要完全对齐参考库可再扩展排序与模式能力。
- Oracle caveat 2：`webfetch` 成功态返回结构化对象；失败态目前仍经 sandbox 标准错误字符串回传，后续若要完全对齐参考描述可再统一错误输出形态。
- Oracle caveat 3：`GLOB-03` 的越权拒绝应在后续补一条 `glob` 专项回归测试。
- 已补：`GLOB-03` 越权拒绝回归测试、`web_search` 旧名 sandbox 兼容执行回归测试。
- `edit` caveat 1：当前“先 read 后 edit”通过 `audit_logs` 回看成功 `read/workspace_read_file` 事实来实现，且已修复为按真实执行会话 `sessionId` 检查；满足最小闭环，但还不是 OpenCode 那种更细粒度的文件版本一致性检查。
- `edit` caveat 2：当前已对齐核心输入与行为，但输出仍采用 OpenAWork 结构化对象而非参考实现的纯文本 diff 输出。
- `batch` caveat：当前禁止 `batch` 自调用与 `mcp_*`，并限制 25 个调用；返回结构化结果数组，但尚未覆盖全部 reference 文案边角行为。
- `skill` caveat：当前优先加载已安装技能，并通过 registry/cache 的 `manifestUrl` 拉取真实 `SKILL.md`；对缺少原文来源的 builtin 技能退化为结构化摘要。
- `bash` caveat：当前是强护栏版 shell——要求审批、限制工作区 cwd、拒绝高危 shell 模式；尚未做完整 shell 语法支持。
- `apply_patch` caveat：当前 parser/harness 支持 `Add File / Update File / Delete File / Move to` 的最小集合，update hunk 走精确文本替换，不含更复杂 diff 语义。
- `question` caveat：当前已具备持久化问题请求、回答接口与恢复入口，但尚未补专门的路由/恢复集成测试。
- `task` caveat：当前已创建真实 child session 并持久化 task graph 入口，但还不是完整“自动执行并回传 <task_result>”的子代理生命周期。
