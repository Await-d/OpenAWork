## 任务概览

基于已确认技术栈（Expo + Tauri + TypeScript Monorepo + Agent Gateway），输出一份可直接执行的详细任务计划，目标是在小团队（3-6人）条件下，12周内交付可上线 MVP，并为 Beta 阶段的混合推理能力预留扩展路径。

## 当前分析

- 当前仓库无现有研发文档框架（无 `docs/` / `.agentdocs/workflow` 既有模板），可直接建立标准计划结构。
- 外部最佳实践已覆盖关键风险点：
  - Expo 开发构建与 EAS 工作流
  - Tauri 签名/更新发布链路
  - WebSocket 心跳与断连治理
  - 客户端安全存储边界
  - 跨端测试矩阵与 CI 质量门禁
- 核心挑战不是“能否开发”，而是“如何稳定交付”：需要从 Day 1 固化质量门禁、发布门禁与回滚策略。

## 方案设计

### 目标与边界

**目标（12周）：**

1. 交付 iOS/Android + macOS/Windows 可用的统一体验客户端
2. 交付可流式响应、可恢复会话、可观测、可发布的 AI 智能体 MVP
3. 建立可持续迭代基线（自动化测试、灰度发布、回滚）

**本阶段不做：**

- 不在 MVP 强行引入大规模端侧推理
- 不做复杂多 Agent 图编排平台化（保留接口，后续演进）

### 角色分工（建议）

- Tech Lead（1）：架构、质量门禁、发布治理
- Client Engineer（2）：Expo / Tauri 客户端实现
- Backend Engineer（1）：Gateway + Tool 执行层 + 观测
- QA/全栈测试（1，可兼职）：E2E、回归、发布验收

## 风险矩阵与控制措施

1. **多端发布复杂度高（高概率/高影响）**
   - 控制：W1 即接入 CI 发布演练；W10 前完成签名链路打通

2. **WebSocket 连接稳定性不足（中概率/高影响）**
   - 控制：强制心跳、断连重连策略、网关侧连接清理

3. **客户端密钥泄露风险（中概率/高影响）**
   - 控制：端上仅存短 token；密钥只在服务端；日志默认脱敏

4. **端侧能力差异导致体验不一致（高概率/中影响）**
   - 控制：统一协议 + UI token；核心逻辑放 `agent-core`

5. **自动更新失败导致版本割裂（中概率/中影响）**
   - 控制：灰度发布 + 版本门禁 + 一键回滚
