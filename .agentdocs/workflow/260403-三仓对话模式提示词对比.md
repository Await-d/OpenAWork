# 三仓对话模式提示词对比

## Task Overview

对比 `temp/opencode`、`temp/oh-my-opencode`、`temp/OpenCowork` 三个本地参考仓库中与模式提示词、agent prompt、system prompt、消息前缀注入相关的实现方式，重点分析 OpenCowork 的 prompt 配置机制，并将 OpenAWork 的 `clarify / coding / programmer` 三模式从前端文本前缀拼接收口为 **gateway 单点注入**。

## Current Analysis

- 当前 OpenAWork 已完成三仓调研，并确认最值得借鉴的是 OpenCowork 的“稳定 system prompt + request-scoped mode reminder”分层体系。
- Gateway 现已作为模式提示词的唯一正文注入点：`services/agent-gateway/src/routes/stream-system-prompts.ts`。
- Web / Mobile 前端已移除模式 prompt 正文与发送前拼接逻辑，仅保留 UI 元数据与默认 agent 映射。
- 结构化合同已统一为 `dialogueMode / yoloMode` 请求字段，并同时覆盖：
  - gateway `stream.ts` / `stream-runtime.ts`
  - web hook / mobile hook
  - `packages/web-client` SSE / WS 客户端
- 用户明确要求借鉴三个参考仓库，且特别关注 `temp/OpenCowork` 的提示词配置方式。
- 参考仓库调研已完成：
  - `OpenCowork` 采用“稳定 system prompt + mode reminder 注入 + provider instructions prompt + sub-agent initial/system prompt”的分层体系。
  - `opencode` 更偏向 `.opencode/agent/*.md`、`.opencode/command/*.md` 与 `instructions` 文件化配置。
  - `oh-my-opencode` 更强调 `prompt / prompt_append`、Hook 注入、目录级 `AGENTS.md` / README 自动注入与运行时拼装。
- 用户已明确同意实施改造，并进一步要求“最好放到网关里面，这样只需要调整一个地方都可以生效”。该要求已经落地。

## Solution Design

- 在 gateway 中集中维护模式正文：`DIALOGUE_MODE_SYSTEM_PROMPTS` 与 `YOLO_MODE_SYSTEM_PROMPT` 只保留在 `stream-system-prompts.ts`。
- 统一请求合同为结构化字段 `dialogueMode / yoloMode`，由 gateway 在 request-scoped system prompt 阶段注入。
- 保留前端的 mode UI 元数据与默认 agent 映射，但删除 prompt 正文定义与 `apply*ToMessage()` 发送前拼接逻辑。
- Web / Mobile / `packages/web-client` 统一通过结构化字段透传模式状态，而不是在用户消息前塞 `<system-reminder>` 文本。
- 更新测试，改为验证：
  - gateway mode/yolo prompt 注入顺序与内容
  - web/mobile/web-client 透传结构化字段
  - ChatPage 发送原始用户消息，同时带上 `dialogueMode` 与默认 agent
