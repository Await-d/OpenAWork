# 260331 聊天刷新恢复与运行控制方案

## Task Overview

用户要求围绕 Chat 对话页创建一份正式方案，解决以下体验断点：

1. 会话仍在运行时继续发送消息会进入待发送队列，但刷新后队列消失；
2. 刷新后仍能看到会话处于 running / paused，却无法继续 stop 当前运行；
3. 界面缺少明确的状态管理与恢复语义，用户离开页面再回来后难以判断“能否继续控制这次运行”。

本方案目标不是立即改代码，而是冻结一套可执行的产品 + 前端 + 网关协同方案，供后续按阶段实施。

## Current Analysis

- `apps/web/src/pages/ChatPage.tsx` 当前把 `queuedComposerMessages` 作为本地 `useState` 管理，刷新后天然丢失；切换会话时也会主动清空。
- `apps/web/src/hooks/useGatewayClient.ts` 在发起流式请求时生成 `clientRequestId`，并仅保存在 `activeRequestRef` 内存中；`stopStream()` 依赖这块内存态。
- `services/agent-gateway/src/routes/stream.ts` 会把 `sessions.state_status` 持久化为 `running / paused / idle`，并在 `session_runtime_threads` 中记录 `client_request_id + heartbeat`；因此刷新后前端可以恢复“会话仍在运行”的观察状态。
- 真正的 stop 句柄仍在 `services/agent-gateway/src/routes/stream-cancellation.ts` 的进程内 `Map<sessionId:clientRequestId, AbortController>` 中；若前端丢失 `clientRequestId`，就无法精确停止。
- `apps/web/src/components/chat/ChatComposer.tsx` 现在已隐式区分“本地仍可 stop”和“远端 session busy 但不可 stop”，但文案与按钮层级没有把这种差异解释清楚。

## Solution Design

### 核心决策

1. **把运行状态与控制能力拆开建模**：
   - `runtimeState`：`idle | running | paused`
   - `stopCapability`：`precise | best_effort | observe_only`
   - `queueState`：`empty | ready | needsReattach | blockedByRuntime`
2. **待发队列采用客户端分层持久化**：Phase 1 仅持久化可序列化字段（文本、附件元信息、创建时间、session 归属、`needsReattach`），不直接持久化 `File[] / Blob`。
3. **停止能力采用双轨模型**：
   - 精确停止：继续走现有 `/sessions/:id/stream/stop + clientRequestId`
   - 尽力停止：新增会话级 stop-active 路由，供刷新后无句柄场景使用
4. **复用现有轮询与恢复链路**：继续依赖 `loadCurrentSessionSnapshot()` 与 `loadSessionRuntimeSnapshot()` 恢复观察态，不重新发明一套运行同步机制。
5. **UI 必须显式表达“当前页是否接管了本次运行”**：避免用户看到 running 就误以为还能精确停止。

### 方案边界

- Phase 1 不做跨设备精确接管；目标仅覆盖“同一浏览器刷新 / 离开后回来”的恢复体验。
- Phase 1 不做附件 Blob 自动恢复；带附件的待发项以“文本恢复 + 附件需重新选择”降级处理。
- 不把本地待发队列直接作为后端主事实；后端继续专注 session runtime 真相源。

### 现有可复用实现

- `apps/web/src/stores/auth.ts` 与 `apps/web/src/stores/uiState.ts` 已使用 Zustand `persist`，并带有 `name / version / migrate / partialize` 模式，可直接作为聊天恢复状态的前端持久化参考。
- `packages/web-client/src/gateway-ws.ts` 与 `packages/web-client/src/gateway-sse.ts` 的发送接口已支持可选 `clientRequestId`，说明“前端自行恢复并复用请求 ID”在客户端 API 层并不需要额外扩容。
- `services/agent-gateway/src/routes/stream-cancellation.ts` 已实现 `stopAnyInFlightStreamRequestForSession()`，且 `services/agent-gateway/src/__tests__/stream-cancellation.test.ts` 已覆盖会话级 stop-any 行为，适合作为 best-effort stop 的后端基础。
- `apps/web/src/pages/ChatPage.test.tsx` 已覆盖三类关键现象：同一 SPA 生命周期内切走再切回仍可 stop、待发队列在当前页内可排队/自动 flush、以及仅恢复远端 `running` 状态时主按钮会退化为禁用的“运行中”。这些现有测试可直接扩展为刷新恢复场景回归用例。

### 后端能力边界（基于补充搜索收敛）

- `services/agent-gateway/src/session-runtime-thread-store.ts` 已具备 request-scoped runtime thread 心跳（5s heartbeat / 20s stale）；适合继续承担“观察态”恢复真相源。
- `services/agent-gateway/src/session-run-events.ts` + `services/agent-gateway/src/routes/stream.ts` 已具备 request-scoped durable run events；但当前 replay 主要在存在 terminal 事件时才稳定成立，不等于“中途刷新后自动续流”。
- `services/agent-gateway/src/routes/stream-routes-plugin.ts` 当前仅对外暴露 precise stop：`POST /sessions/:id/stream/stop`（必须带 `clientRequestId`）。
- `stopAnyInFlightStreamRequestForSession()` / `stopAllInFlightStreamRequestsForSession()` 目前主要用于 session delete / task cancel 等内部链路；因此 best-effort stop 的核心缺口不是实现缺失，而是**没有产品化为统一对外控制面**。
- `services/agent-gateway/src/session-runtime-state.ts` 与 `session-runtime-reconciler.ts` 已经清晰定义了 observe-only 的后端语义：refresh/重启后即使无法精确控制，也能把 `running / paused / idle` 对账到合理状态。

### 外部案例对照（已完成）

#### 运行恢复 / 控制面

- **Vercel AI SDK**：把 `chatId` 作为 stream resume 主键，说明“观察态恢复”应该绑定会话/线程 ID，而不是绑定当前页面实例；但其官方明确指出 resumable stream 与 abort/stop 存在天然冲突，因此 stop 不能只依赖浏览器 AbortSignal。
- **Microsoft Direct Line**：用 `conversationId + watermark` 分离“会话事实”和“流位置游标”，适合借鉴为 OpenAWork 后续的 request/event offset 恢复模式。
- **vLLM**：把 `request_id` 作为一等公民，并提供显式 abort API，强烈支持本方案对 precise stop 的定义：可精确控制必须绑定 request/run 级 ID，而不是仅依赖会话状态。
- **OpenAI Realtime reference client**：把 cancel 视为独立控制命令（`response.cancel`），说明 stop 最稳的产品方向是“显式控制面协议”，而不是隐含在数据流断开中。

#### 草稿 / 队列 / 附件恢复

- **Mattermost**：草稿里保存 message + 附件引用/上传状态，但同步时只提交 `file_ids`，不是持久化 `File` 本体；说明“附件引用”和“附件二进制”应该分层。
- **Zulip**：草稿模型只持久化文本与收件人上下文，没有把附件纳入草稿主结构；说明文本草稿恢复完全可以先独立落地。
- **Signal Desktop**：能恢复附件，但依赖“落盘 + 路径引用”，不是把 `File/Blob` 直接塞进状态树；这更适合桌面端，不适合作为 Web Phase 1 默认方案。
- **localForage / IndexedDB 文档**：Blob 可存，但存在 driver/fallback/体积膨胀与配额问题；因此 Web 端附件恢复应放到 Phase 2/3，而不是阻塞当前方案落地。

### 最终推荐路线

1. **Phase A（本次方案主张）**：以客户端持久化 + 后端对账为主，先解决“刷新后看得见、解释得清、文本队列不丢”。
2. **Phase B（最小控制增强）**：在 Gateway 暴露 `stop-active`（best-effort），让刷新后无 `clientRequestId` 的页面仍能管理该 session 的当前运行。
3. **Phase C（可选增强）**：如果后续业务明确要求“刷新后附件也可直接重发”或“跨标签页继续精确 stop”，再分别进入 IndexedDB 附件层或 runtime control token 设计。

### 关键设计结论

- **观察态（running/paused/idle）和控制态（precise/best_effort/observe_only）必须分开表达。**
- **`clientRequestId` 不应该只活在 React ref 里**；至少要有“当前标签页可恢复”的最小快照层。
- **best-effort stop 是当前仓内最具性价比的补强点**：实现基础已存在，只差对外协议与前端 UI 映射。
- **不要在 Phase 1 持久化 `File[]`**；Phase 1 只承诺文本与附件元信息恢复，否则会把方案从“可快速落地”拖成“重存储工程”。

## Acceptance Criteria

- 刷新当前正在进行中的会话后，纯文本待发队列仍可见，不会静默消失
- 刷新后若当前页失去精确句柄，界面仍能明确显示“可尝试停止”或“仅可观察”
- 后端能提供会话级 best-effort stop，前端不会再把 `running` 误显示为可精确 stop
- 带附件的待发项刷新后不会假装可直接发送，而是明确要求重新选择附件

## Notes

- 当前方案优先解决“恢复后可管理”而非“恢复后精确续流”；若后续要求跨标签页/跨设备精确接管，再进入更高复杂度的 runtime control token 设计。
- 当前仓内已有的最佳复用模式是：前端用 Zustand `persist` 保存可序列化 UI 状态，后端用 request-scoped `clientRequestId` + session runtime thread 持续表达运行事实；本方案应顺着这两条既有路径扩展，而不是重造状态中心。
- 经过四路补充搜索后，方案已正式收敛：**前端 Phase 1 用 Zustand persist / sessionStorage 恢复最小状态，后端新增 stop-active 补齐控制面，UI 显式表达 stopCapability。**
- Oracle 复核后的两条阻断问题已消化：1) 刷新后的 precise stop 不再卡在“停止中”；2) 陈旧 active snapshot 在 stop 失败后会清理并回拉会话状态，避免持续显示假可控。
- 后续补强已落地：1) 自动 flush 现在要求当前会话快照已可靠建立，刷新后快照未知时不会误发送队列；2) `useGatewayClient` 已新增“恢复 snapshot 成功 stop 后清理”和“陈旧 snapshot stop 失败后降级失效”的 hook 级测试覆盖。
- 附件恢复增强已落地：刷新后若 IndexedDB 中仍有对应二进制，队列项可恢复为真正可发送附件；只有二进制缺失或环境不支持时才提示重新选择附件。
