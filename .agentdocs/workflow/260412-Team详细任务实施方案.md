# Team 详细任务实施方案

## Task Overview

为 OpenAWork 的 Team 重整创建一份可执行的详细任务实施方案，目标不是继续讨论方向，而是把已冻结的 Team / Personal / Shared 三域边界、multica 理念吸收结论、以及当前 Team 实现偏差，收束成后续开发可以直接照着执行的阶段计划。

本方案服务于三个目标：

1. 先把 Team 从 `session/sharedSession` 投影层里拔出来
2. 再把 multica 的核心理念落入 OpenAWork 自己的 TeamWorkspace 模型
3. 最后才恢复前门 interaction-agent 与页面深化实现

## Complexity Assessment

- Atomic steps: 5+（架构收束、阶段拆分、模块批次、验证矩阵、风险控制） → +2
- Parallel streams: yes（gateway / web-client / web state / multica 理念吸收） → +2
- Modules/systems/services: 3+（Team docs / web / gateway / web-client / temp/multica） → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1

- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 这是一个跨文档、跨模块、跨阶段的实施方案任务，需要把多份设计文档压缩成单一执行计划，并为后续开发提供明确依赖与验收基线。

## Current Analysis

### 当前已确认事实
- 当前 `/team` 仍然是 Team 自有数据 + `sessions/sharedSessions/sessionShares/runtimeTaskGroups` 的混合投影层。
- 当前 Team 的主状态仍然 heavily 依赖：
  - `selectedSharedSessionId`
  - `sharedSessions`
  - `workspacePath`
- 当前 Team 创建路径仍然存在写回普通 `/sessions.create` 的错误倾向。
- interaction-agent 当前还只是轻量入口脚手架，并没有形成真正的“分析需求 → 拆解任务 → 派发团队执行”主链。

### 当前已冻结的目标共识
- Team / Personal / Shared 三域分离
- `TeamWorkspace` 成为 Team 一级根对象
- `TeamThread / TeamTask / TeamReview / TeamArtifact` 成为 Team-owned 主业务对象
- Shared / Bridge 只负责显式分享 / 引用 / 导入
- 办公室视图只读展示，不承担管理语义

### 当前需要吸收的 multica 理念
- Agent 是正式队友
- 控制面 / 执行面分离
- 任务驱动
- workspace 硬边界
- thread / task 双轨并存
- skills compound

## Implementation Goals

### G1. 建立 Team 的 canonical write model
让 Team 的创建、更新、执行与评审都挂到 Team 自己的对象图上，而不是继续借用普通 `session`。

### G2. 建立 Team 的 canonical read model
让 `/team/:teamWorkspaceId` 的所有主面板都可以从 Team-owned snapshot 与 Team 子域资源读取，而不是继续拼 session/sharedSession 投影。

### G3. 为 interaction-agent 和多 agent 协作链创造正确挂点
interaction-agent 必须建立在正确的 Team 边界之上，不能作为修复旧状态模型的替代物。

## Non-Goals

- 本阶段不继续做办公室视图的视觉深化
- 本阶段不做 issue/board 风格 UI 迁移
- 本阶段不复制 multica 的 daemon-first 执行 substrate
- 本阶段不把 interaction-agent 提前升级为完整前门调度中枢

## 外部实施顺序校准（来自 multica 周边理念与其他产品/OSS 迁移模式）

### 原则 1：先做兼容导入态，不直接切主模型
- 外部强信号：Google Chat import mode 先把空间建成“最终用户不可见”的导入态，再在该窗口内完成历史 membership 导入。
- 对 OpenAWork 的含义：
  - 在 Team 读写主链切换前，应允许 TeamWorkspace 先以“兼容态/桥接态”存在
  - 不要一刀切地把旧 shared-session 投影直接替换成新 Team 模型

### 原则 2：先分清 workspace 边界，再分清 session / thread 边界
- 外部强信号：OpenClaw 明确把 `workspace/agent` 与 `sessionKey`、`dmScope` 分开建模。
- 对 OpenAWork 的含义：
  - 先把 `TeamWorkspace` 作为一级边界立起来
  - 再把 `TeamThread` 从普通 session 里拆出来
  - 不要先做线程替换，却让 workspace 身份仍然依赖 `workspacePath` 推导

### 原则 3：采用 expand → backfill → cutover → cleanup 顺序
- 外部强信号：Zulip 的多条 migration 都是先加结构/索引，再写回填脚本，最后切换读取。
- 对 OpenAWork 的含义：
  - 先新增 Team canonical contract
  - 再回填 Team 读模型 / TeamSnapshot
  - 然后切换页面主读取来源
  - 最后删除旧 `selectedSharedSessionId` 投影链

### 原则 4：兼容字段/兼容路由要留到最后一个消费者迁完
- 外部强信号：Mattermost 仍保留 deprecated 字段与 route 层兼容，直到所有消费者完成迁移。
- 对 OpenAWork 的含义：
  - `/team/runtime`、`/team/messages`、`/team/tasks`、`/team/session-shares` 可短期保留为过渡层
  - 但新 contract 必须逐步向 `teamWorkspaceId` 对齐
  - 不要在消费者未迁完时过早删旧接口

### 原则 5：membership 和内容迁移不要绑成同一刀
- 外部强信号：Google Chat 导入明确把历史 memberships 作为独立约束步骤。
- 对 OpenAWork 的含义：
  - `TeamMember / TeamRoleBinding` 的迁移或冻结应单独成阶段
  - 不要把成员/角色迁移和 thread/task/message 主链切换绑成一个大批次

## 外部原则对应到 OpenAWork 的直接顺序

```text
兼容 TeamWorkspace 结构
-> 新增 Team canonical contract
-> 回填 TeamSnapshot / Team 子域读模型
-> 单独校准 TeamMember / TeamRoleBinding
-> 切前端主读取来源
-> 退役旧 selectedSharedSessionId 投影链
```
