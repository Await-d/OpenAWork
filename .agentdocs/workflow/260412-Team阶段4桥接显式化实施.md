# Team 阶段4桥接显式化实施

## Task Overview

在 Stage 1 已完成 TeamWorkspace 根边界、Stage 2 已完成最小 Team-owned 写路径、Stage 3 已完成 workspace snapshot 主读链切换的基础上，继续推进 **Stage 4：Bridge 显式化**。本阶段目标是把 `sharedSessions / selectedSharedSessionId / shared-with-me` 相关能力明确降级为 bridge/import/detail 层，不再承担 Team 主导航与默认选中。

## Current Analysis

- Stage 3 已完成：Team 主读链已经开始优先消费 `TeamWorkspaceSnapshot`
- 当前仍未完成的部分：
  - `useTeamCollaboration()` 仍保留 shared-session detail rail 与 `selectedSharedSessionId`
  - `team-runtime-reference-data.tsx` 仍保留 shared session 作为一部分默认选中回退链
  - Team 页面尚未出现显式的 import/share panel 模型

## Solution Design

- Stage 4 只做 bridge 显式化，不提前推进 Stage 5 页面热点与视觉深化。
- 最小实施顺序：
  1. 先补 bridge/import 的后端最小显式入口
  2. 再补 web-client 对应契约
  3. 最后把 Team 页面中 sharedSessions 的角色降级到 bridge/import/detail 面板
- 本阶段不碰：
  - 办公室画布深化
  - interaction-agent 主流程
  - 页面视觉细节重排

## Complexity Assessment

- Atomic steps: 5+（gateway / web-client / web bridge 入口 / 验证 / git） → +2
- Parallel streams: yes（gateway/client/frontend bridge 切口探索可并行） → +2
- Modules/systems/services: 3+（gateway / web-client / web） → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: bridge 显式化涉及 route、client、UI 降级与状态约束，且必须和 Stage 3 主读链保持兼容，因此需要独立工作流与运行态总表。
