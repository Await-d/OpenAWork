# .agentdocs/workflow/260402-对话模式默认agent方案.md

## Task Overview

为 OpenAWork 设计一套“**对话模式派生默认 agent，并允许用户手动覆盖**”的正式实施方案。目标行为为：`coding → sisyphus-junior`、`programmer → hephaestus`，同时保持 **手动覆盖 > 模式默认 > 现有默认** 的稳定优先级。

**Phase 1 范围冻结：**
- **Web：** 完整实现“模式默认 agent + 手动覆盖优先”。
- **Gateway：** 完整实现可选 `agentId` 协议、恢复链兼容、降级与可观测性。
- **Mobile：** 仅保证**活跃 Expo Router 路由**与协议兼容，不承诺在 Phase 1 达成 `coding/programmer` 默认 agent 行为。

## Current Analysis

- 当前 `DialogueMode` 只存在于前端提示词链，而不是结构化 agent 选择链：
  - Web：`apps/web/src/pages/dialogue-mode.ts` + `apps/web/src/pages/ChatPage.tsx`
  - Mobile（legacy）：`apps/mobile/src/components/DialogueModeSelector.tsx` + `apps/mobile/src/screens/ChatScreen.tsx`
- 当前流式请求协议没有 `agentId`：
  - `services/agent-gateway/src/routes/stream.ts` 的 `streamRequestSchema`
  - `apps/web/src/hooks/useGatewayClient.ts`
  - `apps/mobile/src/hooks/useGatewayClient.ts`
- Gateway 当前会把 `系统 Agents` 注入 capability context（`services/agent-gateway/src/routes/capabilities.ts`），但这只是能力目录，不等价于本轮聊天已绑定具体 agent。
- 当前“现有默认”并不是 agent 语义，而是现有的 **provider/model/variant/systemPrompt 解析链**：请求字段 → session metadata → user active selection / provider default。
- Web 侧当前发送链的关键点：
  - `apps/web/src/pages/ChatPage.tsx:2261-2270` 先计算 `requestText`，再 `client.stream(...)`
  - `apps/web/src/pages/chat-page/support.ts:865-924` 会从 `metadata_json` 恢复 `dialogueMode/yoloMode/webSearch/thinking/provider/model`
  - 目前没有 `selectedAgentId` 的结构化发送链。
- Mobile 侧当前存在一主一辅两条实现，但**当前活跃入口**是 Expo Router：
  - **活跃路径：** `apps/mobile/app/chat/[sessionId].tsx`
  - **legacy 参考实现：** `apps/mobile/src/screens/ChatScreen.tsx`
  - 二者都经过 `apps/mobile/src/hooks/useGatewayClient.ts`，而当前 payload 仅 `{ clientRequestId, message }`。
- Gateway 侧已有可借鉴但不可直接复用的 agent 路由基础：
  - `services/agent-gateway/src/agent-catalog.ts`：managed agent 来源与 enabled 判断
  - `services/agent-gateway/src/task-agent-resolution.ts`：按 id/label/alias 匹配 agent 的现有思路，但带有 delegated child session 语义
  - `services/agent-gateway/src/task-model-selection.ts`：按候选模型选择 provider/model 的现成策略，但输出形态与普通 chat stream 不完全一致

## Solution Design

### 核心决策

1. **采用 stream 协议扩展，而不是仅靠前端拼文本暗示 agent。**
2. **新增可选 `agentId`**：旧客户端不传时保持现状，新客户端可显式指定本轮请求使用的 agent。
3. **现有默认明确定义为：** 当前 `providerId/model/variant/systemPrompt` 解析链，不包含任何新的 agent 推导逻辑。
4. **模式默认仅提供候选值，不得覆盖用户显式选择。**
5. **无效/禁用 agentId 安全降级**：忽略该字段，回退到现有 provider/model/systemPrompt 解析逻辑，不直接让整条请求失败。
6. **Phase 1 不把 agentId 持久化进 session metadata 作为强依赖。** 手动覆盖先做请求级/UI 级状态；若后续要求跨刷新/跨设备保持，再开启 Phase 2 持久化。
7. **Phase 1 保留现有 mode 文本 prompt 注入逻辑。** `dialogueMode` 继续影响用户消息前缀；mode-derived agent 只作为额外的结构化路由信号。

### 合同冻结

#### 1. Effective Agent 优先级

```text
effectiveAgentId = manualSelectedAgentId ?? modeDefaultAgentId ?? undefined
```

- `manualSelectedAgentId`：用户显式选择的 agent
- `modeDefaultAgentId`：由 `dialogueMode` 推导出的默认 agent
- `undefined`：不发送 `agentId`，完全回退到现有默认

#### 2. Manual Override 生命周期

- **Web Phase 1：** 手动覆盖为**当前会话页面内的 UI 状态**。
  - 切换会话时重算
  - 显式点击“清除手动覆盖”时失效
  - **刷新页面后不保证保留**
- **Mobile Phase 1：** 不提供完整的手动覆盖 UI，只保证协议链可透传 `agentId`
- **Phase 2 才考虑**：把手动覆盖持久化到 session metadata，实现跨刷新/跨设备一致性

#### 3. Clarify / Coding / Programmer 的具体值

- `clarify` → `modeDefaultAgentId = undefined`，不发送 `agentId`
- `coding` → `modeDefaultAgentId = 'sisyphus-junior'`
- `programmer` → `modeDefaultAgentId = 'hephaestus'`

### `agentId` 覆盖矩阵（Phase 1 冻结）

| 配置项 | 优先级规则 | 说明 |
| --- | --- | --- |
| `providerId` | 显式 request 字段 > agentId 派生候选 > session metadata > 当前现有默认 | `agentId` 只提供候选，不得覆盖显式 request |
| `model` | 显式 request 字段 > agentId 派生候选 > session metadata > 当前现有默认 | 与 `providerId` 同步处理 |
| `variant` | 显式 request 字段 > agentId 派生候选 > session metadata > 当前现有默认 | 仅在 agent 侧有定义时生效 |
| `systemPrompt` | delegated child-session prompt > 显式 request 字段 > agentId 派生 prompt > session metadata > 当前现有默认 | 普通 `agentId` **不能压过** delegated 语义 |
| mode 文本 prompt | Phase 1 始终保留；与 agentId 并行生效 | 这是兼容策略，不是最终长期方案 |

#### 特殊规则

- 如果当前会话是 delegated child session，且 metadata 已存在 `delegatedSystemPrompt` / delegated model 语义，则 **忽略普通聊天 `agentId` 对 systemPrompt 的覆盖**。
- 如果 `agentId` 对应的 model/provider 在用户当前 provider 集合中不可用，则只允许：
  - 注入 agent 派生 `systemPrompt`（如果安全）
  - 或完全回退到现有默认
  - **不得让请求因 agent 不可用直接失败**

### Resume / Background 的事实源

Phase 1 中，`agentId` 的恢复事实源不是 session metadata，而是**请求级 requestData**：

- WS / SSE 进入 `streamRequestSchema` 后的 `requestData`
- permission resume 持久化的 `request_payload_json`
- question resume 持有的 request payload
- background / task-parent-auto-resume 继续使用的 requestData

因此 Phase 1 的硬要求是：

1. `streamRequestSchema` 接受 `agentId`
2. `permissionResumeRequestSchema` 接受 `agentId`
3. `stream-runtime.ts` 的 resume/background 链不丢 `agentId`
4. 如果恢复链找不到 `agentId`，则明确视为“无结构化 agent 选择”，回退到现有默认

### 协议合同示例

#### 1. 新版 WS 请求示例

```json
}
```

### 推荐数据流

#### Web（Phase 1 完整行为）
- `dialogueMode` 继续按现有方式保存在 `ChatPage` 与 session metadata 中。
- 新增前端状态：
  - `manualSelectedAgentId: string | null`
  - `modeDefaultAgentId: string | null`
  - `effectiveAgentId = manualSelectedAgentId ?? modeDefaultAgentId ?? undefined`
- 发送时：`client.stream(sid, requestText, { ..., agentId: effectiveAgentId })`
- 提供“清除手动覆盖”入口，回退到 mode default。
- Phase 1 不要求跨刷新保持手动覆盖。

#### Mobile（Phase 1 协议兼容）
- 当前活跃路由 `app/chat/[sessionId].tsx` 只接入 request 级 `agentId` 透传能力。
- **Phase 1 不承诺** 在 Mobile 活跃路由实现 `coding/programmer` 默认 agent 行为。
- legacy `src/screens/ChatScreen.tsx` 视为参考实现，不作为 Phase 1 必达范围。

#### Gateway（Phase 1 完整行为）
- `streamRequestSchema` 增加可选 `agentId`。
- WS 与 SSE 入口共用该 schema，但**必须分别验证** JSON body 与 query string 两条发送链。
- `resolveStreamModelRoute()` 读取 `requestData.agentId`：
  - 若匹配到 enabled agent，则按覆盖矩阵计算有效 provider/model/variant/systemPrompt
  - 若未匹配到，则忽略并走现有默认
- permission/question/background/task-parent-auto-resume 路径必须保留 `agentId`。

### 降级与可观测性

当 `agentId` 无效、禁用或对应模型不可用时：

- 请求继续执行，不返回 hard error
- Gateway 至少记录以下字段用于排障：
  - `clientRequestId`
  - `requestedAgentId`
  - `effectiveAgentId`（若有）
  - `downgradeReason`
- Phase 1 可只做日志级可观测性；Phase 2 再决定是否把 warning 回传给前端

### 非目标（Phase 1 不做）

- 不把 slash 插入文本 `使用 Agent「xxx」` 继续当作主 agent 选择机制
- 不重构 capability context
- 不引入新的 planner/orchestrator 状态机
- 不让 Mobile Phase 1 达成完整的 mode default 行为
- 不处理 legacy `src/screens/ChatScreen.tsx` 的完整对齐，只标记为非主路径/待收口实现

## Risks & Guardrails

- **风险 1：WS 与 SSE 字段漂移。** 必须同时修改 JSON body 与 query string 两条链。
- **风险 2：resume 路径丢字段。** `permissions.ts`、`stream-runtime.ts` 与 `task-parent-auto-resume.ts` 都是高风险点。
- **风险 3：task/delegated session 语义被覆盖。** 普通聊天 `agentId` 不能无条件压过 delegated system prompt。
- **风险 4：双重 steering 不可控。** Phase 1 保留 mode 文本 prompt + mode-derived agent，会引入叠加 steering；必须在实现里显式记录为兼容策略。
- **风险 5：Mobile 口径漂移。** 本次仅保证活跃 Expo Router 路由协议兼容，不对 legacy 路径做等价承诺。
