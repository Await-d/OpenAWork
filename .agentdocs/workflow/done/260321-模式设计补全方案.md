# 260321 模式设计补全方案

## Task Overview

用户已确认继续实施历史方案中尚未完全落地的三组模式设计：

1. Worker 执行模式：`Local / Cloud Worker / Sandbox`
2. 多 Agent 数据流模式：`sequential / parallel / conditional`
3. CLI 审批模式：`auto / prompt / deny`

目标不是回到大而全的历史蓝图，而是在当前仓库已有骨架基础上，把这三组模式补成**最小可运行、可测试、可验证**的闭环能力。

## Current Analysis

- `packages/agent-core/src/onboarding/cloud-worker-connection.ts` 只有 `CloudWorkerConnection` stub，缺统一 worker mode 概念，也没有 Local/Sandbox 对应管理器。
- `packages/shared-ui/src/WorkerStatusIndicator.tsx` 已有 Worker UI 卡片，但只消费简单 `WorkerEntry[]`，尚未表达 mode 维度。
- `packages/multi-agent/src/types.ts` 已定义 `DataFlow = 'sequential' | 'parallel' | 'conditional'`，但 `packages/multi-agent/src/dag.ts` 中 `getReadyNodes()` / `executeDAG()` 并未真正利用它。
- `packages/agent-core/src/cli/non-interactive.ts` 已有 CLI/non-interactive runner，但没有 `ApprovalPolicy` 和 auto/prompt/deny 行为。

## Solution Design

### 最小落地原则

1. Worker 模式先实现统一抽象与 session 状态，不在本轮引入真实云平台编排系统。
2. 数据流模式先落实到 DAG 运行时语义，而不是只留在类型层。
3. CLI 审批模式基于现有 `NonInteractiveRunnerImpl` 扩展，不另起全新 CLI 包。
4. 每一组模式都必须有 red→green 测试链、typecheck、以及命令级或运行级手工 QA。

## Complexity Assessment

- Atomic steps: 9+ → +2
- Parallel streams: worker / dataflow / cli policy → +2
- Modules/systems/services: agent-core + multi-agent + shared-ui (+ docs) → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 三组模式跨 core/orchestrator/UI/CLI，多条实现流可并行，但都需要统一在同一文档内跟踪状态与验证证据。
