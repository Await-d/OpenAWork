# 260405 Chat CI 分层门禁实施

## Task Overview

在 chat 刷新恢复链已具备 route/hook/page regressions、gateway verification、browser recovery E2E 与 real live E2E 的前提下，继续把这些验证落到 CI 中，形成 fast 与 live 两层门禁。

## Current Analysis

- 当前 chat 验证矩阵已经很强，但主要靠本地运行。
- 如果所有验证都放进同一层 CI，要么太慢，要么太脆。
- 最合理的结构是：
  - fast gate：route / hook / page / lightweight browser recovery / gateway verification
  - live gate：real web + real gateway + mock upstream 的 heavy E2E

## Solution Design

- 先梳理现有 `.github/workflows/ci.yml` 与 package scripts；
- 给 chat 恢复链补两个清晰脚本入口：
  - fast
  - live
- 在现有 CI 里最小接线：
  - fast 进入主门禁
  - live 单独成 job（当前先限制到 push 才跑）

## Complexity Assessment

- Atomic steps: 3–4 → 0
- Parallel streams: CI 现状梳理 + package scripts + workflow 落地 → +2
- Modules/systems/services: GitHub Actions + apps/web + services/agent-gateway → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 4
- **Chosen mode**: Full orchestration
- **Routing rationale**: 这是工程化落地而非单点修复，需要同时改脚本、CI 工作流与归档记录，适合保留完整实施轨迹。
