# 260404 Chat 真实端到端验收实施

## Task Overview

在 mock 浏览器验收、gateway verification 与 ChatPage 回归之外，进一步补上一条真正带 gateway 后端的浏览器端到端验证：真实 web + 真实 gateway + 可控 mock upstream AI，覆盖发送消息、刷新、attach 恢复到最终 assistant 消息落定。

## Current Analysis

- 现有层级验证已经较完整，但仍缺少一条真正穿过真实 gateway 的浏览器 E2E。
- 浏览器层 mock 用例足以验证恢复体验，却不能替代真实进程链路。
- 最小真实闭环需要三段：
  1. mock upstream AI SSE
  2. real gateway
  3. real web

## Solution Design

- 新增 `apps/web/e2e/mock-openai-upstream.mjs`，提供可控 `/responses` 与 `/chat/completions` SSE；
- 新增 `apps/web/playwright.attach-live.config.mjs`，用 Playwright `webServer` 同时拉起：
  - mock upstream
  - real gateway
  - real web
- 新增 `apps/web/e2e/chat-stream-attach-live.spec.ts`：
  - 真实登录 `/auth/login`
  - 真实配置 `/settings/providers`
  - 真实创建 `/sessions`
  - 在浏览器中发送消息
  - 等待第一段流文本出现后刷新
  - 验证刷新后 attach 恢复并最终落成完整 assistant 消息
- 新增脚本：`pnpm --filter @openAwork/web run test:e2e:attach-live`

## Complexity Assessment

- Atomic steps: 5+ → +2
- Parallel streams: upstream mock + gateway + web + e2e 验证 → +2
- Modules/systems/services: web + gateway + upstream mock → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 这是多进程真实链路验证，需要启动真实服务、处理认证与 provider 配置，并保留完整实施与验证记录。
