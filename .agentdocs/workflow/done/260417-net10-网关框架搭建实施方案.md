# .agentdocs/workflow/260417-net10-网关框架搭建实施方案.md

## Task Overview
- 目标：为 OpenAWork 创建 .NET 10 gateway 骨架搭建实施方案，只搭框架，不迁移现有业务功能。
- 硬约束：EF Core、MediatR、默认 SQLite 且可正向扩展其他数据库、兼容当前树结构工作流日志、保留桌面 sidecar 场景。
- 交付物：阶段化实施路线、项目结构蓝图、数据库/provider 策略、日志兼容策略、sidecar 发布与验收标准。

## Current Analysis
- 当前 `services/agent-gateway` 是唯一后端入口，承载 HTTP、SSE、WebSocket、LSP、GitHub、channels、artifacts、cron 等能力，不能被误判为“普通 REST API 服务”。
- 当前桌面端通过 Tauri 启动本地 sidecar，实际运行约束是本地回环监听、平台数据目录、桌面宿主管理生命周期。
- 当前日志模型不是普通文本日志，而是 `WorkflowLogger + RequestContext + request_workflow_logs.workflow_json` 的树结构工作流日志，并已被请求生命周期与数据库持久化使用。
- 当前 durable 语义以 SQLite 为核心，且已有 runbook / rollout 文档围绕数据库路径、恢复与文件持久化建立了运维假设。

## Solution Design

### 目标态架构
1. `OpenAWork.Gateway.Host`：ASP.NET Core 10 宿主，承载 HTTP/SSE/WebSocket、配置、ProblemDetails、认证、健康检查。
2. `OpenAWork.Gateway.Application`：MediatR command/query、pipeline behaviors、应用级接口与验证。
3. `OpenAWork.Gateway.Domain`：领域对象、错误类型、枚举、业务内核约束。
4. `OpenAWork.Gateway.Infrastructure`：日志、时间、当前用户、后台任务、外部 HTTP、sidecar 运行期辅助。
5. `OpenAWork.Gateway.Persistence.EFCore`：共享 DbContext、实体映射、provider-neutral 持久化模型。
6. `OpenAWork.Gateway.Persistence.Sqlite`：默认 SQLite provider、SQLite migrations、设计时工厂。
7. `OpenAWork.Gateway.Persistence.PostgreSql`：预留 PostgreSQL provider 扩展位与独立 migrations。

### 核心设计原则
1. 先冻结宿主/持久化/日志/sidecar 兼容边界，再建骨架，不先迁业务功能。
2. EF Core 使用共享模型 + provider 独立 migrations，避免把 SQLite 假设泄漏到应用层。
3. MediatR 负责应用编排与横切行为，不替代领域建模。
4. 保留现有树结构工作流日志语义：`RequestContext`、`WorkflowStep`、`request.handle` 根节点、`workflow_json` 落盘。
5. sidecar 作为一等约束：本地 loopback、可自包含发布、可由桌面宿主管理启动/停止/日志与数据目录。

### Reference Rules
- `Host` → 可引用 `Application`、`Infrastructure`、`Persistence.*`、`Contracts`
- `Application` → 可引用 `Domain`、`Contracts`，不得直接引用具体 provider
- `Infrastructure` → 可引用 `Application`、`Domain`、`Contracts`
- `Persistence.EFCore` → 可引用 `Application`、`Domain`
- `Persistence.Sqlite` / `Persistence.PostgreSql` → 只承载 provider 注册、设计时工厂、迁移与 provider-specific 适配
- `Domain` → 不引用 ASP.NET Core、EF Core、MediatR

### Database Strategy
- 默认 provider：SQLite，本地文件落在平台数据目录，对齐现有 gateway durable 假设。
- 扩展 provider：PostgreSQL 优先，其它 provider 后续再评估，不要求首批落地。
- 迁移策略：
  - 共享实体模型与 `GatewayDbContext`
  - provider 独立 migrations 目录或独立 migrations assembly
  - 启动时数据库初始化只对当前选中 provider 生效
- 禁区：
  - Application / Domain 不允许出现 provider-specific SQL、列类型、函数调用
  - 不做“一套 migrations 通吃所有 provider”的假设

### Tree Workflow Logging Compatibility Mapping
| Current TS Model | .NET 10 Target | Compatibility Rule |
|---|---|---|
| `RequestContext` | `RequestContext` record/class | 保留 `requestId/method/path/ip/userAgent/startTime` 语义 |
| `WorkflowStep` | `WorkflowStep` entity/model | 保留 `name/status/message/durationMs/fields/children` |
| `WorkflowLogger.start/startChild/succeed/fail/flush` | 同名或语义等价方法 | 保留树构建与自动收口行为 |
| Fastify hooks in `request-workflow.ts` | `RequestWorkflowMiddleware` | 用 ASP.NET Core 生命周期替代，但保留根节点 `request.handle` |
| `request_workflow_logs.workflow_json` | EF Core entity + JSON column/text column | 保留“整棵树落盘”的读取与展示方式 |

### Sidecar / Packaging Strategy
- 运行模型：桌面端宿主负责拉起本地 gateway 进程，gateway 只监听 `127.0.0.1`。
- 配置输入：端口、host、数据目录、桌面专属开关由宿主注入。
- 发布策略：优先 `self-contained + single-file`；首版不强上 trimming / AOT。
- smoke 要求：
  - 可被桌面宿主启动
  - 可输出启动日志
  - 可写平台数据目录
  - 可被宿主停止并返回可判断的退出状态

## Acceptance Criteria
- 方案明确区分“骨架搭建”与“业务迁移”，且没有偷跑功能迁移。
- 方案明确说明默认 SQLite 如何落地，以及未来多 provider 如何扩展而不污染应用层。
- 方案明确说明树结构工作流日志如何与现有 `WorkflowLogger` / `request_workflow_logs` 保持语义兼容。
- 方案明确说明桌面 sidecar 的运行、配置、数据目录与发布形态。
- 方案给出分阶段任务、里程碑、风险与最小验收标准，后续可直接转为编码任务。

## Risks
- 若先做代码脚手架再补 provider/log/sidecar 边界，后续会出现大规模返工。
- 若把树结构日志退化为普通文本/结构化日志，后续会失去现有工作流观察方式与持久化查询能力。
- 若默认 SQLite 没有在第一阶段就隔离 provider 差异，未来扩 PostgreSQL 会从“扩展”变成“重构”。
- 若 sidecar 只在开发态跑通、未收敛发布形态，桌面集成会成为后期最大阻塞项。
