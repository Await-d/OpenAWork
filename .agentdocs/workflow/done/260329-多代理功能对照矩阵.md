## Notes

- 这是对上一轮《移植多代理对话流程差异分析》的后续展开，目标是形成可执行对照表，而不是重新做一轮泛分析。
- 已确认的新增结论：
  - OpenAWork 的 `background_output/background_cancel` 当前仅出现在 `/capabilities` 的 reference 工具快照中，`callable: false`，并没有对应的真实 tool handler；当前真正存在的只是 child session 后台执行与 auto-resume 机制。
  - OpenAWork 的 permission / question 不是像 opencode 那样沿 session tree 向父会话汇总；permission 按 sessionId 隔离，task 子代理默认不会创建 permission request；question 需要切入子会话面板处理，父会话只通过 task 状态感知 pending-interaction。
  - OpenAWork 当前 managed agents 通过 `agent-catalog.ts` 内建注册 14 个 agent，systemPrompt 优先来自 `agent-reference-snapshot.ts` 对 temp 参考目录的抽取；`task` 调用时由 `task-agent-resolution.ts` 解析 `subagent_type`，缺省默认路由到 `sisyphus-junior`。
  - `temp/oh-my-openagent` 的后台任务不是“自己回流”给父代理，而是通过 `<system-reminder>` 显式通知父 session，再由 `background_output(task_id)` 读取结果；sync/background continuation、todo-continuation-enforcer、atlas/boulder 提醒、spawn depth/budget limit 都是运行时强约束，不是 UI 装饰。
  - `temp/oh-my-openagent` 在子 session 创建时硬编码注入 `QUESTION_DENIED_SESSION_PERMISSION`，即子代理默认不能用 `question` 向用户追问；question 仅保留在主 session/UI 层。
  - `temp/opencode` 的 permission/question 不在后端按 session tree 汇总；后端只维护平铺 pending map，前端通过 `session-request-tree.ts` 对 session 树做 BFS，把父会话和全部子孙会话的 pending request 拼装成“当前会话被阻塞”的 UI 心智。
  - `temp/opencode` 没有 OpenAWork 那种“后台 task 投影 + 自动回流续跑”路径；它的“主代理在等子任务”体感主要来自同步子任务语义 + 前端树遍历阻塞感知。

## 逐功能对照矩阵

| 功能维度 | OpenAWork 当前 | temp/opencode | temp/oh-my-openagent | 当前对齐判断 |
| --- | --- | --- | --- | --- |
| `task` 默认执行语义 | 默认偏后台；`run_in_background` 不填通常仍走 child session 后台执行 | 同步内联子任务 | 显式 sync / background 双模态 | **未对齐** |
| 子代理结果返回给父代理的方法 | 事件投影 → 回写 tool result → 某些场景 auto-resume | 当前回合直接得到 tool output | sync 立即返回；background 用 `background_output` 显式拉取 | **未对齐** |
| 后台任务完成通知 | `task_update/session_child` 事件 + 子会话面板；没有 OMO 式 `<system-reminder>` | 无独立 background 通知心智 | `<system-reminder>` 明确通知父 session | **未对齐** |
| 后台任务结果读取 | `background_output/background_cancel` 仅 capability 快照，当前主路径仍偏自动回流 | 不适用 | `background_output(task_id)` / `background_cancel(task_id)` 是运行时主路径 | **未对齐** |
| 父会话 continuation/继续方式 | runtime 可能自动注入回流消息继续跑 | 主 loop 在同步子任务后自然继续 | sync 自然继续；background 由通知 + 显式读取驱动 | **未对齐** |
| 子代理是否默认可 `question` | child session 默认不含 `question`，需 metadata 显式开启；question 在子会话面板中处理 | 无“子代理禁问”强约束，更多是 session tree 消费方式 | 硬编码 deny `question` 到子 session | **半对齐** |
| 子代理是否默认可继续递归 `task` | child session 默认不含 `task`，深度上限 4、总后代上限 24 | 无独立 background spawn budget 语义 | child session 默认可委派，但有深度 3 / 总后代 50 强约束 | **半对齐** |
| permission 汇总方式 | 父会话 runtime snapshot 会主动拉 parent + child sessions 的 pending permissions，并可直接回给目标 child session | 后端平铺，前端 BFS 聚合 session tree | 子 session 默认禁问用户，因此不依赖树汇总 | **部分对齐，但方法不同** |
| question 汇总方式 | 不向父会话汇总；父会话只通过 `pending-interaction` / 子会话 paused 感知 | 后端平铺，前端 BFS 聚合 session tree | 子 session question 入口直接 deny | **未对齐** |
| 子代理阻塞如何被用户感知 | Chat UI 展示 subagent run list、detail panel、waiting/paused 状态；question 需切子会话处理 | 输入区 blocked + 侧栏 badge，来自 session tree BFS 聚合 | 主 session 收 system-reminder / continuation 提醒，子 session 不问用户 | **外形差异大，语义未对齐** |
| `category` 的含义 | 更像先映射到 `sisyphus-junior` 再附加 contract | 无 category 层 | category 是执行策略层（模型/方法/skills） | **未对齐** |
| `session_id` 续接同一子代理 | 技术上支持 child session 复用，但用户心智较弱 | 不强调此层概念 | 是标准用法，明确推荐 | **半对齐** |
| managed agents / agent 目录 | `agent-catalog.ts` 注册 14 个 builtin agents，description/systemPrompt 来自 temp 抽取 | 原生 agent/session 体系 | 完整 agent registry + category routing + prompt discipline | **外形对齐，方法未对齐** |
| capability 目录是否等于真实能力 | `/capabilities` 含 reference-only 工具，`background_output/background_cancel` 为 `callable:false` | 能力更多直接来自真实 session/app 能力 | 能力目录与运行时 task/background manager 语义一致 | **未对齐** |
| 子会话树 / 任务树的事实源 | session tree + task projection 双轨并存 | 只有 session tree（parentID） | child session + background task manager + lineage/spawn limits | **实现模型不同** |

## 已可确定的对齐状态

### 已对齐外形、未对齐语义

1. **Agent 名称与 prompt 外形**：OpenAWork 已通过 `agent-reference-snapshot.ts` / `agent-catalog.ts` 抽取参考文案，但 runtime contract 不同。
2. **子代理 UI 可见性**：OpenAWork 比 opencode 更强，但这只是投影层，不代表 task 语义已对齐。
3. **权限聚合“能看到 child request”**：OpenAWork 与 opencode 都能做到父会话感知 child session 的 pending permission，但方法不同：OpenAWork 偏服务端聚合快照，opencode 偏前端 BFS 聚合。

### 明确未对齐的核心语义

1. `task` 默认语义仍不是显式 sync/background 双模态。
2. background 结果消费仍不是“通知 + 显式读取优先”。
3. auto-resume 仍可能替代父代理显式控制 continuation。
4. `category` 尚未恢复为执行策略层。
5. capability 目录仍在暴露 reference-only 能力名，容易让使用者误判真实运行时能力。
6. question 交互仍主要停留在子会话面板，不像 opencode 的父树聚合，也不像 OMO 的子层禁问心智那样清晰。

## 第一阶段实施清单

### 推荐目标：优先对齐到 `temp/oh-my-openagent` 的 task 契约

原因：OMO 的 sync/background 双模态是 opencode 同步子任务的超集。只要 `run_in_background=false` 路径做对，就天然兼容“像 opencode 一样同步等待”；而 `run_in_background=true` 则保留你们现在已有的 child session / UI 投影能力。

### Phase 1A：先让能力目录与真实运行时说同一种话

1. **处理 `background_output/background_cancel` 的真假不一致**
   - 方案 A：先从 `/capabilities` 的 reference-only 列表中移除这两个工具，避免假能力
   - 方案 B：真正实现它们，并让 task/background 运行时走这套契约
   - **建议**：如果马上做 task 双模态，就直接选 B；如果还没开始改 runtime，就先选 A 以免误导

**落点文件：**
- `services/agent-gateway/src/routes/capabilities.ts`
- 若实现真实工具：`services/agent-gateway/src/tool-definitions.ts`、`services/agent-gateway/src/tool-sandbox.ts`、`services/agent-gateway/src/command-loop-runtime.ts`

### Phase 1B：把 `task` 收敛成显式双模态

2. **`run_in_background` 改为显式必填**
3. **同步模式：真正等待 child session 完成，并把最终 assistant 文本 / 摘要直接作为父代理的当前 tool output 返回**
4. **后台模式：立即返回 task handle，不再默认 auto-resume 父会话**

**落点文件：**
- `services/agent-gateway/src/task-tools.ts`
- `services/agent-gateway/src/tool-sandbox.ts`

### Phase 1C：把 auto-resume 从默认主路径降级

5. `scheduleTaskParentAutoResume()` 改为显式可选策略，而不是后台 task 的默认收尾方式
6. 父会话继续执行应优先通过：
   - sync 结果的当前回合继续
   - 或 background 完成后的显式结果读取/继续

**落点文件：**
- `services/agent-gateway/src/task-parent-auto-resume.ts`
- `services/agent-gateway/src/tool-sandbox.ts`

### Phase 1D：恢复 `category` 的执行策略地位

7. `category` 不应只 alias 到 `sisyphus-junior`；至少要把模型路由 / skill 注入 / method contract 挂回 category
8. `subagent_type` 与 `category` 的职责需要重新切清：
   - `subagent_type` = 指定代理身份
   - `category` = 指定执行策略

**落点文件：**
- `services/agent-gateway/src/task-agent-resolution.ts`
- `services/agent-gateway/src/agent-catalog.ts`

### Phase 1E：最后再改 UI，使其忠实反映语义

9. 区分展示：
   - 同步子任务
   - 后台任务
   - 等待 question/permission
   - 已完成但待主会话继续
10. 不再把不同运行态都压扁成“子代理运行中/等待中”

**落点文件：**
- `apps/web/src/pages/ChatPage.tsx`
- `apps/web/src/pages/chat-stream-state.ts`
- `apps/web/src/pages/chat-page/sub-agent-run-list.tsx`
- `apps/web/src/pages/chat-page/use-sub-session-detail.ts`

## 最终建议

- 如果你的目标是**最小代价对齐参考使用方法**：以 OMO 的 `task` 契约为目标，`opencode` 的同步体感作为 `run_in_background=false` 的自然结果。

- 就当前仓库已有资产来看，**最优路线不是退回 opencode，而是把 OpenAWork 收敛成更真实的 OMO 双模态 task。**
