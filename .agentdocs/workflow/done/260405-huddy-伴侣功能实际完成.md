# Huddy / Buddy 伴侣功能实际完成

## Task Overview
基于仓库内已存在的半成品 companion / buddy 实现，把用户口中的 “Huddy 功能” 真正收口为可用闭环：补齐统一设置真相源、显式 `/buddy` 入口、request-scoped companion prompt 注入、确定性 profile 返回、前端面板控制与对应测试验证。

## Current Analysis
- 代码库中不存在名为 `Huddy` 的正式模块；当前功能线实际以 `buddy / companion / 伴侣功能` 命名，已有 `CompanionStage`、本地 TTS MVP、`/settings/companion` 路由和部分测试。
- 现状并非“未实现”，而是“实现了一半”：Web UI 已挂在 `ChatPage`，但 `/buddy` 命令、prompt 注入、gateway profile、统一设置控制和 composer 入口仍未闭环。

## Solution Design
- gateway：通过 `services/agent-gateway/src/companion-settings.ts` 统一 companion preferences、deterministic profile 与 prompt 注入文案，供 `/settings/companion` 与 stream 主链复用。
- 命令层：新增 `/buddy` client command，并让 composer slash suggestions 暴露该命令；发送时在 `ChatPage` 拦截为客户端动作，打开 companion 面板而不是发给模型。
- Web：将 `useBuddyVoicePreferences` 升级为 companion 设置真相消费 hook，覆盖 enabled / injectionMode / muted / reducedMotion / voice output / remote profile；`CompanionStage` 尊重主开关并支持外部打开信号；Settings 增加独立 Buddy tab。

## Complexity Assessment
- 原子步骤：至少 6 个核心步骤（gateway profile / prompt injection / command / 前端控制 / Settings 入口 / 验证） → +2
- 并行流：检索、gateway 收口、Web 入口与测试可分流推进 → +2
- 模块/系统/服务：`services/agent-gateway` + `apps/web` + `packages/shared` 命令合同 → +1
- 单步 >5min：多模块接线与回归验证超过 5 分钟 → +1
- 持久化审查产物：需同步 workflow / runtime / index → +1
- OpenCode 可用：→ -1
- **总分：6**
- **选定模式：Full orchestration**
- **路由理由：这是一个跨 gateway 与 Web 的真实功能收口任务，需要保存实施状态与验证证据。**

## Success Criteria
- `/settings/companion` 返回非空 deterministic companion profile，并保留安全默认值与 feature gate。
- companion intro 能在 `stream.ts` 与 `stream-runtime.ts` 两条请求链上按注入策略一致注入。
- `/buddy` 可以通过 composer slash 显式触发，并且不会把指令误当成普通聊天消息发送。
- ChatPage 的 companion panel 可被外部动作显式打开，前端 companion 开关与 gateway `/settings/companion` 使用同一写入链路。
- 相关测试、受影响包 typecheck 与 Web build 通过。
