# .agentdocs/workflow/260401-usage写入链路追踪.md

## Task Overview

继续上一轮 usage 页面实现度审计，往上游追踪 `usage_records` 本应如何生成与写入，定位当前 token/cost 数据链路与月度 usage 落库之间的断点。

## Current Analysis

- 上一轮已确认：usage 页面与读取接口存在，但 `usage_records` 在仓库中只有建表和读取，没有任何显式写入。
- 初步直搜已发现 `packages/agent-core/src/token-usage/index.ts` 存在 token/cost 聚合语义，说明系统并非完全没有 usage 数据模型。
- 本轮分析目标是把“运行时 usage 数据对象”与“SQLite 月度 usage_records”之间缺失的桥接位置找出来。

## Solution Design

1. 追踪 provider/stream/agent-core 中 usage 与 cost 数据是如何被提取和聚合的。
2. 检查 gateway 在请求完成、会话完成、流式收尾、cron/job 等边界是否存在应落库但未接入的候选位置。
3. 综合判断缺口究竟是“已有聚合器未接线”还是“从上游开始就没完整产出计费事实”。

## Notes

- 当前任务继续保持只读分析，不修改业务代码。
- 已验证 `packages/agent-core/src/token-usage/index.ts` 提供了 `TokenUsageManagerImpl`（带月度汇总与 `byProvider`），但当前仓库未发现任何实例化或 `record/getMonthly/listRecords` 调用，属于“设计存在、运行时未接线”。
- 已验证 `packages/agent-core/src/provider/utils.ts` 提供了 `calculateTokenCost()`，但当前仓库无任何调用；价格字段主要停留在 provider/model 配置与前端估算展示层。
- 已验证 `services/agent-gateway/src/verification/verify-openai-responses.ts` 会构造 `response.completed.response.usage`，而 `services/agent-gateway/src/routes/stream-protocol.ts` 在 `response.completed` 分支只返回 `done` chunk，并未提取或下传 usage 字段。
- 已验证 `services/agent-gateway/src/routes/stream-model-round.ts` 的 `finalizeAssistant/completeRound` 只负责 assistant message、snapshot 与 done 事件；`services/agent-gateway/src/routes/stream.ts` 在请求结束时只回写 session runtime state，不涉及 usage 聚合或 `usage_records` upsert。
- 已验证 usage 概念当前至少分裂成三份：前端 `ChatPage` 的消息级估算 usage、`agent-core` 的 `TokenUsageManagerImpl` 内存聚合、gateway 的 `usage_records` 月表；三者之间没有统一真相源，也没有桥接写入。
- 当前最可信的断点链为：上游 provider `response.completed.response.usage` 已存在 → `stream-protocol.ts` 丢字段 → `stream-model-round.ts` / `stream.ts` 没有 usage 聚合状态与月度 upsert → `routes/usage.ts` 只能读取空壳或历史手工数据。
- Oracle 复核结论与上述判断一致：最佳修复插点不是 `usage.ts` 读取端，也不是未接线的 `TokenUsageManagerImpl`，而是 `stream.ts` 的 `runModelRound()` 主循环；前提是先在 `stream-protocol.ts` 与 `stream-model-round.ts` 完成 usage 透传。
- Oracle 同时确认了两个边界风险：其一，当前 `usage_records` 只支持月总量，无法天然支撑 `byProvider` 与 `breakdown`；其二，chat completions 路径目前同样未见 usage 提取，不能假设修完 Responses 路径就自动完整。
