# 260404 Chat 浏览器续流最终落定实施

## Task Overview

在已完成“浏览器层恢复中的内容与控制态”验收后，继续尝试把刷新后 attach 完成时的最终 assistant 消息落定也纳入自动化验证。

## Current Analysis

- 现有浏览器层用例已稳定覆盖“已恢复内容 + 会话运行中提示 + 停止控制”。
- 进一步把 attach `text_delta + done` 直接塞进 Playwright 浏览器环境时，暴露出 EventSource / reload 时序在该层不稳定。
- 更稳妥的分层是：
  - 浏览器层验用户可见恢复体验
  - `ChatPage` 回归验最终消息落定

## Solution Design

- 保留 Playwright 用例：验证刷新后恢复内容、运行态提示和停止控制；
- 新增 `ChatPage` 回归：验证 attach 完成后最终 assistant 消息真正落定，且运行态提示消失；
- 不再把所有 attach wire 细节继续硬塞进浏览器层。

## Complexity Assessment

- Atomic steps: 3–4 → 0
- Parallel streams: e2e 用例 + ChatPage 回归 + 本地验证 + 记录同步 → +2
- Modules/systems/services: web + playwright infra → +0
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 3
- **Chosen mode**: Full orchestration
- **Routing rationale**: 这次核心价值在于把验收边界重新收敛到稳定且可维护的分层，需要保留结论与取舍。
