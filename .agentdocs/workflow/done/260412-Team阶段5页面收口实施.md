# Team 阶段5页面收口实施

## Task Overview

在 Stage 1 已完成 TeamWorkspace 根边界、Stage 2 已完成最小 Team-owned 写路径、Stage 3 已完成 workspace snapshot 主读链切换、Stage 4 已完成 bridge 显式化的基础上，继续推进 **Stage 5：页面热点与遗留壳收口**。本阶段目标是把 Team 前端里仍然只承担过渡职责的 runtime reference layer / legacy shell / 热点页面块逐步收口，使新的 workspace-first + bridge-explicit 数据流在页面结构上也稳定下来。

## Current Analysis

- Stage 4 已完成：shared session 已进一步降级为 bridge/import/detail 层
- 当前仍未完成的部分：
  - `team-runtime-reference-page.tsx` 仍承载较重的过渡壳层职责
  - `team-runtime-shell.tsx` / `team-runtime-shell-frame.tsx` 等旧壳层仍可能有遗留职责
  - `TeamSessionSidebar.tsx`、`MainWorkspace.tsx`、`TopTeamHeader.tsx` 等热点区域仍混有 reference-era 结构负担

## Solution Design

- Stage 5 只做页面收口，不再重开架构边界问题。
- 最小实施顺序：
  1. 先确认哪个 legacy shell / 热点块最值得第一刀收口
  2. 再做最小 UI 结构清理
  3. 最后用现有测试链验证页面收口没有打穿前 4 个阶段的数据流
- 本阶段不碰：
  - gateway / web-client contract
  - interaction-agent 主流程
  - 办公室视觉重设计

## Complexity Assessment

- Atomic steps: 3-4（前端壳层探索 / 最小 UI 收口 / 验证 / git） → 0
- Parallel streams: yes（热点块/legacy shell/首刀评估可并行） → +2
- Modules/systems/services: 1（web） → 0
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 3
- **Chosen mode**: Full orchestration
- **Routing rationale**: 虽然主要在 web 内，但要在不破坏前 4 阶段的情况下收口遗留壳层，仍需要独立工作流与清晰提交边界。
