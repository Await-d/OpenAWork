# .agentdocs/workflow/260329-移植多代理对话流程差异分析.md

## 任务概览

对比 OpenAWork 当前仓库与 `temp/oh-my-openagent`、`temp/opencode` 中和 agent 团队、子代理、整体对话流程相关的实现，判断为什么当前集成后的“使用感受”与参考实现不一致，并整理需要对齐的移植方法，特别是每个功能的使用逻辑与方法。

## 当前分析

- `.agentdocs/index.md` 已记录多轮与 `temp/opencode` / `temp/oh-my-openagent` 的对标与阶段性实现，说明当前系统并非缺少所有能力，而是存在“已有能力、已冻结语义、当前运行体验”三者可能未完全一致的问题。
- 已有记忆明确指出：`task` 工具当前至少经历过两轮演进——早期仅为 child session + task graph 入口，后续已补 delegated agent、结果回流与 UI 运行态投影；因此本次重点不只是“有无功能”，而是“调用方式、触发时机、上下文边界、结果回流方式”是否真正对齐参考实现。
- 用户关注的是“为什么感觉不一样”，因此必须把差异拆成可感知维度：主代理是否直接回答 / 先委派、子代理是否拥有独立身份、工具是否由主线程统一调度、结果是内联回流还是仅在侧栏可见、权限与追问是谁负责、会话连续性如何维护。

## 最终结论

### 1. 为什么当前集成后“感觉不一样”

- **根因不在 agent 名字，而在 `task` 的运行时语义已经变了。**
- `temp/opencode` 的 `task` 更接近**主 loop 内联同步子任务**：主代理在当前回合里等待子代理跑完，把子代理最后的 assistant 文本立即作为 tool output 写回，再继续当前推理。
- `temp/oh-my-openagent` 在 opencode 之上进一步把 `task` 分成了**显式 sync / background 双模态**，并且由 Sisyphus prompt 强制规定什么时候该后台并行、什么时候该同步等待、什么时候该用 `session_id` 续同一子会话。
- OpenAWork 当前实现则更接近**异步 child session + 事件投影 + 自动回流续跑**：父会话先收到 `session_child` / `task_update`，子会话完成后再通过 `syncParentTaskToolResult()` 与 `scheduleTaskParentAutoResume()` 把结果写回甚至自动推动主会话继续。
- 因此当前体感不是简单“少了某个 UI”，而是父代理与子代理之间的等待方式、继续方式、结果消费方式已经不同。

### 2. 表层差异 vs 运行时差异

**表层/UI 感知差异：**
- `ChatPage.tsx` / `chat-stream-state.ts` / `sub-agent-run-list.tsx` 把 `task_update`、`session_child`、轮询结果投影成“子代理运行列表 / 详情面板 / DAG”。
- 这会强烈影响“看起来像不像 agent team 在工作”，但它本身不是主因。

**运行时语义差异：**
- OpenAWork 的 `task` 默认后台；参考实现的关键路径则是同步子任务，或至少由调用者**显式**声明 sync/background。
- OpenAWork 默认允许“后台完成后自动回流并继续主会话”；参考实现更强调由父代理在当前回合直接消费子结果，或由调用者显式用 `background_output` 拉取。
- OpenAWork 当前 `category -> sisyphus-junior` 更像 agent alias；而在 oh-my-openagent 中，`category` 是完整执行策略层的一部分，决定模型/方法/skill 注入。

## 功能使用逻辑对比

### `task` 的推荐使用方法

**temp/opencode：**
- 语义：同步子任务
- 使用心智：主代理把一段工作临时交给子代理，等待结果回来后立刻继续
- 结果消费：当前回合 tool output

**temp/oh-my-openagent：**
- 语义：显式双模态
- `run_in_background=false`：同步等结果
- `run_in_background=true`：立即返回 background task id，后续用 `background_output` 显式取结果
- 使用心智：由 Sisyphus 编排层决定何时同步、何时并行

**OpenAWork 当前：**
- 语义：隐式偏后台
- `run_in_background` 不填时大多仍会后台跑
- 结果消费：先走事件投影，随后回写 tool_result，某些情况下再自动注入“子代理结果回流”消息推进主会话
- 使用心智：更像“父代理放手让子代理跑，系统自己决定如何把结果塞回来”

### `category` 的推荐使用方法

**temp/oh-my-openagent：**
- `category` 代表执行风格与模型路由策略，是使用方法的一部分，不只是展示标签

**OpenAWork 当前：**
- `category` 被先收敛成 `sisyphus-junior`，更偏“统一执行器 + 提示补充”

### `session_id` / 子会话续接的推荐使用方法

**temp/oh-my-openagent：**
- 续接同一个子代理是默认推荐路径，明确强调节省 tokens、保持完整上下文

**OpenAWork 当前：**
- 已支持 child session 复用，但整体体验仍被默认后台与自动回流语义覆盖，用户不容易形成“我在明确续接同一个子代理”的稳定心智

## 对齐建议（按优先级）

### P0：先统一 `task` 契约

1. **把 `run_in_background` 变成显式必填，而不是默认后台。**
2. **同步模式必须等价于“子代理完成后立即把摘要/文本返回父代理”。**
3. **后台模式必须等价于“立即返回句柄，后续显式查看结果”。**

### P1：关闭“后台完成后自动回流续跑”作为默认主路径

4. `scheduleTaskParentAutoResume()` 不应继续充当默认路径，否则 OpenAWork 仍会与 opencode / OMO 的父代理心智偏离。
5. 若要保留 auto-resume，应降级为可选策略，而不是默认行为。

### P1：恢复 `category` 的执行策略地位

6. `category` 不应只映射成 `sisyphus-junior` 名义上的代理，而应重新承担模型路由、技能注入与执行风格约束。

### P2：最后才是 UI 对齐

7. 当前子代理运行列表、详情面板、task projection 可以保留，但 UI 必须反映真实语义：同步子任务 / 后台任务 / 等待权限 / 显式结果拉取，不能再把不同语义伪装成同一种“子代理执行中”。

## 推荐落点文件

- `services/agent-gateway/src/task-tools.ts`：先收敛 `task` 输入契约，特别是 `run_in_background` 是否必填
- `services/agent-gateway/src/tool-sandbox.ts`：重写 task sync/background 主分支，使同步模式真正立即返回子代理结果
- `services/agent-gateway/src/task-parent-auto-resume.ts`：把 auto-resume 从默认主路径降为显式可选策略
- `services/agent-gateway/src/task-agent-resolution.ts`：恢复 `category` 的执行策略语义，而不是仅 alias 到 `sisyphus-junior`
- `apps/web/src/pages/ChatPage.tsx` 与 `apps/web/src/pages/chat-page/sub-agent-run-list.tsx`：在运行时语义稳定后再校准子代理 UI 呈现

## Notes

- 本工作流创建于 2026-03-29，用于本轮“集成后为什么感觉不一样”的差异分析。
- 本轮优先产出结构化结论，不先假设必须改代码；若发现明确缺口，再把差异转成具体改造项。
- 已确认的中间结论：
  - `temp/opencode` 的 `task` 更接近“同步内联子任务”：主 loop 遇到 `SubtaskPart` 后直接执行 `TaskTool`，等待 `SessionPrompt.prompt()` 返回最后 assistant 文本，再把结果作为当前回合 tool output 写回主 session 后继续。
  - `temp/oh-my-openagent` 在 opencode 之上增加了更强的 orchestration 语义：`category -> sisyphus-junior`、`load_skills`、显式 `run_in_background`、`background_output`、以及 Sisyphus prompt 对并行探索 / Oracle / 类别路由的强约束。
  - OpenAWork 当前实现更偏“异步 child session + 父会话事件投影”：`tool-sandbox.ts` 中 task 默认走后台 child session，父会话先收到 `session_child` / `task_update`，完成后通过 `syncParentTaskToolResult`、`publishSessionRunEvent` 和 `scheduleTaskParentAutoResume()` 把结果回流到主对话。
  - 因此当前“感觉不一样”很可能不只是 UI 命名问题，而是 task 的默认执行语义已经从“主代理同步等待子代理结果”变成了“主代理先放手、靠事件和自动回流补齐”。
