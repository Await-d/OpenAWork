# Agent 角色体系规范化方案

## Task Overview

在已完成 OpenAWork 现有 agent/reference capability、omo/Claude Code/Codex/ccg-workflow 角色体系对照的基础上，
将“角色本体、专项预设、覆盖能力、生态别名”正式收敛为一套可落地的内部规范，
用于后续指导 `packages/multi-agent`、`packages/agent-core`、`services/agent-gateway` 与 Web UI 的统一实现。

## Current Analysis

- 当前仓库已经存在 `AgentRole` 接口（`packages/multi-agent/src/types.ts`），但没有内建的角色库与规范化枚举。
- `services/agent-gateway/src/routes/capabilities.ts` 已暴露多组 reference agents：`general / explore / hephaestus / prometheus / oracle / librarian / metis / momus / atlas / multimodal-looker / sisyphus-junior` 等，但这些名称混合了通用角色、神话人格名与参考别名。
- `packages/multi-agent/src/team.ts` 的 `TeamMember.role` 仍是自由字符串；当前有 team runtime，但没有 `team-plan / team-review / team-exec / team-research` 这类固定预设包。
- `packages/agent-core/src/plan/types.ts` 已天然存在 `drafting -> approved -> implementing -> completed/rejected` 的计划生命周期，说明“规划/执行/评审”已经是系统里的真实阶段，而不是纯概念。
- 外部生态中，OMCC/OMCX 的 `analyst/planner/executor/critic/verifier` 边界较直接；omo 更偏神话命名与“规划链”；CCG 的 `/ccg:team-*` 本质上是阶段命令而非角色本体。
- 当前最大的语义风险不是缺少名称，而是把“角色、命令、team preset、人格别名”混成同一层，导致后续路由、权限、UI 展示和统计口径都容易漂移。

## Solution Design

### 1. 规范化分层

内部模型统一拆成四层：

1. **核心角色（Core Role）**：系统真正用于路由和职责归类的一级角色。
2. **专项预设（Preset）**：在核心角色下细分的工作模式，用于模型/tool profile/提示词差异化。
3. **覆盖能力（Overlay）**：跨角色可叠加的能力标签，如写作、多模态。
4. **生态映射（Alias Mapping）**：把外部生态名称映射到内部核心角色 + preset，显式保留置信度与备注。

### 2. 核心角色冻结为 5 个

```ts
type CoreRole =
  | 'general'
  | 'researcher'
  | 'planner'
  | 'executor'
  | 'reviewer';
```

设计理由：

- `general`：兜底入口，避免未知任务强行误分流。
- `researcher`：统一承接 `explore / analyst / librarian` 这类“收集和澄清上下文”的职责。
- `planner`：统一承接 `planner / architect` 这类“结构化决策与顺序编排”的职责。
- `executor`：统一承接 `executor / debugger` 这类“动手解决问题”的职责。
- `reviewer`：统一承接 `critic / code-review / test / verifier` 这类“否决、验收、证明完成”的职责。

### 3. 专项预设与覆盖能力

建议的预设：

```ts
type RolePreset =
  | 'default'
  | 'explore'
  | 'analyst'
  | 'librarian'
  | 'architect'
  | 'debugger'
  | 'critic'
  | 'code-review'
  | 'test'
  | 'verifier';
```

建议的覆盖能力：

```ts
type RoleOverlay = 'writer' | 'multimodal';
```

### 4. 生态映射原则

- `Sisyphus / Hephaestus / Oracle / Momus / Atlas` 这类人格名只保留为 **legacy alias**，不直接作为 canonical role id。
- `/ccg:team-plan / team-review / team-exec / team-research` 这类条目视为 **team preset / stage command**，不与内部核心角色一一等价。
- 每条 alias mapping 必须携带：`coreRole`、`preset?`、`overlays[]`、`confidence`、`notes`。
- 低置信度映射不得参与自动路由，只能用于 UI 展示或人工确认。

### 5. 推荐的契约落点

- **跨端共享契约**：放在 `packages/shared`，承载 `CoreRole / RolePreset / RoleOverlay / RoleAliasMapping` 等纯类型与常量。
- **运行时 Agent 角色对象**：继续保留在 `packages/multi-agent` 的 `AgentRole`，并扩展对 canonical role 的引用。
- **gateway 能力目录与 reference agent 暴露**：在 `services/agent-gateway` 维护 alias registry，把外部/历史名映射到 canonical role。
- **前端展示**：消费 `@openAwork/shared` 的 canonical contract，不再在 UI 层自行猜测角色语义。

## Guardrails

1. 不允许把人格名（如 `Sisyphus`、`Oracle`）直接当作内部 canonical role id。
2. 不允许把 `/ccg:team-*` 这类阶段命令硬映射为一级角色。
3. 不允许把 `writer`、`multimodal` 升格为一级核心角色；它们必须保持 overlay 身份。
4. 在没有合规/审计硬要求前，不提前把 `critic / code-review / test / verifier` 拆成多个一级角色。
5. 路由、权限、日志、UI 展示必须共享同一份 canonical taxonomy，禁止各层自行维护别名表。

## Verification Plan

1. **Contract 验证**：`packages/shared` 导出的 taxonomy 被 gateway、multi-agent、web 同时消费，无重复定义。
2. **Mapping 验证**：reference agents 与外部生态名都能映射到 `coreRole + preset + overlays + confidence`。
3. **Runtime 验证**：`AgentRole`、`TeamStore`、`TaskPlan` 至少能无损携带 canonical role 信息。
4. **UI 验证**：能力目录与团队展示以 canonical role 为主，alias 为辅，且旧名兼容不丢失。
5. **Backward compatibility 验证**：已有 reference agent label 与历史对话中的旧名仍可展示和查询。
