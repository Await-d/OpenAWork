# .agentdocs/workflow/260322-参考剩余功能集成方案.md

## 任务概览

在已完成 Provider 配置、第三方通道订阅、`/channels` 独立页面与基础团队/技能/调度页面之后，继续补齐对标参考仓库时仍明显缺失或偏弱的功能，形成一份可执行的集成方案。

目标不是一次性把所有能力硬塞进当前仓库，而是给出一条**按依赖顺序、可分阶段验收、可并行推进**的集成路线，覆盖剩余高价值差距：

1. 桌面自动化 / RPA
2. 文件改动审阅 / 接受 / 回滚 UI
3. SSH 远程文件管理 / 上传 UX
4. 更深的团队多 Agent 协同闭环
5. 更广的消息渠道覆盖
6. Prompt 优化器
7. 专门的翻译工作流

---

## Current Analysis

### 已完成且不再视为差距的能力

- Provider 自定义配置、预置 Provider 编辑、端点覆写
- 第三方消息频道订阅配置
- `/channels` 独立页面与设置页入口
- 调度 / 定时任务页面与基础后端
- Skills 市场 / 已安装技能管理 UI
- 基础 Team 页面与 Team CRUD 路由
- Provider OAuth 基础能力
- MCP、Sessions / Chat、Usage / Billing 视图

### 剩余差距的本质

当前项目的差距已经不在“基础框架”，而集中在三类高阶产品能力：

1. **操作执行层差距**
   - 桌面自动化仍只有浏览器自动化，缺少本机截图 / 鼠标 / 键盘控制
   - SSH 仍偏连接与执行，缺少完整远程文件浏览 / 上传 / 批量操作 UX

2. **Agent 结果治理层差距**
   - 缺少 agent 文件改动的审阅 / 接受 / 回滚界面
   - 缺少更深的 team orchestration：任务认领、成员状态演进、回传结果闭环

3. **专项工作流层差距**
   - 缺少 Prompt 优化器
   - 缺少专门的翻译工作流
   - 缺少更广的消息渠道覆盖（WhatsApp / WeCom / QQ 等）

## Complexity Assessment

- Atomic steps: 7+ 个明确模块 → **+2**
- Parallel streams: 能拆成 4 条以上并行流（执行层 / 结果治理层 / 专项工作流层 / 渠道扩展）→ **+2**
- Modules/systems/services: 涉及 `apps/web`、`apps/desktop`、`packages/shared-ui`、`packages/agent-core`、`services/agent-gateway`、可能新增 desktop-control / review 模块 → **+1**
- Long step (>5 min): 多个单步都显著大于 5 分钟 → **+1**
- Persisted review artifacts: 方案需写入 `.agentdocs` 供后续实施与复核 → **+1**
- OpenCode available: 当前环境支持原生 task 协作 → **-1**
- **Total score**: **6**
- **Chosen mode**: **Full orchestration**
- **Routing rationale**: 这是一个跨模块、跨层级、可并行拆分的长期能力补齐方案，必须使用持久化 workflow + runtime master plan 才能避免后续实施时范围漂移。

## Solution Design

### 设计原则

1. **优先补“平台能力”而不是只补页面**
   - 桌面自动化、文件改动审阅、SSH 文件管理这类能力先落协议和数据模型，再做 UI。

2. **尽量复用现有模块，不重复造新系统**
   - 浏览器自动化基于 `packages/browser-automation/`
   - SSH 基于 `packages/agent-core/src/ssh/`
   - 团队协作基于现有 `TeamPage` + `services/agent-gateway/src/routes/team.ts`
   - 技能市场与 Provider / Channels 的现有设置页延伸，而不是另起平行配置系统。

3. **按“高价值 / 高耦合优先”排序**
   - 先补 File Change Review 和 Desktop Automation / SSH UX，这些最影响“参考仓库体验差距”。
   - 再补 Team 深协同。
   - 最后补 Prompt Optimizer / Translation 等专项工作流。

## 集成方案（分阶段）

### Phase 1：执行层能力补齐（最高优先级）

#### P1-A 文件改动审阅 / 接受 / 回滚

**目标**
- 把 agent 造成的文件变更做成一等对象。

**集成点**
- `packages/agent-core/`：定义变更快照 / review 状态 / revert API
- `services/agent-gateway/`：持久化 run-level file changes，暴露读取与 revert 路由
- `packages/shared-ui/`：新增 `FileChangeReviewPanel` / `FileDiffCard` / `RevertActionBar`
- `apps/web`、`apps/desktop`：在 Chat / Session / Artifact 视图挂接变更审阅入口

**验收标准**
- 每次 agent 写文件后都能看到 before/after diff
- 用户可对单个变更 accept / revert
- revert 失败有显式错误态

#### P1-B 桌面自动化 / RPA

**目标**
- 在桌面端支持 screenshot / click / type / scroll / wait 等系统级自动化能力。

**集成点**
- `apps/desktop`：新建 Tauri 命令或 sidecar 控制层
- `packages/agent-core`：桌面自动化工具定义与权限模型
- `packages/shared-ui`：桌面自动化权限确认 / 预览结果组件

**验收标准**
- 桌面端 agent 可请求截图与输入操作
- Web 端显示“仅桌面可用”降级态
- 权限模型至少区分 once / session / permanent

#### P1-C SSH 远程文件管理 / 上传 UX

**目标**
- 从现有 SSH 基础能力扩展到真正可用的远程文件体验。

**集成点**
- `packages/agent-core/src/ssh/`：目录树、上传、下载、状态追踪
- `packages/shared-ui`：`SshFileExplorer`、上传进度、远程文件预览
- `apps/web` / `apps/desktop`：增加 SSH 文件浏览入口

**验收标准**
- 可浏览目录树
- 可上传文件并显示进度
- 可在远程文件与本地工作区间切换

---

### Phase 2：协同与治理层增强

#### P2-A 深团队协同

**目标**
- 从“成员 / 任务 / 消息 CRUD”升级到真正的 team orchestration。

**集成点**
- `packages/multi-agent` / `packages/agent-core`：teammate lifecycle、任务认领、阶段回传
- `services/agent-gateway/src/routes/team.ts`：实时状态与事件接口
- `packages/shared-ui/src/TeamPanel.tsx`：成员状态流、任务认领、结果摘要

**验收标准**
- 成员状态随子 Agent 执行实时变化
- 任务可认领 / 完成 / 失败
- lead 能收到子任务结果回传

#### P2-B 更广渠道覆盖

**目标**
- 在现有通道体系上继续补更多 adapter，而不重做频道配置系统。

**优先顺序**
1. WeCom
2. WhatsApp
3. QQ

**集成点**
- `services/agent-gateway/src/channels/`
- `apps/web/src/components/ChannelSubscriptionSettings.tsx`

**验收标准**
- 新渠道使用与现有频道一致的配置/订阅/连接语义

---

### Phase 3：专项工作流补齐

#### P3-A Prompt 优化器

**目标**
- 提供单独的 Prompt 优化工作流，而不是让用户手工在普通对话里试。

**集成点**
- `packages/agent-core`：prompt optimization pipeline / schema
- `packages/shared-ui`：Prompt Optimizer 输入页 + 多候选结果比对卡片
- `apps/web`：挂到独立页面或 settings/developer 工具区

**验收标准**
- 输入一个 prompt，输出多版优化候选
- 可复制 / 应用到技能 / 应用到对话 preset

#### P3-B 翻译工作流

**目标**
- 提供面向文档 / 文本 / 多文件的专门翻译模式。

**集成点**
- `packages/agent-core`：translation mode、术语表、输出缓冲区
- `packages/shared-ui`：源文 / 译文对照与批量任务状态
- `apps/web` / `apps/desktop`：独立翻译页或 Artifact 内翻译入口

**验收标准**
- 单文件翻译
- 多文件批量翻译
- 支持完成标记与导出

## 依赖关系 / DAG

### 可并行流

1. **执行层流**：文件改动审阅 + 桌面自动化 + SSH UX
2. **协同流**：团队协同增强
3. **渠道流**：更多 channel adapter
4. **专项工作流流**：Prompt Optimizer + 翻译工作流

### 关键依赖

- `P1-A 文件改动审阅` 不依赖其它模块，可最先启动
- `P1-B 桌面自动化` 与 `P1-C SSH UX` 可并行
- `P2-A 深团队协同` 依赖现有 team 基础，但不依赖 P1
- `P3-A / P3-B` 可独立推进，但推荐在执行层能力稳定后开始
