# .agentdocs/workflow/260403-lsp-自动使用集成方案.md

## Task Overview

为 OpenAWork 制定一份关于“集成 opencode 风格 LSP 能力，并修复当前系统无法正确自动使用 LSP 工具”的详细执行方案。目标是先完成基于证据的分析与实施规划，而不是立即修改代码。

## Current Analysis

- 参考仓库 `temp/opencode` 已确认具备一条完整的 LSP 工作流：server 启动/发现、LSP 请求封装、diagnostics 聚合、agent/tool 集成、以及 UI/SDK/CLI 暴露。
- 已补充完成“内部所有对话是否考虑 LSP”这一轮链路审计：
  - `runModelRound(...)` 在仓库中的生产调用点只发现两条：`services/agent-gateway/src/routes/stream.ts` 与 `services/agent-gateway/src/routes/stream-runtime.ts`。
  - 这两条路径都会显式构造 `buildCapabilityContext(...)` 与 `buildRequestScopedSystemPrompts(...)`，再把 `enabledTools` 传入模型回合。
  - 因此所有最终汇入这两条主链的内部对话，都会继承当前 session-scoped tool surface 与 capability context。
  - 已确认会汇入这条主链的内部对话入口包括：
    - Web / Desktop / Mobile 经 WS/SSE 发起的普通聊天对话（`stream-routes-plugin.ts -> handleStreamRequest`）
    - 权限批准后的恢复执行（`permissions.ts -> resumeApprovedPermissionRequest -> stream-runtime.ts`）
    - question 回答后的恢复执行（`questions.ts -> resumeAnsweredQuestionRequest -> stream-runtime.ts`）
    - `runSessionInBackground(...)` 驱动的内部对话
    - channel 自动回复（`channels/router.ts -> runSessionInBackground`）
    - GitHub trigger 会话（`github/router.ts -> runSessionInBackground`）
    - task 子会话自动运行（`tool-sandbox.ts -> runSessionInBackground`）
    - 子任务完成后的父会话 auto-resume（`task-parent-auto-resume.ts -> runSessionInBackground`）
- 当前 OpenAWork 并非“没有 LSP”，而是“已有能力但自动使用链路不完整”：
  - `packages/lsp-client` 已支持 `definition/references/documentSymbols/workspaceSymbols/prepareRename/rename/diagnostics` 等能力。
  - `packages/agent-core/src/tools/lsp.ts` 当前仅导出 `lsp_diagnostics` 与 `lsp_touch`。
  - `services/agent-gateway/src/lsp-tools.ts` 已额外定义 `lsp_goto_definition/lsp_find_references/lsp_symbols/lsp_prepare_rename/lsp_rename`。
  - `services/agent-gateway/src/tool-definitions.ts` 与 `tool-sandbox.ts` 已把 richer LSP tools 放进模型可见面与执行白名单。
- 已确认的高概率根因：
  - `session-tool-visibility.ts` 只把 `lsp_diagnostics/lsp_touch` 归类到 channel `read`，未覆盖 richer LSP tools。
  - `stream-system-prompts.ts` 缺少“何时优先使用 LSP”这一层专项 system guidance。
  - `lsp-tools.ts` 与 `tool-definitions.ts` 中对 richer LSP tools 的描述偏短，难以稳定驱动模型自动选用。
  - `agent-core` 与 `gateway` 的 LSP tool surface 不一致，存在长期漂移风险。
  - `services/agent-gateway/src/lsp-tools.ts` 当前直接调用 `lspManager.definition/references/documentSymbols/workspaceSymbols/prepareRename/rename`，**没有像 `opencode` 的 `tool/lsp.ts` 那样在语义查询前先 `touchFile(..., true)`**，这与参考实现存在实际偏差。
- 已确认的“并未全部考虑进去”的缺口：
  - channel 会话虽然也走主链，但 `session-tool-visibility.ts` 对 richer LSP tools 没有 channel tool key 映射，因此这些内部对话并不能被视为“已完整考虑 LSP”。
  - `session-tool-visibility.test.ts` 当前未覆盖 richer LSP tools 的 read/edit 分类。
  - `capabilities.test.ts` / `capabilities-routes.test.ts` 只出现了 `lsp_diagnostics`，没有证明 richer LSP tools 在 session-scoped capability surface 中得到稳定覆盖。
  - `services/agent-gateway/src/verification/` 当前未发现面向 richer LSP tools 自动选用的真实验收链。
- 已确认 `claude-code-tool-surface-profiles.ts` 中 `openawork` profile 不做裁剪，因此问题大概率不在 profile 过滤，而在可见性、提示词与合同一致性层。

## Solution Design

### 目标

1. 把当前 OpenAWork 已有的 richer LSP tools 变成**在真实会话中稳定可见、可调用、可被模型自动正确选择**的能力。
2. 保留 OpenAWork 现有的 `rename/prepareRename` 优势，不机械照搬 opencode。
3. 通过统一 tool contract、session visibility、system prompt guidance 与验证矩阵，形成可长期维护的 LSP 自动使用闭环。

### 参考实现优先原则（新增）

本方案后续实现必须遵守以下约束：

1. **触发语义以 `opencode` 源码为准，不以本地偏好为准。**
2. **不得发明新的 LSP 触发家族。** 当前可接受的触发类型只包括：
   - read 后预热
   - write/edit/apply_patch 后同步 diagnostics
   - 受控的手动语义查询
   - 人工/调试状态查看
3. **prompt 不是主触发器。** 参考库的核心触发逻辑在 tool executor、tool registry 与读写 side effect 中；系统提示最多只做辅助，不得替代实现层语义。
4. **如果 OpenAWork 保留与参考库不同的 API 形态（例如多个 `lsp_*` 工具而不是单个 `lsp` 工具），也必须在内部执行链上对齐参考库语义。**
5. **任何与参考库不一致的本地化改动，都必须在方案中显式标记为“偏离项 + 原因 + 验证方式”。**

### 非目标

- 本期不追求完整 IDE 级 LSP 功能（如 completion/codeAction/semanticTokens）。
- 本期不优先扩展 UI diagnostics 面板或 SDK 新接口，除非它们成为自动使用闭环的硬依赖。
- 本期不直接改动 `temp/opencode` 镜像，也不追求与其全部 naming 100% 对齐。

### 建议分层

1. **Capability layer**：冻结本期 LSP 能力集合（diagnostics/touch/definition/references/symbols/prepareRename/rename）。
2. **Contract layer**：收口单一事实源，避免 `agent-core` 与 `gateway` 双轨定义。
3. **Visibility & Policy layer**：修复 channel/session 对 richer LSP tools 的分类与 gating。
4. **Prompt & Guidance layer**：向模型明确说明“何时用 LSP、何时用 grep、rename 先 prepare”。
5. **Verification layer**：补齐 tool exposure、自动选用、fallback 与 rename 顺序的测试矩阵。

## Trigger Strategy（参考实现优先）

### 参考库中的触发语义

根据已审阅的 `opencode` 源码与文档，LSP 触发并不是单一模式，而是分成 5 类：

0. **按文件类型自动启用 LSP server**
   - 代表文档：
     - `temp/opencode/packages/web/src/content/docs/lsp.mdx`
   - 已证实语义：
     - 当检测到受支持扩展名且依赖条件满足时，LSP server 自动启用。
   - 方案含义：
     - OpenAWork 的自动触发前提必须建立在“文件类型受支持、服务可用”之上，而不是由模型消息内容猜测是否该开 LSP。

1. **手动语义查询触发**
   - 代表文件：
     - `temp/opencode/packages/opencode/src/tool/lsp.ts`
     - `temp/opencode/packages/opencode/src/tool/registry.ts`
     - `temp/opencode/packages/web/src/content/docs/lsp.mdx`
   - 含义：`goToDefinition / findReferences / documentSymbol / workspaceSymbol / goToImplementation / call hierarchy` 这类能力，是在 agent/模型明确需要语义导航时才调用。
   - 特征：
     - 不是后台常驻自动轮询。
     - 受实验开关控制（`tool/registry.ts` 中由 `Flag.OPENCODE_EXPERIMENTAL_LSP_TOOL` 决定是否注册 `LspTool`）。
     - 在真正执行查询前，`tool/lsp.ts` 会先 `await LSP.touchFile(file, true)`。
     - 属于“按需手动调用”的代码理解工具，而不是每轮默认执行。

2. **自动预热触发（read 路径）**
   - 代表文件：
     - `temp/opencode/packages/opencode/src/tool/read.ts`
   - 含义：当系统读取代码文件时，会顺手触发 LSP 预热/触达，让后续 diagnostics 与符号能力更容易命中。
   - 特征：
     - `read.ts` 在成功读取文件后直接执行 `LSP.touchFile(filepath, false)`。
     - 自动触发，但属于轻量 side effect。
     - 目标是让后续语义能力 ready，而不是立刻替代 read 的主要职责。

3. **自动刷新触发（write/edit/apply_patch 路径）**
   - 代表文件：
     - `temp/opencode/packages/opencode/src/tool/write.ts`
     - `temp/opencode/packages/opencode/src/tool/edit.ts`
     - `temp/opencode/packages/opencode/src/tool/apply_patch.ts`
   - 含义：代码变更成功后，自动 `touchFile + diagnostics`，把 LSP 状态与新文件内容重新对齐。
   - 特征：
     - 自动触发。
     - `write.ts` / `edit.ts` / `apply_patch.ts` 都是在**文件改动落盘成功后**才触发 `touchFile(..., true)` 与 `diagnostics()`。
     - 触发点明确依附于“代码已改动成功”这一事件。
     - 属于写后同步，不是独立的模型决策行为。

4. **人工/运维观测触发**
   - 代表文件：
     - `temp/opencode/packages/opencode/src/cli/cmd/debug/lsp.ts`
     - `temp/opencode/packages/opencode/src/server/instance.ts`
     - `temp/opencode/packages/sdk/js/src/gen/sdk.gen.ts`
   - 含义：通过 debug CLI、HTTP `GET /lsp`、SDK `client.lsp.status()` 等方式查看 LSP 状态。
   - 特征：
     - 手动触发。
     - 面向人类操作者/调试面板，不是面向模型的默认回合动作。

### OpenAWork 应采用的触发规则（按参考实现收敛）

#### A. 自动触发（应该自动发生）

1. **代码文件首次读入后的轻量预热**
   - 必须镜像 `opencode` 的 `tool/read.ts`：
     - 在**成功完成代码文件读取后**，于 read 执行链内部触发一次 `touch(false)` 等价语义。
   - 这不是额外的模型工具选择步骤，而是 read 实现内部 side effect。
   - 原则：
     - 自动，但轻量、幂等、低噪音。
     - 仅对受支持代码文件生效。
     - 不能改写成“先由 prompt 决定是否 warmup”。

2. **代码写入后的状态刷新**
   - 必须镜像 `opencode` 的 `write.ts / edit.ts / apply_patch.ts`：
     - 在**文件改动成功落盘后**，于写工具内部触发 `touch(true)`。
     - 随后立即获取 `diagnostics()`，并把错误反馈回本次工具结果。
   - OpenAWork 若把 `lsp_rename` 视作语义级写操作，也应并入同一套“写后 touch + diagnostics”语义。
   - 原则：
     - 这是写后同步，不依赖模型再次决定。
     - 应只在“文件真实改动成功”后发生。

3. **后台恢复执行沿用当前主链，不额外偷跑 LSP 查询**
   - 对 permission/question resume、task child run、parent auto-resume、channel、GitHub trigger：
     - 应继续复用统一 `handleStreamRequest / stream-runtime` 主链。
     - 自动触发只限于 capability context、tool visibility、生效的系统提示与写后 `touch/diagnostics`。
     - 不应在这些路径中偷偷自动执行 `definition/references/symbols` 之类语义查询。

#### B. 手动触发（由模型按需选择）

1. **精确语义导航**
   - 参考 `opencode` 的单一 `LspTool` 语义。
   - OpenAWork 若保留 `lsp_goto_definition / lsp_find_references / lsp_symbols` 这些拆分工具，必须让它们共享同一套内部执行器语义：
     - 先检查是否有可用 client
     - 再执行 `touch(true)`
     - 再进行 definition/references/symbols 查询
    - 触发条件：
      - 用户在问“定义在哪”“谁引用了它”“这个文件/工作区有哪些符号”
      - 当前问题是**符号级语义查询**，不是普通文本搜索
    - 规则：
      - 由模型按需手动调用。
      - 不应作为每轮默认动作。
      - 参考 `opencode` 会放在 feature flag / controlled rollout 之下；OpenAWork 本期已决策暂不引入该门控，先以显式工具可见性、系统提示约束与测试/verification 保证行为可控。

2. **受保护的重命名链路**
    - `lsp_prepare_rename` → `lsp_rename`
    - 触发条件：
      - 用户明确要求“重命名符号/全局 rename”
     - 或实现任务中需要语义级 rename 才安全
   - 规则：
      - 必须先 `prepareRename`，后 `rename`
      - **绝不自动 rename**

3. **未来扩展能力的手动触发**
    - 若后续补 `hover / implementation / call hierarchy`
    - 也应遵循 `opencode` 的模式：
      - 由同类手动语义工具承载
      - 在工具内部先做 `touch(true)`
      - 不纳入默认自动执行链

#### C. 条件触发（有前提才触发）

1. **LSP capability 仅在支持语言/有效 workspace 下触发**
   - 如果当前文件语言不支持、server 未启动、workspace root 不合法：
     - 不要强推 LSP
     - 直接走 fallback

2. **channel/internal session 下的 richer LSP 要以 visibility 先打通为前提**
   - 只有在 `session-tool-visibility.ts` 完整映射 richer LSP tools 后，才能宣称这些内部对话“已支持 LSP 自动正确使用”。

3. **仅当请求是代码理解/代码修改任务时才加载 LSP guidance**
    - 普通闲聊、运营通知、纯自然语言问答，不应因为 session 里有代码上下文就触发 LSP 指令偏好。

4. **任何 prompt guidance 都只能是辅助，不得替代 executor 级触发**
   - 参考实现里 read/write/lsp 的关键行为都发生在工具实现内部。
   - 因此 OpenAWork 的 `stream-system-prompts.ts` 只能辅助模型选工具，不能承担 warmup、pre-touch、write 后同步这些责任。

#### D. 禁止自动触发（必须明确禁止）

1. **禁止自动 rename**
   - 不允许模型因为“看起来应该统一命名”就自动执行 `lsp_rename`

2. **禁止每轮都自动跑 definition/references/symbols**
    - 这些属于语义检索动作，不应变成默认 background noise

3. **禁止把 prompt guidance 当成主实现机制**
   - 如果没有在 read/write/lsp executor 内部实现与参考库等价的 side effect，单靠提示词不算完成参考实现对齐

4. **禁止在 channel 会话中因 policy 未配置而隐式放开 richer LSP**
   - 要么明确映射为 `read/edit` 并受权限控制
   - 要么明确不可见
   - 不能处于“定义存在但行为不确定”的灰区

#### E. 失败回退（必须写进方案）

参考 `opencode` 的设计思想，OpenAWork 需要把 LSP 视为“优先的语义工具”，不是“唯一工具”。因此需要明确：

1. `lsp_goto_definition / references / symbols` 失败或无结果
   - fallback 到 `grep + read`

2. `prepareRename` 不通过
   - 禁止继续 `rename`
   - 回退到人工确认或文本级替代方案

3. `touch/diagnostics` 失败
   - 不影响写操作本身的提交结果
   - 但要把 diagnostics 获取失败视为可观测事件，而不是静默吞掉

### 当前方案相对参考实现的偏离项（需修正）

1. **偏离项：把 `stream-system-prompts.ts` 写成主要触发层**
   - 修正：降级为辅助层；真正触发语义必须放回 read/write/lsp executor。

2. **偏离项：把手动语义查询前的预同步写成外部规则，而不是工具内部行为**
   - 修正：参考 `opencode` 的 `tool/lsp.ts`，在 LSP 查询工具 execute 内部先 `touch(true)`。

3. **偏离项：直接照搬参考库的 rollout 假设，但未结合 OpenAWork 当前风险面做取舍**
   - 修正：确认参考库使用 feature flag / controlled rollout；OpenAWork 本期经评估决定暂不引入该门控，后续仅在出现真实误触发或运营灰度需求时再补。

4. **偏离项：未明确 read/write side effect 必须是实现层钩子，不是额外的模型调用**
   - 修正：参考 `read.ts / write.ts / edit.ts / apply_patch.ts`，将 warmup 与 diagnostics 同步绑定到工具内部生命周期。

### 方案层面的落地要求

为了把“触发策略”真正落地，而不是只写概念，需要把它分别写进：

1. `stream-system-prompts.ts`
   - 写入“何时优先 LSP、何时用 grep、rename 先 prepare”的模型 guidance

2. `session-tool-visibility.ts`
   - 写入 channel / internal session 下 richer LSP 的可见性规则

3. `lsp-tools.ts` / `tool-definitions.ts`
   - 写入工具 description 中的触发意图、保护条件与副作用边界

4. `verification/` 与测试
    - 验证自动触发（read/write 后）
    - 验证手动触发（definition/references/symbols/rename）
    - 验证禁止自动触发（不应自动 rename / 不应每轮自动语义检索）
    - 验证失败回退（LSP unavailable → grep/read）

## Risks

- 只修 prompt 不修 visibility，会出现“模型想调用但工具根本不在真实会话里”的假修复。
- 只修 visibility 不修 guidance，会继续出现“工具存在但模型默认走 grep”的问题。
- 若继续让 `agent-core` 与 `gateway` 各自维护 LSP tool 定义，后续 prompt/schema/allowlist 很快再次漂移。
- `lsp_rename` 是多文件编辑，若不把其纳入 edit 权限与 diff/observability 语义，后续审计与权限模型会失真。
- 若不把“自动触发 vs 手动触发”写清楚，后续实现极易出现两种反模式：
  - 什么都自动跑，造成噪音、性能浪费和行为不可控
  - 什么都不自动跑，导致 richer LSP 永远只停留在“定义存在”而不是“真实好用”
- 若不把“参考实现不变量”冻结下来，后续实现很容易演变成“接口名称像 opencode，但执行语义不像 opencode”的伪对齐。

## Verification Strategy

- 单元测试：tool definitions、session visibility、capabilities、system prompts。
- 集成测试：stream 自动选用 definition/references/symbols/prepareRename/rename。
- 负向测试：channel 只读权限、LSP 空结果、语言不支持、fallback 到 grep/read。
- 验收脚本：在 `services/agent-gateway/src/verification/` 增加真实回归链，证明“自动正确使用”而非“定义存在”。
- 触发策略测试：
  - read 代码文件后发生轻量预热
  - write/edit/apply_patch/lsp_rename 成功后触发 touch/diagnostics
  - definition/references/symbols 仅在语义查询场景触发
  - rename 必须先 prepareRename，且永不自动触发
  - LSP 不可用时稳定回退到 grep/read
  - 手动语义查询工具在 execute 内部先 `touch(true)`，而不是依赖外层 prompt
  - richer LSP 手动查询能力当前不引入 feature flag；若未来补门控，需额外增加 rollout gating 验证
