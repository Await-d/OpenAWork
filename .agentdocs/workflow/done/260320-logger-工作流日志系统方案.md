# 工作流日志系统方案

## 任务概述

参考外部服务的结构化请求日志格式（树状 workflow 步骤、emoji 状态、耗时、关键字段），在 OpenAWork 中实现等价的日志打印系统。

## 设计目标

1. **后端**：每个 HTTP 请求产生一条树状 workflow 日志，格式与参考日志一致
2. **前端**：封装统一 Logger 替代裸 console，支持级别过滤和 ring buffer
3. **复用**：以 `packages/logger` 共享包形式提供，gateway 和未来其他服务均可接入

---

## 参考日志格式分析

目标输出格式（基于用户提供样本）：

```
[21:19:32 INF] 🟢 200 POST /v1/messages 15535ms -
├── requestId: 0HNK5VR0TA463:00000001
├── workflow:
│   ├── [成功] http.request.start - 请求开始：路径=/v1/messages
│   ├── [成功] ingress.validate-api-key (0ms)
│   ├── [进行中] auth.order-resolve - protocol=Claude, path=/v1/messages
│   ├── [成功] upstream.response - status=200, stream=True
│   ├── [进行中] upstream.http.stream - provider=anthropic, model=...
│   │   ├── [成功] upstream.http.stream.usage-callback (0ms)
│   │   └── [成功] upstream.http.stream (264ms)
│   └── [成功] usage.audit - order=..., tokens=141338
├── res: status=200, provider=anthropic
├── ip: 117.151.83.161
└── ua: ai-sdk/anthropic/2.0.65
```

关键设计点：
- **步骤状态**：`[成功]` / `[进行中]` / `[失败]`（对应 success/pending/error）
- **步骤耗时**：`(264ms)` 仅在有意义时显示
- **树状前缀**：`├──` / `└──` / `│   ` 标准 Unicode 树形
- **嵌套子步骤**：缩进一级，`│   ├──` / `│   └──`
- **请求行**：HTTP method + path + total ms + 状态 emoji
- **元数据**：requestId、ip、ua 平铺在 workflow 外

## 方案设计

### 包结构

```
packages/logger/
├── package.json          # @openAwork/logger
├── tsconfig.json
└── src/
    ├── types.ts          # LogLevel, WorkflowStep, RequestContext
    ├── workflow-logger.ts # WorkflowLogger — 核心步骤树引擎
    ├── fastify-plugin.ts  # requestLoggerPlugin — Fastify 自动埋点
    ├── frontend-logger.ts # FrontendLogger — console 封装 + ring buffer
    └── index.ts          # 统一导出
```

### 核心类型（types.ts）

```typescript
export type LogLevel = 'trace' | 'debug' | 'info' | 'warn' | 'error';

export type StepStatus = 'success' | 'pending' | 'error';

export interface WorkflowStep {
  name: string;          // e.g. 'ingress.validate-api-key'
  status: StepStatus;
  message?: string;      // e.g. '请求开始：路径=/v1/messages'
  durationMs?: number;   // 仅在有意义时设置
  fields?: Record<string, string | number | boolean>; // key=value 附加字段
  children?: WorkflowStep[];
}

export interface RequestContext {
  requestId: string;
  method: string;
  path: string;
  ip?: string;
  userAgent?: string;
  startTime: number;     // Date.now()
}
```

### WorkflowLogger（workflow-logger.ts）

```typescript
export class WorkflowLogger {
  private steps: WorkflowStep[] = [];

  // 开始一个根步骤（返回步骤引用，可加子步骤）
  start(name: string, message?: string, fields?: Record<string, unknown>): WorkflowStep

  // 完成一个步骤（记录耗时）
  succeed(step: WorkflowStep, message?: string, fields?: Record<string, unknown>): void
  fail(step: WorkflowStep, message?: string): void

  // 打印完整树状日志（请求结束时调用）
  flush(ctx: RequestContext, statusCode: number): void
}
```

**flush 输出逻辑：**
1. 首行：`[HH:mm:ss INF] 🟢/🔴 {statusCode} {method} {path} {ms}ms -`
2. `├── requestId: {id}`
3. `├── workflow:` + 递归树状步骤
4. `├── res: status={code}, ...`
5. `├── ip: {ip}`
6. `└── ua: {ua}`

### requestLoggerPlugin（fastify-plugin.ts）

Fastify `onRequest` hook：
- 创建 `WorkflowLogger` 实例，挂到 `request.workflowLogger`
- 注入 `request.workflowContext`（requestId, method, path, ip, ua）

Fastify `onResponse` / `onError` hook：
- 调用 `request.workflowLogger.flush(ctx, reply.statusCode)`

路由内使用：
```typescript
const step = request.workflowLogger.start('ingress.validate-api-key');
// ... 做事 ...
request.workflowLogger.succeed(step, undefined, { ms: 0 });
```

### FrontendLogger（frontend-logger.ts）

```typescript
export class FrontendLogger {
  constructor(options: { level?: LogLevel; ringBufferSize?: number; prefix?: string })
  debug(msg: string, ...args: unknown[]): void
  info(msg: string, ...args: unknown[]): void
  warn(msg: string, ...args: unknown[]): void
  error(msg: string, ...args: unknown[]): void
  getLogs(): LogEntry[]   // 返回 ring buffer 内容
  clear(): void
}
```

特性：
- 按 `level` 过滤输出
- Ring buffer（默认 200 条），避免内存无限增长
- 在 console.log 基础上加 `[prefix][LEVEL][HH:mm:ss]` 前缀
- 生产环境自动静默 debug/trace
