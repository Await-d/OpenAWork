# .agentdocs/workflow/260324-子任务层级实现方案.md

## 任务概览

基于 `temp/opencode` 与 `temp/oh-my-openagent` 的参考实现，为 OpenAWork 补齐真正的一等“子任务”能力。目标不是继续复用“子会话近似子任务”的表达，而是在现有 `task-system`、gateway、shared event contract 与 Chat 右侧面板之上，建立可持久化、可投影、可展示的父子任务层级结构。

## 当前分析

- 当前仓库已有两条相近能力线：
  - `services/agent-gateway/src/routes/sessions.ts` + `apps/web/src/pages/ChatPage.tsx` 的 `parentSessionId` 子会话分支
  - `packages/agent-core/src/task-system/*` 的依赖图任务系统（`blockedBy[]`）
- 任务系统当前缺少层级字段：`AgentTask` 尚无 `parentTaskId` / `children` 等父子关系语义，因此右侧任务面板只能消费平铺任务列表。
- `.agentdocs/index.md` 中已冻结一条关键决策：会话级任务与子会话可见性采用“SQLite sessions + `.agentdocs/tasks` 双源收敛”，这意味着本次实现必须在现有双源结构上做向后兼容，而不是推翻。
- 用户已明确要求参考 `temp/opencode` 与 `temp/oh-my-openagent` 落地实现，因此本次任务必须先对照参考仓库，再确定最终模型、事件与 UI 投影方式。

## 方案设计

### 目标收敛

1. **一等子任务语义进入 task-system**：父子关系必须在 `AgentTask` 模型中可表达，并能和 `blockedBy` 共存。
2. **gateway 继续做投影层**：会话的 `/sessions/:id/tasks` 返回结果需要能表达层级，不让前端自行重建任务树。
3. **shared 协议同步升级**：若运行时事件或任务 DTO 缺少父子关系，需要在共享类型中补齐。
4. **UI 先以现有面板兼容扩展**：优先在现有 Chat 右侧任务面板/相关会话视图中呈现层级，不新增影子状态机。
5. **子会话保留为并行补充语义**：子会话仍可作为“从消息分叉的新上下文”，但不再承担“真正子任务”的主语义。

## Complexity Assessment

- Atomic steps: 5+ → +2
- Parallel streams: 主仓现状 / opencode 参考 / oh-my-openagent 参考 → +2
- Modules/systems/services: agent-core + shared + agent-gateway + web(+可能 mobile) → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 该任务是跨数据模型、服务端投影、共享协议与 UI 呈现的联动改造，还要求以两个参考仓库校准设计，必须保留明确的阶段计划和状态同步，避免把“子会话”“依赖图”“子任务树”三种语义再次混淆。

## Acceptance Criteria

- `AgentTask` 层能表达真正父子任务关系，而不只是依赖关系
- `/sessions/:id/tasks` 返回结果能直接支撑前端识别父子结构
- 现有 `blockedBy` 依赖图逻辑继续可用，不因层级化而退化
- 现有 `parentSessionId` 子会话分支继续可用，并与子任务语义区分清楚
- 前端任务视图能消费新的父子关系信息，而不是本地猜测重建
