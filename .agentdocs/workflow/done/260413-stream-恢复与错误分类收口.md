# Stream 恢复与错误分类收口

## Task Overview

在 message-v2 主线归档后，继续收口 gateway 剩余改动中最值得优先推进的一簇：**stream-model-round / upstream-error / resume reconcile**。目标是把流式回合在上游 429 / context overflow / 审批后恢复等非 happy path 下的错误分类、恢复信号与 V2 step 元数据写入补齐，并通过最贴边的单测、typecheck 与 verification 验证。

## Current Analysis

- 剩余 gateway 改动并不属于同一主线，至少分成三簇：
  - `stream-model-round` / `upstream-error` / `stream-resume-reconcile`
  - `command-loop-runtime`
  - `look-at-tools`
- 本轮只收口第一簇，因为它直接影响流式执行在溢出、429 与 resume 场景下的用户可见行为，而且验证面最大。
- `services/agent-gateway/scripts/message-v2-*.mjs` 仍属于 message-v2 手工脚本尾巴，本轮明确排除。

## Solution Design

- 在 `stream-model-round.ts` 中，把 round 级 snapshot / patch 元数据补写到 V2 adapter：
  - assistant 回合落定后，除保留原有 `persistSessionSnapshot(...)` 之外，再写 `appendSnapshotPart(...)` 与 `appendPatchPart(...)`
- 在 `upstream-error.ts` 中显式区分 429：
  - `quota_exceeded`
  - `rate_limit`
  - `unknown`
- 在 stream 相关测试里补齐：
  - `stream-model-round.test.ts` 的 V2 adapter mock
  - `stream-resume-reconcile.test.ts` 的 `sqliteTransaction` mock
  - `upstream-error.test.ts` 的 429 分类回归

## Complexity Assessment

- Atomic steps: 3–4（round 写入、错误分类、测试/verification、提交） → 0
- Parallel streams: no（这是一条收口单线） → 0
- Modules/systems/services: 3+（stream round / upstream error / verification） → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 2
- **Chosen mode**: Lightweight
- **Routing rationale**: 这是独立且边界清晰的一条 gateway 收口线，虽不需要完整 runtime 目录，但仍值得记录 workflow 与验证面。
