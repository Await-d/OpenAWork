# .agentdocs/workflow/260403-操作电脑语音github工作流集成.md

## Task Overview

将参考自 `temp/claude-code-sourcemap` 的三类能力在当前 OpenAWork 中收口为可用主链：

1. 操作电脑（首期收口为桌面浏览器自动化 / screenshot / click / type 的 agent 可用能力）
2. 语音模式（首期收口为 Web/Desktop 语音输入可用，保持 Mobile 既有能力一致）
3. Git/GitHub 工作流（首期收口为真实可触发的 GitHub 工作流入口与最小自动执行闭环）

## Current Analysis

- `services/agent-gateway/src/routes/desktop-automation.ts` 与 `services/agent-gateway/src/desktop-automation.ts` 已存在桌面自动化路由与管理器，但尚未成为模型可直接调用的正式工具面。
- `packages/browser-automation` 已提供 `DesktopBrowserAutomation`，说明“操作电脑”首期不需要从零接入底层自动化驱动。
- `packages/shared-ui/src/VoiceRecorder.tsx` 已有录音 UI，但没有实际填充 transcript；`apps/mobile/src/components/MobileVoiceRecorder.tsx` 已有完整 STT 识别闭环，可作为 Web/Desktop 对齐参照。
- `services/agent-gateway/src/github/router.ts` 与 `src/github/github-trigger.ts` 已有 GitHub trigger 注册与 webhook 处理，但当前 route handler 只创建会话壳子，还未形成真正的自动执行闭环。
- `apps/web/src/pages/SettingsPage.tsx` 与 `apps/web/src/pages/settings/workspace-tab-content.tsx` 已暴露 desktop automation 与 GitHub trigger 的设置/调试入口，适合作为首期运营入口；聊天主链则已有 `ChatComposer` + `VoiceRecorder` 挂点。

## Solution Design

- 采用“已有能力主链化”方案：优先复用现有 gateway route、tool definition、shared-ui、desktop sidecar 和 settings/chat 挂点，不另起平行系统。
- 操作电脑首期收口为 **desktop browser automation tool**：把现有 desktop automation manager 暴露为 agent 可用工具，并在 capability / visibility / tool definitions 中可见。
- 语音模式首期收口为 **Web/Desktop STT 输入闭环**：为共享 `VoiceRecorder` 接入浏览器 SpeechRecognition 能力；Mobile 保持现有 `expo-speech-recognition` 实现不动。
- Git/GitHub 工作流首期收口为 **GitHub trigger → session 创建 → 自动续跑入口** 的最小闭环，并补充前端/命令可见入口或状态反馈。
- 所有变更优先落在现有 canonical surface 上；只有必要时再补 reference-style visible names。

- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1

## Notes

- 首期“操作电脑”明确收口为桌面浏览器自动化，不承诺一次实现鼠标/键盘/窗口级全系统 computer-use。

## Delivery Summary

- Desktop automation 已收口为 gateway-managed `desktop_automation` 工具，复用现有 manager，并修复了 Web 设置页截图请求 method 与后端路由不一致的问题。
