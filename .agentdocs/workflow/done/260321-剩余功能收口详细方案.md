1. Chat 流式事件链路未将 `tool_call_delta` / 计划 / 可视化面板打通

## Solution Design

### 核心原则

1. **代码真相优先**：以当前仓库代码而非旧 checklist 作为执行真相源
2. **先锁契约再补 UI**：先统一 stream 事件契约，再收口 Chat 右侧面板
3. **Web Settings 先稳定，Mobile 再对齐**：避免跨端配置模型分叉
4. **平台能力通过 adapter 落地**：Artifacts 不直接在页面里分叉实现
5. **文档作为控制车道**：每个阶段结束即回写 `.agentdocs`，避免再次漂移

### 已冻结的流式契约（T-02）

- Web Chat 的客户端事件真相源统一为 `packages/shared/src/index.ts` 中的 `StreamChunk`
- 对外客户端只消费四类 chunk：`text_delta` / `tool_call_delta` / `done` / `error`
- `start` / `requestId` 若仍需保留，仅允许作为网关内部或传输层调试字段，不能再作为 Web UI 的核心渲染协议
- WS 与 SSE 必须输出同构 chunk；`packages/web-client` 与 `apps/web/src/hooks/useGatewayClient.ts` 最终应收敛为同一事件语义
- 第一阶段仅要求打通最小真实工具源，优先使用仓库内已存在且最稳定的 `web_search`，不在此阶段扩展全量工具暴露

### UI-01 最小交付口径（T-03）

- 触发一次真实 `web_search` 工具调用后，**Plan** 标签页至少出现 1 条非空任务
- 同一次会话中，**Tools** 标签页至少显示 1 张真实 `ToolCallCard`，可见工具名与输入
- 同一次会话中，**Viz** 标签页至少出现 `agent_started → tool_call → tool_done/agent_done` 这一条最小事件链中的 3 个以上事件
- WS 正常路径与 SSE fallback 路径都必须满足以上 3 条
- 未触发真实工具调用前，允许保留空态；一旦触发工具调用，禁止继续显示“暂无工具调用记录”占位卡作为唯一内容

### 推荐工作流

- **P0**：文档基线校准 + Chat 流式链路 + Web Settings 去 demo 化
- **P1**：Mobile Settings 产品化 + Artifacts 平台集成
- **P2**：Multi-Agent 编排闭环与剩余文档清理

## Complexity Assessment

- Atomic steps: 12+ → +2
- Parallel streams: Chat / Web Settings / Mobile Settings / Artifacts / Docs → +2
- Modules/systems/services: web + shared + web-client + gateway + mobile + artifacts + docs → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: +6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 剩余工作跨协议、前后端、移动端、平台适配与文档治理，多条流可以并行，但必须先锁定 Chat 事件契约和 Settings 真相源，否则后续会反复返工。

- **Lane A（核心）**：Phase 0 + Phase 1

- Chat 右侧 Plan / Tools / Viz 不再长期为空，且至少能显示一次真实工具事件
- Web Settings 页面不再保留显眼的 demo 值、空回调入口和假仓库示例
- Mobile Settings 具备真实持久化与重启恢复能力
- Artifacts 的 open/share/download 在各平台均有明确行为与降级策略
- `.agentdocs` 状态与当前代码一致，活跃/已完成 workflow 不再互相矛盾
