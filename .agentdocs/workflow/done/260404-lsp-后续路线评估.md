# 260404 LSP 后续路线评估

## Task Overview

在 `260403-lsp-自动使用集成方案` 与 `260404-lsp-hover-轻量增强实施` 已归档的基础上，继续推进 OpenAWork 的 LSP 主线，但本轮先不盲目实现新能力，而是评估下一条最值得做的 follow-up：

1. `implementation`（go to implementation）
2. call hierarchy（prepare / incoming / outgoing）
3. 扩展 gateway 侧 language server 覆盖

目标是基于当前仓库真实状态、既有 LSP 路线决策与外部协议约束，给出明确优先级、依赖和推荐下一步。

## Current Analysis

- 已归档的 richer LSP Phase 1 已完成 definition / references / symbols / prepareRename / rename / diagnostics / touch 主链收口。
- `lsp_hover` 作为最小 follow-up 也已完成归档，当前 richer LSP tool surface 已包含 8 个工具。
- 最近一轮修补已补齐：
  - `lsp_find_references.includeDeclaration` 真实透传
  - `references / symbols` 正向 verification
  - `lsp_hover` guidance
  - `lsp_rename` permission gating
  - `packages/lsp-client/src/client.ts` initialize capabilities 对齐
- 当前已知未做的后续方向主要剩三条：
  - `implementation`
  - call hierarchy
  - 更多 gateway-side language server
- `.agentdocs/index.md` 的 durable memory 已明确记录：implementation / call hierarchy / status-event-UI diagnostics 需要在补齐 client capabilities 与明确消费场景后再推进。

## Solution Design

- 本轮先做**路线评估**，不是直接扩 scope 到 UI、事件消费或大范围 server 接入。
- 评估维度固定为：
  1. 当前仓库已有底座
  2. 需要改动的模块/文件面
  3. 外部协议/能力声明复杂度
  4. 验证矩阵成本
  5. 用户收益与误用风险
- 结论输出要尽量收敛成：
  - 推荐优先路线
  - 暂缓路线及原因
  - 如果适合继续推进，下一条 workflow 应如何定 scope

## Complexity Assessment

- Atomic steps: 5+ → +2
- Parallel streams: 仓库现状 / 外部协议资料 / workflow 同步 → +2
- Modules/systems/services: `.agentdocs` + `packages/lsp-client` + `packages/agent-core` + `services/agent-gateway` → +1
- Long step (>5 min): yes → +1
- Persisted review artifacts: yes → +1
- OpenCode available: yes → -1
- **Total score**: 6
- **Chosen mode**: Full orchestration
- **Routing rationale**: 该任务需要并行梳理内部实现、外部协议与已有工作流决策，并把结论沉淀为下一条 LSP 主线的正式真相源，因此按 full orchestration 管理最稳妥。

## Notes

- 本轮仍遵循既有边界：不要把 LSP 主线扩成前端 diagnostics 面板或更大的 IDE 项目。
- 若 external protocol 与当前实现状态出现冲突，以“最小保守声明 + 可验证闭环”为优先原则。
- 路线比较结论：`implementation` 应优先于 call hierarchy 与 language server coverage。原因是它与现有 `definition` 在协议形状上完全同构、验证面最小、收益明确，最适合作为 hover 之后的下一条最小增量。
- 暂缓结论：call hierarchy 需要 prepare / incoming / outgoing 三步与新类型透传，复杂度明显更高；language server coverage 更依赖运行环境与 CI 外部条件，当前不应抢在 richer semantic query 工具前面推进。
