# .agentdocs/workflow/260402-对话级文件变更记录保障融合方案.md

## Task Overview

为 OpenAWork 设计并冻结一套“**对话级文件变更详细记录与保障**”的正式实施方案。目标不是新建平行子系统，而是在当前已存在的 `session_run_events + session_file_diffs + session_snapshots + audit_logs` 基础上，融合 OpenCode 的结构化快照/差异/查询能力与 Claude Code 的写前原文备份/恢复语义，形成可审计、可回放、可恢复、可回滚的统一主链。

## Current Analysis

- 当前仓库已经具备 durable 基础：
  - `services/agent-gateway/src/session-run-events.ts` → `session_run_events`
  - `services/agent-gateway/src/session-file-diff-store.ts` → `session_file_diffs`
  - `services/agent-gateway/src/session-snapshot-store.ts` → `session_snapshots`
  - `services/agent-gateway/src/tool-sandbox.ts` / `db.ts` → `audit_logs`
- `services/agent-gateway/src/routes/stream.ts` 已在工具结果后抽取结构化 diff 并落 `session_file_diffs`；`stream-model-round.ts` 已在回合结束时落 `session_snapshots`。
- `packages/agent-core/src/tools/hash-edit.ts` 已具备哈希锚定编辑安全能力，但尚未形成完整的“写前备份 → 恢复预演 → 正式回滚”闭环。
- `packages/artifacts` 与 `services/agent-gateway/src/routes/artifacts.ts` 已提供可复用的文件型产物存储能力，适合作为原文备份/恢复包的载体，而非把大文本直接塞进主表。
- 已知架构约束：仓库历史记忆已冻结“后端安全与留痕优先采用在现有 SQLite 主事实上增量扩表，而不是重做 event-sourcing”，因此本方案必须避免引入第二套并列真相源。
- 已知风险：
  - 仅依赖结构化 diff 时，遇到 bash/外部进程/重命名等场景，恢复能力不稳。
  - 仅依赖文件备份时，审计与查询会退化，难以构建 request-scoped replay 与稳定 API。
  - 如果把 SQLite 主链与独立 snapshot repo 同时当真相源，会造成状态漂移与恢复歧义。

## Solution Design

### 核心原则

1. **SQLite 是唯一真相源**：`session_run_events + session_file_diffs + session_snapshots` 组成对话级文件变更主链。
2. **写前原文备份是强恢复兜底层**：对可控写类工具，必须在写入前保存原文副本。
3. **展示层不是事实源**：UI 可以消费 turn diff / summary，但不得以 tool result 临时拼接结果取代主链。
4. **恢复优先级固定**：`写前备份 → 快照恢复 → 反向 patch → git/workspace fallback`。
5. **分层演进，不推倒重来**：优先强化 gateway 当前主链，再把 agent-core 的安全编辑与 artifacts 的文件存储嫁接进来。

### 推荐分层架构

#### 1. Truth Source（主记录层）
- 真相表：
  - `session_run_events`
  - `session_file_diffs`
  - `session_snapshots`
- 统一关联键：
  - `sessionId`
  - `clientRequestId`
  - `toolCallId`
  - `round/seq`
  - `filePath`
- 目标：所有写类动作都必须留下可重建的请求链、工具链与文件链。

#### 2. Backup Layer（写前备份层）
- 新增独立元数据表，例如：`session_file_backups`
- 真实文件内容落 artifacts / 专用文件目录，避免主表膨胀。
- 支持：
  - `before_write`
  - `after_write`（可选）
  - `snapshot_base`
- 优先覆盖文本文件；二进制与 notebook 作为二阶段扩展。

#### 3. Presentation Layer（读模型与展示层）
- 提供统一查询面：
  - 会话级文件变更汇总
  - 请求级文件变更详情
  - snapshot 列表与对比
  - 变更可追溯时间线（request → tool → file）
- 在消息内容中保留 `modified_files_summary` 作为 UI 快速展示，但其数据必须来自 durable 主链。

#### 4. Recovery Layer（恢复与回滚层）
- 先支持 dry-run / validate-only，再开放 apply。
- 用 `hash-edit` 做 patch 可应用性校验。
- 用 `workspace-review` / git fallback 兜底非结构化工具改动。

### 取舍结论

#### 应吸收 OpenCode 的部分
- 结构化 diff 作为正式产物，而不是 UI 衍生物
- 明确的 step/round/request 边界与对外查询模型
- snapshot 作为读模型与审计支点
- replay / event exposure 的一等语义

#### 不建议第一阶段照搬 OpenCode 的部分
- 把独立 snapshot git repo 作为主真相源
- 双主存储并行维护（SQLite + 独立 repo 同时 authoritative）

#### 应吸收 Claude Code 的部分
- 写前原文备份
- message/turn 级恢复语义
- 用户可理解的 rewind / restore 视角

#### 不建议第一阶段照搬 Claude Code 的部分
- 让 transcript / JSONL 成为唯一事实源
- 仅靠 `structuredPatch`/tool result 作为主 diff 语义
- 无约束的全量快照膨胀

## Evidence Baseline

### 当前仓库承接链（已确认）

- `services/agent-gateway/src/routes/stream.ts`
  - 初始化 `turnFileDiffs`
  - 在工具结果后执行 `collectFileDiffsFromToolOutput()` / `mergeFileDiffs()`
  - 调用 `persistSessionFileDiffs()` 把结构化 diff 写入 `session_file_diffs`
- `services/agent-gateway/src/routes/stream-model-round.ts`
  - 在非 `tool_use` 回合结束时执行 `persistSessionSnapshot()`
  - 通过 `buildModifiedFilesSummaryContent()` 把摘要写入 assistant message content
- `services/agent-gateway/src/session-run-events.ts`
  - request-scoped durable replay 主链，支持 `client_request_id + seq` 顺序回放
- `services/agent-gateway/src/tool-sandbox.ts`
  - 统一工具执行与权限门控入口，同时承担 `audit_logs` 审计写入
- `services/agent-gateway/src/workspace-review.ts`
  - 提供 git/workspace 级变更检查与回滚兜底
- `packages/agent-core/src/tools/hash-edit.ts`
  - 提供哈希锚定编辑与原子回滚基础能力
- `packages/artifacts` + `services/agent-gateway/src/routes/artifacts.ts`
  - 可作为写前原文备份与恢复包的文件载体

### 当前仓库已识别缺口（需纳入实施）

- 缺少正式的会话级 / 请求级 `file-diffs` HTTP 读模型接口
- `session_file_diffs` 当前主要信任工具输出，尚未形成对实际文件系统状态的完整性校验链
- 同一文件在同一轮/同一请求被多次修改时，当前冲突检测与归并语义偏弱
- 文件变更与权限批准链尚未形成强关联（例如 `permission_request_id` 级联）
- workspace/git 真实状态与 durable diff 之间缺少统一 reconcile 机制
- 回滚工作区后，durable 记录如何标记 `reverted/restored` 仍未标准化

## 参考设计吸收矩阵

### 推荐第一阶段直接吸收

| 能力 | 来源 | 判断 | 说明 |
|---|---|---|---|
| request/round 边界清晰的 durable diff 主链 | OpenCode | 推荐 | 适合直接嫁接到现有 `session_run_events + session_file_diffs + session_snapshots` |
| 结构化 diff 作为正式产物 | OpenCode | 推荐 | 当前仓库已具备雏形，应继续强化而非退化为 UI 派生语义 |
| 写前原文备份 | Claude Code | 推荐 | 是恢复兜底的关键能力，当前系统最缺这一层 |
| turn/message 级可恢复语义 | Claude Code | 推荐 | 对用户理解与 restore preview 很重要，但应建立在 SQLite 主链之上 |
| `structuredPatch` / unified diff 作为展示与诊断输出 | Claude Code | 推荐 | 适合作为 UI / API 展示格式，但不应替代主 diff 事实源 |

### 不建议第一阶段照搬

| 能力 | 来源 | 判断 | 原因 |
|---|---|---|---|
| 独立 snapshot git repo 作为 authoritative 主存储 | OpenCode | 不推荐 | 与当前 SQLite 主链重复，容易造成双主真相源漂移 |
| transcript / JSONL 作为唯一事实源 | Claude Code | 不推荐 | 查询、关联、回放、权限与审计都会退化 |
| 只靠 tool result/structuredPatch 作为文件变更真相 | Claude Code | 不推荐 | 适合展示，不足以承担审计、恢复与一致性校验 |

### 后续阶段再考虑

| 能力 | 来源 | 判断 | 说明 |
|---|---|---|---|
| 异步 snapshot git repo 镜像 | OpenCode | 后续考虑 | 只适合做镜像/导出，不适合作为 Phase 1 主链 |
| 更细的 part/step 级 recoverability | OpenCode | 后续考虑 | 当前系统先做 request/turn 级就能获得主要价值 |
| 更重的事件溯源化重构 | 外部 event sourcing 模式 | 后续考虑 | 成本高，当前增量扩表路线更稳 |

## 外部最佳实践旁证（低权重，不改变主决策）

- **Claude 官方 file checkpointing 文档**：进一步佐证“写前原文备份 + checkpoint ID + 可 rewind”是低成本高收益路径。
- **`claude-file-recovery` 这类恢复工具**：证明从持久化工具日志/会话链回放文件状态是可行的；适合作为 restore preview 的读模型参考。
- **Vanna AuditLogger 抽象模式**：佐证把审计接口抽象为独立层是合理的，但本方案仍以 gateway 现有 durable stores 为主，不额外引入平行日志子系统。
- **生产检查清单类建议**：一致强调幂等键、checkpoint 保留期、故障注入与恢复演练应尽早纳入 rollout checklist。
- **采纳原则**：外部资料只作为旁证；若与当前仓库已冻结的“SQLite 主事实源 + 增量扩表”原则冲突，一律以后者为准。

## 依赖 DAG（摘要）

1. **必须先完成 Phase 0**，否则后续字段和 API 会反复返工。
2. `WS-1` 先于 `WS-2/WS-3`，因为共享合同与 observability 字段是下游所有落库与读取的公共约束。
3. `WS-3` 的写前备份接线依赖 `WS-1` 合同，但可以与 `WS-2` 的 store 扩展并行。
4. `WS-4` 恢复链必须等待 `WS-2 + WS-3` 完成最小闭环后再做，否则没有稳定数据源可回放。
5. `WS-5` 读模型必须建立在 durable 主链与备份引用稳定之后，避免 UI 绑定临时字段。
6. `WS-6` 验证贯穿全程，但最终 verification 脚本应在 `WS-2/3/4/5` 基本落地后统一补齐。

## 验收标准

### 架构验收
- [ ] 任何一次写类工具执行都能用 `sessionId + clientRequestId + toolCallId` 串起 run event、tool result、file diff、snapshot、backup ref
- [ ] durable 主链只有一套 authoritative source，不存在 SQLite 与第二主链并列真相
- [ ] UI/读模型消费 durable 投影，而不是直接依赖 tool output 临时语义

### 功能验收
- [ ] `edit/apply_patch/write/workspace_write_file` 具备写前原文备份
- [ ] 非结构化工具至少能被降级标记并通过 git/workspace fallback 进入审计链
- [ ] restore preview 能明确显示将恢复哪些文件、覆盖哪些内容
- [ ] hash mismatch / backup miss / git unavailable 等失败路径均有明确定义

### 测试验收
- [ ] 单测覆盖 store/schema/adapter/恢复决策逻辑
- [ ] verification 覆盖 request-scoped replay、background/resume、delete cleanup、restore dry-run
- [ ] 至少一条真实链路证明：对话 → 工具写文件 → diff 落库 → snapshot 落库 → 备份可取回 → restore preview 可执行

## 主要风险与规避

1. **双主真相源漂移**
   - 规避：Phase 0 明确 SQLite 为唯一真相源；snapshot repo 若未来引入，只能做异步镜像。
2. **bash / 外部进程无法稳定产出结构化 diff**
   - 规避：引入 guarantee level，允许弱保障但要求 git/workspace reconcile。
3. **备份层带来存储膨胀**
   - 规避：哈希去重 + artifacts 落盘 + 保留期策略。
4. **恢复链太早开放 apply，误伤用户文件**
   - 规避：先实现 preview / validate-only，最后再开放真正 apply。
5. **展示层抢跑绑定临时字段**
   - 规避：先定义读模型，再接 UI，不让前端自行拼装内部语义。
