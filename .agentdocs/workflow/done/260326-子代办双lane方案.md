# .agentdocs/workflow/260326-子代办双lane方案.md

## 任务概览

为 OpenAWork 当前单一会话级 todo 模型设计一套详细的“主待办 + 子代办/临时待办”方案。目标是在保持现有 `todowrite/todoread`、`GET /sessions/:id`、`GET /sessions/:id/todos` 兼容的前提下，引入一条不污染主计划的临时待办泳道，并给出数据结构、工具协议、HTTP 接口、Web/Desktop UI、Mobile 策略、并发控制与迁移阶段。

## 当前分析

- 现状已确认：`services/agent-gateway/src/todo-tools.ts` 中的 `SessionTodo` 只有 `content/status/priority`，且 `replaceSessionTodos()` 对整个 session 执行整单替换。
- 持久化当前只有一张 `session_todos` 表（`services/agent-gateway/src/db.ts`），主键语义为 `(session_id, position)`，说明当前 todo 既没有稳定 item id，也没有 lane/list/scope 概念。
- HTTP 读取契约固定为单列表：`GET /sessions/:id` 与 `GET /sessions/:id/todos` 当前都只返回一个 `todos` 数组。
- Web/Desktop UI 当前只消费一个 `sessionTodos` 状态源：主聊天区 `ChatTodoBar` + 右侧面板 `SessionTodoPanel` / Overview 计数。
- Mobile 端目前不渲染 todo，因此本次方案应优先覆盖 gateway + web/desktop，移动端采用兼容不接入策略。
- 已有历史决策可复用：`.agentdocs/workflow/done/260324-子任务层级实现方案.md` 已明确“子任务树 / 依赖图 / 子会话”三层语义应分离，因此本次“子代办”不应误并为任务树子节点。

## 方案设计（当前冻结方向）

### 设计目标

1. 主待办与临时待办必须语义隔离，临时想法不能默认污染主待办。
2. 现有 `todowrite/todoread` 默认行为保持不变，避免破坏现有 agent 使用习惯。
3. Web/Desktop 优先做展示与兼容接入，移动端先保持接口兼容、不强推 UI 改造。
4. 第一阶段优先最小可落地，不直接跳到无限多列表或树形嵌套。

### 当前推荐方向

- 采用 **双泳道 / 双 scope 模型**：
  - `main`：正式主待办
  - `temp`：临时待办 / 子代办 / parking lane
- 不采用树形 nested children 作为 v1：因为当前写入模式是整单重写，树结构在 LLM 场景下更容易出现误删、错挂和重排风险。
- 不直接采用 arbitrary multi-list 作为 v1：对于当前诉求属于过度设计，可作为未来升级方向。

### 初步数据模型

```ts
interface SessionTodoItemV2 {
  content: string;
  status: 'pending' | 'in_progress' | 'completed' | 'cancelled';
  priority: 'high' | 'medium' | 'low';
  lane: 'main' | 'temp';
}
```

### 初步协议策略

- 兼容层：
  - `todowrite` 只操作 `main`
  - `todoread` 只读取 `main`
- 新能力：
  - `subtodowrite` 只操作 `temp`
  - `subtodoread` 只读取 `temp`
  - 后续可新增 `subtodo_promote`，将临时项原子提升进主待办

## 详细设计（本地现状冻结版）

### 方案对比与取舍

#### 方案 1：树形子代办（nested children）

表现形式：在主待办项下挂 `children[]` 或 `parentTodoId`。

优点：

- 直觉上最像“子代办”
- 未来若有复杂依赖，理论上扩展空间大

缺点：

- 当前 todo 写入是整单替换，树形结构在 LLM 场景下更容易出现父子错挂、误删、重排
- 现有前端只有单列表展示，树形 UI 的改造成本显著更高
- 仓库已经在任务系统中明确区分“子任务树”和“依赖图”，若再把 todo 也树化，语义更易混淆

结论：**不作为 v1 推荐**。

#### 方案 2：固定双 lane（main + temp）

表现形式：同一 session 下两条独立待办列表，共享 item 基本字段，但独立排序、独立读写、独立统计。

优点：

- 与现有单列表模型最接近，改造最小
- 可以精准解决“临时高优想法污染主计划”的核心问题
- 容易与现有 UI 结构对接：主条继续渲染 main，右侧面板扩展 temp
- 更适合 LLM 工具语义：主待办和临时待办可通过不同工具名强分离

缺点：

- 首版只支持两条固定语义清单
- 若未来出现大量自定义清单需求，需要升级为多列表系统

结论：**推荐作为 v1 正式方案**。

#### 方案 3：多列表系统（arbitrary lists）

表现形式：一个 session 下允许多个 list/listId，每个 list 维护自己的元数据与排序。

优点：

- 最灵活，未来可支持更多工作模式

缺点：

- 需要 list 元数据、列表生命周期、命名治理、默认行为与 UI 导航设计
- 当前用户需求只是“主待办 + 临时待办”，直接上多列表属于过度设计
- 对 Agent/LLM 来说选择 list 的决策也会更脆弱

结论：**可作为 v2/v3 升级方向，不作为 v1**。

### 外部参考证据（已回填）

#### 1. 临时收件箱 / 正式归档是成熟模式

- Org mode 的 capture → refile 模式明确支持先把想法捕获到临时位置，再显式归档到正式位置。
- 这与本方案中 `temp` → `main` 的“提升”动作高度一致。

#### 2. 停车场 / waiting / backlog 与执行面分离是成熟模式

- Taskwarrior 的 `waiting` 状态会把任务从典型视图中隐藏，直到满足条件再回到 `pending`。
- Jira Kanban backlog 的官方说明也强调：当 backlog 增大后，需要把执行面与规划面物理隔离，避免规划项打扰进行中工作。
- 这为“temp 临时待办不直接出现在主聊天区执行条”提供了强证据。

#### 3. 视图 / 分组本身可以成为状态输入器

- GitHub Projects v2 支持按字段分组，并在拖入某列时自动写入该列对应字段值。
- 这启发后续 UI：从 `temp` 提升进 `main`，可以被建模为一种显式 lane 迁移动作，而不必手工编辑整条 todo。

#### 4. 计划与过程产物分离是 Agent 系统常见做法

- LangGraph 的 `plan` 与 `past_steps` 是两个独立列表，避免执行过程污染计划。
- Semantic Kernel 也将执行步骤与函数结果沉淀到 `ChatHistory`，而不是直接塞回 plan。
- 对 OpenAWork 而言，这强化了“临时待办不应默认进入主计划”的设计方向。

### A. 数据存储设计

#### A1. v1 最小表结构调整

在当前 `session_todos` 表上增加 `lane TEXT NOT NULL DEFAULT 'main'`。

目标主键从：

```sql
PRIMARY KEY (session_id, position)
```

升级为：

```sql
PRIMARY KEY (session_id, lane, position)
```

这样 `main` 与 `temp` 各自维护排序，不再共享一条位置序列。

#### A2. v1 写入语义

- `replaceSessionTodos(sessionId, todos, 'main')` 仅删除 `main` lane 后重建 `main`
- `replaceSessionTodos(sessionId, todos, 'temp')` 仅删除 `temp` lane 后重建 `temp`
- 禁止任何 lane 级写入清空整个 session 的全部 todo

#### A3. v2 可选增强

若后续要支持单条移动、精确提升与多端冲突处理，可追加：

- `id TEXT NOT NULL`
- `updated_at`
- lane 级 `revision`

但这些不应阻塞 v1。

### B. 工具协议设计

#### B1. 兼容原则

现有工具继续代表“主计划真相源”：

- `todowrite` = 写 `main`
- `todoread` = 读 `main`

这样旧 Agent prompt、旧工具卡片摘要、旧前端逻辑全部无需立即改写。

#### B2. 新增工具

建议新增：

1. `subtodowrite`
   - 输入：`{ todos: SessionTodoItem[] }`
   - 行为：仅覆盖 `temp` lane

2. `subtodoread`
   - 输入：空对象
   - 行为：仅返回 `temp` lane

3. `subtodo_promote`（建议 v1.5 或 v2）
   - 输入：首版可先按 `position`，后续升级为按稳定 `id`
   - 行为：从 `temp` 原子迁移一项到 `main`

#### B3. 命名语义

内部实现推荐用 `lane`，用户文案与工具描述中可使用：

- 主待办 / 正式待办
- 临时待办 / 子代办 / parking lane

避免在底层协议中直接采用 `childTodo` 这类容易与任务树混淆的命名。

### C. HTTP 契约设计

#### C1. 旧接口保持不变

- `GET /sessions/:id` 继续返回 `session.todos = main`
- `GET /sessions/:id/todos` 继续返回 `{ todos: main }`

#### C2. 新接口单独暴露多 lane

新增：

```http
GET /sessions/:id/todo-lanes
```

返回：

```json
{
  "main": [],
  "temp": []
}
```

若后续进入并发控制阶段，可升级为：

```json
{
  "main": [],
  "temp": [],
  "revisions": { "main": 1, "temp": 1 }
}
```

#### C3. 为什么不直接改旧 `/todos`

因为 Web 当前已经每 3 秒轮询 `/sessions/:id/todos`，直接改返回结构会把现有前端同步逻辑全部打断。新结构必须走新路由。

### D. Web / Desktop UI 设计

#### D1. 主视觉原则

- 主聊天区底部 `ChatTodoBar` 继续只代表 `main`
- 临时待办不直接挤入主条，避免用户误判“临时想法已进入正式计划”

#### D2. 推荐落点

首版将 `temp` 放入右侧面板：

- Overview：新增“临时待办”统计行
- History：在现有 `SessionTodoPanel` 下新增一个 `TempTodoPanel`

这样改造最小，也更符合临时事项的辅助性质。

#### D3. 交互语义

首版以展示为主：

- 支持查看 `temp` 的状态与优先级
- 不假设用户侧立即拥有完整 CRUD
- 预留后续“加入主计划 / 清空临时项”入口

### E. Mobile 策略

当前移动端不渲染 todo，因此本次方案推荐：

- v1：仅保持接口兼容，不做 UI 接入
- v2：若移动端补 todo，则优先消费 `/todo-lanes` 聚合接口，而不是继续只接 `/todos`

### F. 并发控制策略

当前写模型是 lane 内整单覆盖，因此 v1 仍有“同 lane 后写覆盖前写”风险。

建议分层处理：

- v1：先接受 last-write-wins，但把冲突范围从“整会话”收缩为“单 lane”
- v2：引入 `expectedRevision` 与 lane 级 revision
- v3：若并发频繁，再升级为稳定 id + item patch

#### F1. 推荐的 lane 级 revision 模型

建议新增一张轻量元数据表：

```sql
CREATE TABLE IF NOT EXISTS session_todo_lanes (
  session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  lane TEXT NOT NULL,
  revision INTEGER NOT NULL DEFAULT 1,
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (session_id, lane)
)
```

用途：

- 给 `main` 与 `temp` 各自维护版本号
- 避免把 revision 挂在 item 行上造成整批覆盖时的读取麻烦

#### F2. 为什么 revision 适合当前系统

外部并发控制参考显示，当前最轻量且成熟的做法是：

- 读时返回 revision / ETag
- 写时带 `expectedRevision` / If-Match
- 服务端使用 CAS 语义判断“我基于的版本是否仍然是最新”

这比直接上 CRDT、多分支合并或事件溯源要轻得多，也更适合当前 SQLite + 轮询 + 工具驱动写入模型。

#### F3. 在 OpenAWork 中的落地口径

因为当前写入既有工具调用，也有 HTTP 客户端读取，建议采用“双表达”而不是只押一种协议：

- **工具层**：在输出中返回 `revision`，在输入中可选接受 `expectedRevision`
- **HTTP 层**：`GET /sessions/:id/todo-lanes` 返回 `revisions`；后续若新增写接口，可进一步映射到 ETag / If-Match

#### F4. 冲突处理策略

- v1：不强制客户端提交 `expectedRevision`，但服务端开始维护 revision
- v1.5：Agent / 工具优先改为“先读后写”并携带 `expectedRevision`
- v2：若 revision 不匹配，返回冲突错误与最新快照，调用方重读后重试

#### F5. 为什么现在不直接做 item patch

外部参考说明 patch 更适合高并发小修改，但当前仓库的 todo：

- 没有稳定 item id
- UI 还没有手动编辑能力
- 核心痛点首先是“主待办被临时想法污染”，而不是单条编辑性能

因此 phase 1 应优先实现 lane 隔离与 revision 护栏，item patch 放到后续演进阶段。

### H. 推荐的工具与接口草案

#### H1. 数据类型

```ts
type TodoLane = 'main' | 'temp';

interface SessionTodoItemV2 {
  content: string;
  status: 'pending' | 'in_progress' | 'completed' | 'cancelled';
  priority: 'high' | 'medium' | 'low';
}

interface SessionTodoLanePayload {
  lane: TodoLane;
  todos: SessionTodoItemV2[];
  revision: number;
}
```

#### H2. 工具接口

```ts
todowrite({ todos, expectedRevision? })           // main
todoread()                                        // main
subtodowrite({ todos, expectedRevision? })        // temp
subtodoread()                                     // temp
subtodo_promote({ position, targetPosition? })    // temp -> main
```

#### H3. HTTP 接口

```http
GET /sessions/:id/todos              -> main（兼容旧前端）
GET /sessions/:id                    -> session.todos = main（兼容旧前端）
GET /sessions/:id/todo-lanes         -> { main, temp, revisions }
```

### I. UI 与产品语义建议

#### I1. 文案

- `main`：待办清单 / 主计划
- `temp`：临时待办

底层协议仍用 `temp`，避免 `childTodo` 这种会和任务树混淆的命名。

#### I2. 优先级解释

`temp` 中的 `high` 仅表示“在临时待办内部优先关注”，不表示自动抢占主计划。

#### I3. 提升语义

从 `temp` 进入 `main` 必须是显式动作：

- Agent 调用 `subtodo_promote`
- 后续 UI 可提供“加入主计划”按钮

### J. 风险与防线

#### J1. 已识别风险

1. **旧前端继续只看 `main`**
   - 这是兼容设计的一部分，但会导致 temp 在旧端不可见
2. **lane 内仍可能丢更新**
   - 通过 revision 分阶段收口
3. **temp 长期堆积**
   - 后续需增加清理或提升机制
4. **类型定义重复易漂移**
   - 需要在 web-client 收敛 todo 类型

#### J2. 产品防线

- 默认 temp 隐藏或收纳在右侧面板，不抢主视觉
- 主待办条继续只表达“本轮正式计划”
- 临时待办的价值在于“记录但不承诺”

### K. 分阶段实施建议

#### Phase 1：双 lane 基础设施

- DB 增加 `lane`
- CRUD 改为 lane 级 replace
- 新增 `GET /todo-lanes`
- 新增 `subtodowrite/subtodoread`
- Web/Desktop 右侧面板展示 temp

#### Phase 2：lane 级 revision

- 新增 `session_todo_lanes`
- 返回 `revisions`
- 工具支持 `expectedRevision`
- 冲突时返回最新快照

#### Phase 3：提升与治理

- 新增 `subtodo_promote`
- 支持 temp 清空或归档
- 视需要补充 UI 手动操作

#### Phase 4：高级演进（仅在需求出现时）

- 稳定 `todoId`
- item patch / reorder
- arbitrary multi-lists
- mobile todo UI

## 迁移与接口细节

### 1. 数据库迁移草案

#### 1.1 v1：为 `session_todos` 增加 lane

```sql
ALTER TABLE session_todos ADD COLUMN lane TEXT NOT NULL DEFAULT 'main';

-- 由于 SQLite 直接修改主键受限，实际迁移建议采用新表搬迁：
CREATE TABLE session_todos_v2 (
  session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  lane TEXT NOT NULL,
  position INTEGER NOT NULL,
  content TEXT NOT NULL,
  status TEXT NOT NULL,
  priority TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (session_id, lane, position)
);

INSERT INTO session_todos_v2 (session_id, lane, position, content, status, priority, created_at)
SELECT session_id, 'main', position, content, status, priority, created_at
FROM session_todos;

DROP TABLE session_todos;
ALTER TABLE session_todos_v2 RENAME TO session_todos;
CREATE INDEX idx_session_todos_session_lane ON session_todos(session_id, lane);
```

#### 1.2 v2：增加 lane 级 revision

```sql
CREATE TABLE IF NOT EXISTS session_todo_lanes (
  session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  lane TEXT NOT NULL,
  revision INTEGER NOT NULL DEFAULT 1,
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (session_id, lane)
);

INSERT OR IGNORE INTO session_todo_lanes (session_id, lane, revision)
SELECT DISTINCT session_id, lane, 1 FROM session_todos;
```

```ts
function promoteTempTodo(input: {

  "actualRevision": 7,
  "todos": [
    { "content": "验证某个临时假设", "status": "pending", "priority": "high" }
  ]
}
```

### 4. 前端状态结构建议

#### 4.1 Web / Desktop 页面态

```ts
type TodoLane = 'main' | 'temp';

interface TodoLaneState {
  todos: SessionTodoItemV2[];
  revision?: number;
}

interface SessionTodoViewState {
  main: TodoLaneState;
  temp: TodoLaneState;
}
```

#### 4.2 ChatPage 建议

- 保留 `main` 为首屏真相源，兼容现有 `ChatTodoBar`
- 新增聚合状态而不是继续只用 `sessionTodos: SessionTodoItem[]`
- 首次载入可先消费 `/sessions/:id` 中的 `session.todos` 初始化 `main`
- 右侧面板打开或轮询时再用 `/todo-lanes` 覆盖 `main + temp`

### 5. UI 信息架构建议

#### 5.1 主聊天区

- 继续只展示 `main`
- 文案维持“待办清单”或“主计划”
- 若 `temp` 中存在高优先项，可选择显示一条轻量提示而非直接展开列表

#### 5.2 右侧面板

建议顺序：

1. 任务（sessionTasks）
2. 主待办（main）
3. 临时待办（temp）
4. 待处理审批
5. Plan history

理由：

- 任务区已代表更高层计划或执行树
- 主待办应高于临时待办
- 临时待办属于辅助信息，不应压过主待办

#### 5.3 统计口径

Overview 推荐新增两行而不是合并：

- 主待办：`mainActive/mainTotal`
- 临时待办：`tempActive/tempTotal`

不要合并成一个总数，否则又会把两类语义混回去。

### 6. 验收标准

#### 6.1 功能验收

- 主待办和临时待办能在同一 session 下独立存在
- 写入 temp 不会覆盖 main，写入 main 也不会覆盖 temp
- 旧 `todowrite/todoread` 调用行为与现有版本保持一致
- Web/Desktop 能分别看到 main 与 temp 的独立展示
- Mobile 在未接入 UI 的情况下不因新字段/新接口崩溃

#### 6.2 风险验收

- lane 迁移后旧数据自动归入 `main`
- `/sessions/:id/todos` 与 `/sessions/:id` 的旧契约不变
- 后续 revision 冲突能以结构化错误而非静默覆盖暴露

## 内部证据回填（第一轮 explore）

### 已确认的扩展锚点

1. **数据库锚点明确**
   - `services/agent-gateway/src/db.ts` 的 `session_todos` 当前主键是 `(session_id, position)`。
   - 这意味着 lane 设计必须先改主键维度，否则无法在同一 session 下并存两套有序列表。

2. **工具协议锚点明确**
   - `services/agent-gateway/src/todo-tools.ts` 中 `todoWriteInputSchema` 当前只接受扁平 `todos[]`。
   - `services/agent-gateway/src/tool-sandbox.ts` 已对 todo 工具做 gateway-managed 分支，因此新增 lane 专用工具或 lane 参数都能在这一层集中接入。

3. **HTTP 兼容锚点明确**
   - `services/agent-gateway/src/routes/sessions.ts` 当前 `/sessions/:id/todos` 与 `/sessions/:id` 都只暴露单一 `todos` 列表。
   - 因为 `apps/web/src/pages/ChatPage.tsx` 已对 `/sessions/:id/todos` 做 3 秒轮询，旧接口不能直接破坏性改形状。

4. **前端展示锚点明确**
   - 主聊天区 `ChatTodoBar` 与右侧面板 `SessionTodoPanel` 都只消费单一 `sessionTodos`。
   - 这支持“主待办留在主条、临时待办先进入右侧面板”的渐进式策略。

5. **类型重复问题已暴露**
   - `web-client`、`ChatPage`、`chat-todo-bar`、`right-panel-sections` 中存在重复的 todo item 类型定义。
