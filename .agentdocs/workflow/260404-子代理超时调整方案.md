# .agentdocs/workflow/260404-子代理超时调整方案.md

## Task Overview
为 OpenAWork 现有子代理 / child session / background task 链路引入可审计的 timeout / deadline 机制，覆盖 child session 执行超时、父任务感知、auto-resume 边界、任务投影与跨端可见性，并保持与现有 task graph、child session、stream replay、run event 体系兼容。

## Current Analysis
- 当前仓库只有**工具级超时**与 `background_output(block=true)` 的**等待超时**；两者都不是 child session 的 wall-clock 执行超时。
- 子代理主链已经完整：`task` 工具可创建 child session + parent task，`stream-cancellation.ts` 已有 AbortController 中央表，`task-parent-auto-resume.ts` 已能把子任务终态回流父会话。
- 现有任务/事件契约里没有 timeout 一等语义：`AgentTaskStatus`、`StreamTaskUpdateChunk.status`、Web/Mobile 活动面板都主要围绕 `pending/running/completed(or done)/failed/cancelled`。
- `.agentdocs/index.md` 已记录两条强约束：
  1. child session 默认隐藏 `task` 工具，避免递归扩散；
  2. 子任务终态应收敛到父会话 `tool_result + task_update + auto-resume` 主链，而不是另起并行机制。
- 历史工作流表明：Mobile/Web 的任务可见性高度依赖 `task_update` 增量 + `/sessions/:id/tasks` 快照恢复；如果只做后端 timeout，不同步 UI 映射，会出现“后端已终态、前端仍显示 running”的假活跃态。

## Solution Design

### 设计原则
1. **Gateway first**：Phase 1 先在 gateway 落地 child session deadline，不推翻 `packages/agent-core` task graph。
2. **状态兼容优先**：Phase 1 不立即扩 `task.status` 枚举；timeout 先收敛为 `failed + terminalReason=timeout`。
3. **统一终结入口**：timeout / cancel / 正常完成都走同一个幂等终结函数，避免双写和状态漂移。
4. **父链可见**：timeout 必须进入 `tool_result / task_update / assistant reminder / auto-resume` 主链，不能只做内存 timer。
5. **在线触发 + 离线补偿**：内存 timer 负责在线终止，reconcile / 读取路径负责进程重启后的 deadline 补偿。

### Phase 1：最小闭环（推荐先做）
- `task` 输入新增 `timeoutMs`（或更明确 `taskTimeoutMs`），默认**不隐式开启超时**；只有显式传入或后续 session policy 配置时才生效。
- child session 启动时计算 `effectiveDeadline = min(parentDeadline, childRequestedDeadline)`；若父无 deadline，则仅使用子配置。
- 最小新增字段：`timeoutMs`、`deadlineAt`、`terminalReason`、`terminatedAt`、`terminationSource`。
- 运行时到点后复用现有 abort 基础设施终止 child stream，并统一收敛为：
  - task.status = `failed`
  - terminalReason = `timeout`
  - errorMessage 采用稳定前缀（如 `TIMEOUT:`）
- `cancelled` 保留给用户/API 主动取消；timeout 不复用 cancelled。
- `background_output.timeout` 继续只表示“调用方等待多久”，**不**终止 child session。

### Phase 2：契约与可见性补齐
- 在 `packages/shared` 与 gateway 投影中为 timeout 增加结构化原因字段，优先新增 `reason/terminalReason`，而不是立刻扩一整套 status 枚举。
- Web / Mobile / web-client 同步支持 `failed + reason=timeout` 的展示与恢复逻辑，避免把 timeout 压扁成 generic error 或旧 running。
- assistant event / completion reminder 文案显式区分：完成、失败、超时、取消。

### Phase 3：DAG / 审批超时 / 生命周期清理
- 为 `packages/multi-agent` 预留 `executionTimeoutMs` / `approvalTimeoutMs` / `onTimeout` 策略，但后置到 Phase 3，不阻塞 child session 最小闭环。
- `TTL / cleanup` 只处理终态后清理，不与执行超时混用。
- 若后续要把 timeout 升级为 task 一等状态，再同步 `AgentTaskStatus`、`RunEvent`、前端枚举与验证矩阵，避免半途扩枚举导致 UI/协议断裂。

### 范围边界（明确不做）
- 不把 timeout 伪装成 cancelled。
- 不让 `background_output.timeout` 去杀 child session。
- Phase 1 不引入外部调度器/队列系统。
- Phase 1 不重做 task graph，也不把 `interrupted` 误当终态。

## Notes
- 方案决策：Phase 1 采用 **gateway-first + timeout 作为终结原因**，先不扩 `task.status` 联合类型，避免 shared/web/mobile 全链枚举爆炸。
- 方案决策：Phase 1 的 deadline 是 **child session 的 absolute wall-clock budget**；approval/interaction 专属超时（`approvalTimeoutMs`）后置到 Phase 3。
- Oracle 复核结论：`failed + terminalReason=timeout`、`effectiveDeadline = min(parent, child)`、gateway-first 分阶段落地均可保留；但 **cancelled 必须显式向父链传播终结，不能只是子任务本地停止**。
- Oracle 复核结论：Phase 1 就应在 `task_update` 与 `/sessions/:id/tasks` 预埋 `terminalReason / effectiveDeadline`，哪怕客户端暂时忽略，也不能把这些关键字段继续藏在内部 metadata。
- 插点优先级：`tool-sandbox.ts`（创建与终结）、`stream-cancellation.ts`（timer + abort）、`task-parent-auto-resume.ts`（父链文案与回流）、`session-task-projection.ts`（任务投影）是第一批必查文件。
- 风险最高的不是“timer 触发不了”，而是**timeout 发生后只写了后台状态，父会话、任务列表、Web/Mobile 却看不见**；因此可见性与终结原因传播不能长期后置。
- 终结语义冻结：timeout 走 `failed + terminalReason=timeout + parent auto-resume`；cancelled 走“沿父链传播终结、禁止 auto-resume”；若父任务已终结，则只审计不复活。
